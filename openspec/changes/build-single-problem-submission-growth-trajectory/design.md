## Context

系统已经保存 `Submission`、测试点结果、正式诊断事实和 `SubmissionIssueTransition`，并能在单次学生反馈中返回 `NEW`、`PERSISTED`、`NOT_OBSERVED`、`RECOVERED`、`RECURRED` 与 `UNCOMPARABLE`。当前缺口不是新的诊断能力，而是缺少统一的单题成长投影：学生结果页只有原始变化计数，历史接口只有普通提交摘要，教师学生详情只有逐问题聚合，三处无法用同一口径回答“这次相对哪次提交发生了什么变化”。

变更涉及提交历史服务、生命周期读取、学生题目页、教师题目下钻和国际化，需要先统一比较规则与数据契约。学生希望获得支持性但真实的下一步，教师需要客观可核验的介入依据；两者必须消费同一事实，只允许表达方式不同。

当前代码复核确认：生命周期投影、批量测试点统计、学生反馈完整问题项和教师逐问题轨迹均可复用；原实现的主要缺口是生命周期仍使用物理相邻提交、历史摘要没有成长对象、教师与学生无法消费同一逐提交投影。实现因此直接修正现有投影并扩展兼容性 DTO，不新增第二套诊断链路或成长事实表。

## Goals / Non-Goals

**Goals:**

- 为同一学生、同一道题、同一作业范围建立全量提交记录和有效比较链。
- 基于判题、可比测试点与完整问题生命周期生成确定性的单次成长状态和结构化证据。
- 让学生当前结果、学生本题时间线、全局提交详情和教师学生单题详情复用同一成长投影。
- 在外部模型建议失败或仍在生成时，继续展示已经可靠获得的判题事实与成长边界。
- 保持历史版本可回看、教师校正后可重算、前端中英文一致且移动端可访问。

**Non-Goals:**

- 不设计班级级复杂统计、成长排行榜或学生之间的成长比较。
- 不引入单一成长分数、能力分或基于修改代码行数的进步判断。
- 不新增诊断 Agent，不让模型决定成长状态，也不宣称 AI 反馈导致学生改善。
- 不把提升建议计入错误总数，不把一次新问题直接定义为长期能力缺陷。
- 不在本变更中重做整个教师控制台或全局提交中心。

## Decisions

### 1. 成长摘要是可重算派生投影，不新增不可逆历史事实

新增 `SubmissionGrowthSummaryService` 一类的领域服务，从提交、测试点统计和已经持久化的问题转换生成结构化摘要。摘要默认在读取或批量组装时计算，不反写覆盖 `SubmissionDiagnosisFact` 与 `SubmissionIssueTransition`。

这样教师校正、生命周期算法升级或历史回填后可以重新计算，不需要迁移旧摘要。备选方案是新增成长摘要表，但会引入同步、回填和版本一致性成本；当前已有转换投影足以支持读取计算，因此暂不新增表。

### 2. 所有提交进入时间线，只有有效且可比较的提交进入比较链

范围固定为同一 `studentProfileId + problemId + assignmentId`；公共题库使用空作业范围，课堂作业不得跨作业比较。每次提交都保留并返回，但比较基线选择当前提交之前最近的、源码或评测/问题集合形成有效变化且诊断事实完整的提交。

完全相同的代码和评测/问题集合标记为 `DUPLICATE_NO_CHANGE`，分析不完整或身份缺失标记为不可比较。它们显示在时间线中，但后续有效提交继续向前寻找最近可比较基线，并明确返回 `comparisonSubmissionId`，避免用户误以为物理相邻提交就是比较对象。

### 3. 成长状态由规则优先级和并列证据组成，不压缩为分数

成长状态使用稳定代码：`FIRST_RECORD`、`DUPLICATE_NO_CHANGE`、`UNCOMPARABLE`、`COMPLETED`、`CLEAR_PROGRESS`、`MIXED_PROGRESS`、`ISSUE_SHIFTED`、`STALLED`、`REGRESSED`。

判定顺序先处理首次、重复和不可比较，再处理当前已通过，最后综合可比评测变化与问题变化：测试点增加、评测阶段明确前进、旧问题未观察或恢复属于正向证据；测试点减少、评测阶段明确倒退、新增或复发属于风险证据；持续问题属于未解决证据。`WRONG_ANSWER`、`TIME_LIMIT_EXCEEDED` 与 `MEMORY_LIMIT_EXCEEDED` 不仅凭枚举顺序互相判定进退；测试点数量只有在测试集合可比时使用。

摘要并列返回正向证据、风险证据、持续问题和优先问题，不计算净成长分数。代码行数和改动行数只可作为审计或对比辅助，不能决定成长状态。

### 4. `NOT_OBSERVED` 与 `RECOVERED` 保持不同证据强度

问题在一次可比较提交中缺失只表示 `NOT_OBSERVED`，不得直接宣称彻底修复。当前提交通过，或满足生命周期服务定义的更强重复验证条件后，才可显示 `RECOVERED`。问题在恢复证据后再次出现标记为 `RECURRED`。

推断文本键匹配的低置信度问题在文案中必须表达为“疑似同类问题”；正式标准库身份才能直接表述为同一问题。

### 5. 成长投影与模型建议状态解耦

学生结果响应将成长摘要作为独立结构返回。判题和完整生命周期事实可用时，即使学生 AI 建议仍在生成或生成失败，也展示相应成长事实；生命周期不完整时返回 `UNCOMPARABLE` 或部分证据状态，不用本地模板伪装完整诊断。

外部模型继续负责当前提交的具体诊断与解释，不能覆盖成长状态。前端不得以 `source === MODEL` 作为展示成长摘要的唯一条件。

### 6. 历史接口批量返回成长摘要，避免前端逐条请求

扩展现有本题历史摘要响应，增加 `growthState`、`effectiveAttempt`、`comparable`、`comparisonSubmissionId`、`duplicateOfSubmissionId`、评测差值、问题变化计数和优先问题。后端批量读取提交、测试点统计和转换记录并一次组装，禁止前端为每条时间线节点分别请求学生反馈形成 N+1 调用。

历史默认按时间倒序，所有节点可重新打开原提交结果。重复和不可比较节点由前端降低视觉权重但不删除；长历史使用增量加载或分页，不改变全量可审计性。

### 7. 学生与教师复用数据契约，使用不同展示模式

前端抽取共享的成长状态映射、证据指标和时间线节点组件。学生模式强调具体进步、风险和下一步；教师模式展示客观状态、比较基线、持续/复发问题和证据提交，不复用鼓励性语气替教师下结论。

学生当前结果页在判题信息下增加跨栏成长摘要，再展示评测、当前修正和可选提升。教师从题目学生列表进入学生详情后，展示同一提交时间线，并保留既有逐问题轨迹和校正功能。教师校正必须绑定具体提交，而不是隐式只针对最新提交。

### 8. 后端返回状态与证据，前端负责双语表达

后端返回稳定状态代码、计数、提交引用和优先问题标题，不返回写死的整段鼓励文案。前端通过 `t(...)` 组合学生与教师两套中英文模板，同时使用文字、图标和布局区分状态，不能只依赖颜色。

### 9. 前端采用数据型成长仪表盘而不是解释型卡片

确认视觉基准为 `docs/ui-mockups/single-problem-growth-dashboard.png`。学生结果页以连续数据框展示提交次数、有效修改、当前测试点和未解决问题；用测试点趋势图回答整体是否向前，用知识点排名条形图回答高频卡点，用逐提交堆叠条回答问题是减少、转移还是复发，用知识点 × 提交矩阵回答每个问题分别出现在哪次提交。

趋势图只有在至少四个有效节点时使用折线；更少节点使用直接前后对比，避免制造虚假趋势。知识点构成不使用饼图作为主图，长标签与精确频次使用横向条形图。图表服务于形状和比较，矩阵与可打开证据负责精确查询。学生与教师复用同一图表数据结构，教师模式只调整文案密度与证据入口。

### 10. 验收样本矩阵先于页面实现

| 场景 | 成长状态 | 必须保留的证据 |
|---|---|---|
| 首次有效提交 | `FIRST_RECORD` | 当前评测与完整问题集合 |
| 完全重复提交 | `DUPLICATE_NO_CHANGE` | 原始记录、重复基线、不增加有效次数 |
| 分析不完整 | `UNCOMPARABLE` | 评测结果、完整性原因、不打断后续基线 |
| 旧问题减少且无新增 | `CLEAR_PROGRESS` | 测试点变化、未观察/恢复问题 |
| 10 个问题降到 5 个且新增 1 个 | `MIXED_PROGRESS` | 4 持续、6 未观察、1 新增、测试点差值 |
| 旧问题被等量新问题替换 | `ISSUE_SHIFTED` | 前后问题集合与实际比较提交 |
| 评测和问题均无明确变化 | `STALLED` | 持续问题与有效尝试证据 |
| 测试点倒退或只新增/复发 | `REGRESSED` | 负向评测差值、新增/复发问题 |
| 当前提交通过 | `COMPLETED` | 通过评测、恢复证据、可选提升建议分离 |

## Risks / Trade-offs

- [旧提交缺少规范事实，时间线出现大量不可比较节点] → 保留真实完整性状态，提供幂等回填；不得把缺失误报为没有问题。
- [成长分类阈值调整导致历史显示变化] → 摘要保持可重算并给规则版本，原始提交和生命周期事实不变。
- [批量历史组装增加查询成本] → 使用提交 ID 批量读取测试点统计和转换记录，避免逐条反馈查询，并为长历史分页。
- [学生把未观察到理解为已修复] → UI 和双语文案严格区分 `NOT_OBSERVED` 与 `RECOVERED`。
- [教师端与学生端出现状态不一致] → 两端只消费同一个后端成长状态与证据对象，不在前端各自分类。
- [混合进步状态信息过载] → 第一层只显示结论、关键正向证据、新增/复发风险和下一步，其余问题变化折叠展开。

## Migration Plan

1. 先增加后端成长摘要服务与兼容性 DTO 字段，不移除旧字段。
2. 为现有历史数据执行或复用生命周期幂等回填，缺失记录继续返回完整性状态。
3. 学生当前结果页先接入新摘要，同时保留原评测和反馈区作为回退。
4. 将最近提交列表替换为成长时间线，再接入全局提交详情。
5. 教师题目学生列表和学生详情接入同一摘要，最后移除重复的前端推断。
6. 若上线后需要回滚，前端可停止读取新增字段并恢复旧列表；新增字段为向后兼容扩展，无需回滚历史事实。

## Open Questions

- 长历史首屏数量和分页方式在实现时按现有页面密度确定，但不得影响全量回看。
- 更强的 `RECOVERED` 重复验证阈值继续使用生命周期配置，不在前端写死。
