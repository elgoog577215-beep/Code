## Context

现有前端已经做过视觉降噪，但产品入口仍保留早期试点形态：学生主路径通过邀请码解析作业，公共题库在顶层导航里独立出现，题目页又天然像学生上下文的一部分。这个模型适合演示单份作业，不适合学校长期使用。

学校落地后的稳定模型应是：

- 学生和老师分别拿到不同网址：学生访问 `/app/student`，老师访问 `/app/teacher`。
- 学生先进入学生端，未登录时可以使用公共题库。
- 学生登录班级身份后，看到公共题库和本班老师作业。
- 教师在教师端维护班级、题目、作业和课堂过程。
- 题目页是公共题库和作业详情共同复用的练习页面。

当前后端已经有班级、学生画像、作业、作业任务、提交和轨迹分析。缺口不是新数据模型，而是一个不依赖邀请码的学生身份入口和按学生班级过滤作业的读取接口。

## Goals / Non-Goals

**Goals:**

- 把 `/app/student` 和 `/app/teacher` 作为真实使用入口，明确学生端与教师端。
- 将 `/app` 作为兼容路径跳转到 `/app/student`，不再做角色选择首页。
- 把 `/app/student` 做成学生作业入口：右上角登录/身份，主体只展示公共题库和老师作业列表。
- 新增 `/app/student/login` 处理班级与姓名登录。
- 新增 `/app/student/assignments/public` 作为公共题库公开作业入口。
- 新增 `/app/student/assignments/:assignmentId` 作为老师作业入口，并自动进入下一道待做题。
- 新增 `/app/student/assignments/:assignmentId/problems/:problemId` 作为 PTA 式做题工作台，左题单、中题面、右代码。
- 保留 `/app/student/problems` 和 `/app/problems` 兼容跳转到公共题库公开作业入口。
- 为学生登录和可见作业补齐薄 API，复用现有身份 key 和作业结构。
- 保留邀请码兼容，不把它作为学生端主要入口。

**Non-Goals:**

- 不做完整鉴权、密码系统、教师登录或权限中间件。
- 不改变提交、判题、AI 诊断、学习轨迹和教师看板算法。
- 不新增数据库表或迁移。
- 不重写教师端复杂业务页面。
- 不引入新 UI 框架、状态管理库或图标库。

## Decisions

### Decision 1: 学生登录复用 StudentProfile，而不是新增账户表

学生登录只需要建立“这个浏览器里的学生是谁、属于哪个班”。现有 `StudentProfile` 已经承载 `classGroupId`、姓名、学号和稳定 identity key，因此本轮新增 `POST /api/student/login`，复用 `StudentIdentityService.buildStableIdentityKey` 查找或创建画像。

替代方案是新增账户表或教师导入名单强校验。它更完整，但会带来权限、迁移、导入校验和失败处理，本轮不需要。

### Decision 2: 作业可见性按 classGroupId 过滤

`GET /api/student/profile/{studentProfileId}/assignments` 返回该学生班级下的可见作业。作业必须是 `ACTIVE`，且时间窗口未关闭；未绑定班级的教师作业不进入学生“我的作业”，避免学生误入别班作业。

替代方案是继续用邀请码解析作业。它保留兼容，但不适合作为主链路，因为学生无法从“我属于哪个班”自然看到老师布置的所有作业。

### Decision 3: 公共题库是公开练习集，不伪装成数据库作业

公共题库使用现有 `/api/problems/catalog`，在前端呈现为“公共练习”。进入题目页时不传 `assignmentId`，可以传当前学生画像用于能力画像和推荐记录，但不强制登录。

替代方案是创建一条公共作业记录。它会把公共题库和教师作业混在同一个作业表语义中，短期增加教师端噪音。

### Decision 4: 旧路由保留兼容，`/app` 不再承载首页选择

旧的 `/app/problems`、`/student`、`/problem/:id` 继续跳转或渲染，避免已有链接失效。`/app` 直接跳到 `/app/student`，因为正式使用时会分别给学生和老师两个网址。主导航和入口文案使用 `/app/student/problems`、`/app/student/assignments` 和 `/app/teacher`。

替代方案是保留角色选择首页。它适合演示，但真实课堂中多一步选择会增加困惑，也容易让学生误入教师端。

### Decision 5: 学生作业入口直接进入 PTA 式工作台

点击公共题库或老师作业后，系统直接进入做题工作台，而不是先停留在独立作业详情页。工作台固定为左侧题目列表、中间题目内容、右侧代码编辑。老师作业默认选下一道未通过题，公共题库默认选第一题。

替代方案是保留“作业详情页 + 再点题目”的二级入口。它更像管理页，但学生每次进入都多一步选择，不符合课堂使用中“马上开始做题”的节奏。

### Decision 6: 提交反馈进入弹窗，不常驻占用右栏

提交结果、测试点、错误提示、优化提示和 Coach 下一问统一放在提交后的结果弹窗里。关闭弹窗后，做题页只保留一个“查看结果”入口。这样右侧长期留给写代码，避免学生还没提交时被 AI 分析区干扰。

替代方案是继续把反馈常驻在右栏。它能持续可见，但会挤压代码区域，也让学生把注意力放到解释区而不是当前代码。

## Risks / Trade-offs

- 学生登录不是强认证，学生可以选择班级并输入姓名。→ 本轮定位为课堂工具入口，不作为安全边界；后续若学校要求真实权限，再引入教师发布班级口令或账号体系。
- 班级名单强校验暂不做，可能出现学生手输姓名生成新画像。→ 保留教师端身份合并/拆分能力，后续可在导入名单稳定后改为名单匹配。
- 新增薄 API 会扩大本轮范围。→ API 只读/创建学生画像和读作业，不触碰提交、AI、评测和数据库结构。
- 公共题库不绑定作业时，轨迹只依赖学生画像和提交历史。→ 题目页继续兼容无 `assignmentId` 的练习模式；作业内练习仍传 `assignmentId`。
