## Why

现有标准库已经能按“知识点 → 能力点 → 易错点 / 提升点”完成细颗粒诊断，但仍缺少把抽象能力落到真实教学活动和竞赛约束中的应用层：AI 能命名问题，却不能稳定读取“课堂怎样观察、教师怎样检查、竞赛怎样迁移”。同时，生产库仍有 17 个正式能力点没有任何提升路径，集中在二分、动态规划、图、前缀和、搜索、模拟、排序和映射结构等核心竞赛能力。

## What Changes

- 新增统一标准库下的“教学—竞赛应用场景”规范表，以稳定场景 code 和迁移对 code 成对保存课堂任务与竞赛任务，不建立平行知识树。
- 每个场景显式关联知识点、正式能力点、易错点和提升点，并保存任务情境、学生可观察证据、常见失败、教师检查动作、约束画像、成功标准和迁移说明。
- 首批人工审校 17 组课堂—竞赛迁移对，共 34 个场景，覆盖二分、DP、图、搜索、前缀差分、贪心、排序、滑动窗口、模拟、哈希与集合映射。
- 为上述 17 个正式能力点各补一条可执行提升路径，使剩余无提升路径条目只保留历史兼容能力，不用兼容标签承载新增正式内容。
- 教师端知识点诊断层和 AI 最终标准库包都返回有界、相关的场景；场景只提供教学与迁移语境，不作为必须命中的诊断 ID，也不限制 AI 自由诊断和 `HIT / PARTIAL / MISS`。
- 增加迁移内阻断门禁、PostgreSQL 快照审计、导航/提示词回归测试和版本差异证据；没有真实学生数据时只证明结构、内容和链路可用，不宣称教学成效。

## Capabilities

### New Capabilities

- `teaching-contest-application-scenarios`: 规定统一标准库中课堂—竞赛场景对的数据合同、引用完整性、读取边界和质量门禁。

### Modified Capabilities

- `durable-ai-diagnosis-workflow`: 标准库挂接完成后，学生建议生成必须能读取与已选知识点和能力点相关的有界应用场景，同时保持单诊断 Agent、证据优先和库外发现边界。

## Impact

- 数据库：新增 Flyway V7、`ai_standard_application_scenarios` 表及索引，补充 17 条规范提升点并同步兼容快照和 legacy mapping。
- 后端：新增场景实体与仓储；扩展标准库诊断层 DTO、导航服务、最终 `StandardLibraryPack` 和提示词合同。
- 测试与审计：增加场景成对性、引用、内容质量、提示词消费、Flyway V1→V7 和生产 V6 快照迁移验证。
- 生产：发布前备份 PostgreSQL 与当前应用镜像；只替换应用容器，不重建或清理数据库 Volume。
