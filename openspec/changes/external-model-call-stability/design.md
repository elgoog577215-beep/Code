## Context

真实外部模型完整评测中，首条提交诊断能够完成并命中教师期望，但后续连续样本大量触发 429、额度不足或限流，导致 6 条样本中 5 条运行回退。这个结果说明当前 agent 的瓶颈已经从“单次回答质量”扩展到“外部模型调用稳定性”。

当前系统中 `AiReportService` 和 `CoachAgentService` 各自直接请求 OpenAI-compatible 接口。提交诊断、成长报告和 Coach 追问的失败归因不完全一致，live eval 也没有样本间节流，因此连续评测容易制造突发调用。

## Goals / Non-Goals

**Goals:**

- 对可重试的外部调用失败增加轻量重试与等待。
- 让提交诊断、Coach 追问、成长报告的失败归因更一致。
- 让 live eval 可以通过环境变量控制样本间等待，避免评测本身触发限流。
- 保持原有安全校验、答案泄露检查和阶段化输出验证不降低。

**Non-Goals:**

- 不引入新的第三方限流组件。
- 不在本轮实现跨进程全局队列或数据库级调用预算。
- 不改变外部模型供应商、模型 ID 或生产部署配置。
- 不把规则 fallback 伪装成外部模型成功。

## Decisions

1. 在调用层做小型重试，而不是重写 AI 客户端。

   当前项目还没有统一 AI client 抽象。为了降低风险，本轮在两个现有服务内加入相同的重试判断和等待逻辑，后续若继续扩大 AI 能力，再抽成共享组件。

2. 只重试调用层瞬时失败，不重试模型内容校验失败。

   429、rate limit、timeout、空响应属于调用稳定性问题；安全拒绝、无效标签、无效证据属于 agent 质量或安全问题。后者不能通过重复调用掩盖。

3. live eval 增加样本间等待配置。

   真实评测的目的不是压测供应商，而是观察 agent 质量。通过 `AI_EVAL_CASE_DELAY_MS` 控制样本间等待，可以减少评测噪声，同时保留不等待的压力测试能力。

4. 失败归因优先保留外部错误类型。

   报告中必须能看出是 `INSUFFICIENT_QUOTA`、`RATE_LIMITED`、`TIMEOUT` 还是质量未命中。否则后续优化会误把供应商/预算问题当成提示词问题。

## Risks / Trade-offs

- 重试会增加单次请求延迟 -> 默认重试次数保持很低，并提供配置关闭或降低。
- 429 可能来自真实额度耗尽，重试也无法解决 -> 报告仍保留 `INSUFFICIENT_QUOTA` 或 `RATE_LIMITED`，不把失败隐藏。
- 两个服务内重复实现重试逻辑 -> 本轮先保证上线稳定性，后续可抽取共享 `ExternalModelClient`。
- 样本间节流会拉长评测时间 -> 通过环境变量启用，默认保持轻量。
