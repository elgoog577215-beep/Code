# 学生优先 UI 主流程再简化设计

## 核心原则

每个页面只保留一个主动作：

- 学生端主动作：开始或继续下一题。
- 题目页主动作：提交后先处理一个反馈项。
- 教师端主动作：先看需关注学生和高频问题。

其他信息只作为第二层信息出现：列表、折叠详情、单作业详情页或教师系统区。

## 信息架构

### 上下文导航

顶部导航不再在所有页面展示同一组入口。入口页保留学生、教师、题库；学生与题目页展示“当前作业、题库、邀请码”；教师相关页面展示“课堂、管理、题库”。这不是权限控制，只是信息降噪。

### 学生端

`/app/student` 首屏顺序调整为：

1. 轻量身份条：姓名、班级、座号、作业名、完成进度。
2. 下一步行动：下一题标题、主要处理说明、开始/继续按钮。
3. 题目列表：题号、题名、难度、状态、操作。
4. 学习记录：默认折叠，展示完成度、提交次数、长期信号和每题状态。

身份未确认时，身份表单仍在首屏，但题目列表只给出禁用操作。

### 题目页

`/app/problem/:id` 继续使用题目、代码、反馈三段布局。提交后反馈区改为：

1. 主反馈卡：只显示本次先做的一件事、当前结果、公开失败证据。
2. 下一问：紧贴主反馈，作为学生下一步思考动作。
3. 折叠详情：测试点详情、系统反馈、作业记录、尝试记录。

提交前反馈区保持空状态，不展示过多解释。

### 教师端

教师首页保持课堂工具密度，但首屏重点是课堂 KPI、需关注学生、高频问题和作业详情入口。AI 质量和系统状态只保留摘要，详细维度继续在折叠区。

## 风险与取舍

- 上下文导航可能让教师入口在学生页不显眼；入口页和教师 URL 仍保留教师入口。
- 学习记录默认折叠可能隐藏一些 AI 能力；这是为了让学生先完成主流程。
- 题目页主反馈更强，会牺牲部分一次性展示的信息密度；教师端仍能看到完整细节。
