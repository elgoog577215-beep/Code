## Context

当前 ModelScope 主链路使用统一 OpenAI 兼容请求体，模型默认可能进入推理模式。V4 Pro 短请求可用，但完整诊断实测出现推理 token 挤占正文、学生 JSON 截断和配额压力。用户决定使用一条可配置调用链，默认关闭推理并将 V4 Pro 设为主模型，同时保留显式开启推理的运维能力。

## Goals / Non-Goals

**Goals:**

- 所有 ModelScope 请求显式发送推理开关，默认值为关闭。
- 默认主模型统一为 `deepseek-ai/DeepSeek-V4-Pro`。
- 保持现有完整诊断阶段、并行策略、失败记录和备用模型机制。
- 本地与学校服务器使用同一运行策略，并通过真实完整链路验证。

**Non-Goals:**

- 不减少问题数量、挂接轮次或 AI 调用数量。
- 不展示模型推理过程。
- 不修改正式诊断的数据模型和教师统计口径。

## Decisions

1. 在 ModelScope 兼容请求体顶层发送 `enable_thinking=${AI_ENABLE_THINKING:false}`。默认关闭已通过 V4 Pro 流式 API 实测；需要实验推理模式时只覆盖环境变量，不复制代码路径或应用镜像。非 ModelScope 请求不附加该字段。
2. 默认模型和模型池由配置统一切换，主模型为 V4 Pro，Qwen 仅在供应商失败时作为完整链路备用模型。
3. 不通过扩大到超大输出预算解决推理截断，因为这会继续放大延迟、配额和成本；关闭推理后保留现有结构化输出预算与重试门禁。
4. 切换门禁包含请求体单测、后端回归、真实困难样本完整阶段成功、服务器 AI smoke 和一次正式提交重试。

## Risks / Trade-offs

- [非推理模式降低极复杂问题的推理上限] → 使用真实困难样本检查问题召回和证据质量，失败则不切换生产。
- [V4 Pro 账户配额仍可能不足] → 保留 Qwen 备用模型和明确运行记录，不降低路径质量。
- [ModelScope 参数行为变化] → 通过请求体测试和部署后 smoke 及时暴露变化。

## Migration Plan

1. 先在本地加入参数和默认模型并运行回归测试。
2. 使用 V4 Pro 非推理模式完成真实困难样本。
3. 备份服务器 `.env`，同步模型与非推理配置，只重建应用容器。
4. 运行 readiness、AI smoke 和正式提交验证；失败时恢复服务器模型配置并重建应用容器。
