## Context

项目现在的问题不是缺少能力，而是能力过度叠加后主链路变得不清楚。`submission` 和 `classroom` 两个包里同时存在：

- 学生/教师真实使用的主功能：作业、题目、提交、判题事实、学生 AI 反馈、课堂概况。
- 内部支撑能力：证据构建、安全校验、模型输出规范化、学习轨迹分析。
- 研发/评测/排障能力：AI quality、eval fixture、runtime profile、fallback、smoke、baseline、comparability。

这些能力都有存在理由，但不能处于同一个产品层级。学生和老师需要的是稳定后盾：判题准、提示有用、老师能看懂班级情况。复杂 AI 评测和运行诊断应该服务研发，不应该继续把后端接口和前端体验拉复杂。

## Goals / Non-Goals

**Goals:**

- 定义后端主链路分类：`keep-main`、`simplify`、`internal-only`、`remove-later`。
- 产出当前后端审计清单，覆盖 submission/classroom 主要服务、DTO 和接口。
- 建立后续新增能力的判断门槛：是否直接服务学生下一步、教师课堂判断或判题稳定。
- 第一轮只做低风险边界收敛，不破坏现有判题和 AI 反馈。

**Non-Goals:**

- 本轮不删除数据库表。
- 本轮不删除已被前端或测试使用的 API。
- 本轮不重写 AI 模型调用链。
- 本轮不把所有 analyzer 合并成一个新大服务。
- 本轮不把研发观测指标搬到教师主界面。

## Decisions

### Decision 1: 先分类，再删除

先建立审计文档和分类规则，不直接大规模删代码。

理由：

- 当前工作区有多轮未提交改动，直接删除风险高。
- 很多复杂代码仍被教师看板、评测测试或 AI 安全链路依赖。
- 分类能让后续删除从“凭感觉”变成“按边界”。

替代方案：直接删除 AI quality、eval、runtime 相关代码。放弃，因为会破坏已有测试和教师端辅助能力。

### Decision 2: 主链路只保留教学动作

后端主链路定义为：

```text
学生：作业 -> 题目 -> 提交 -> 判题事实 -> 学生 AI 反馈
老师：作业 -> 班级概况 -> 需关注学生 -> 共性问题 -> 课堂动作
```

凡是不能直接回答这两个链路的问题，都默认进入 `internal-only`。

理由：

- 这符合学校实际使用方式。
- 可以防止新功能继续变成“为了 AI 而 AI”。
- 前端也能围绕清晰对象消费数据。

### Decision 3: `SubmissionAnalysis` 暂时作为兼容层

旧 `SubmissionAnalysis` 不再作为学生端主体验的组织中心，但暂时保留为内部兼容层，供学习轨迹、教师洞察、评测和历史接口读取。

理由：

- 现在很多 analyzer 和测试仍依赖它。
- 直接删除会把收敛变成重写。
- 更合理的路径是让新主链路先稳定，再逐步抽离内部消费者。

### Decision 4: 内部能力允许复杂，但必须下沉

AI quality、eval fixture、runtime profile、fallback、comparability 这类能力可以保留，但必须满足：

- 不作为学生主接口字段。
- 不作为教师首屏主信息。
- 不使用工程状态码作为面向老师/学生的主文案。
- 只在系统详情、测试、研发报告或内部服务中出现。

理由：

- 内部复杂度对研发有价值。
- 面向学校的产品体验不应承担这些复杂度。

## Risks / Trade-offs

- [Risk] 只做分类看起来“不够改代码”。  
  → Mitigation: 本轮输出可执行审计清单和边界规则，并做一处低风险代码化边界辅助，为后续删除建立依据。

- [Risk] 分类错误可能误导后续清理。  
  → Mitigation: 清单中给出依据和依赖，不把 `remove-later` 等同于立即删除。

- [Risk] 内部复杂能力继续膨胀。  
  → Mitigation: spec 要求新增内部能力必须说明归属，不允许默认进入主链路。

- [Risk] 老接口兼容期过长。  
  → Mitigation: 在审计中标记兼容层，并为下一轮收敛列出候选删除/合并项。

## Migration Plan

1. 创建 OpenSpec 变更和规格。
2. 审计 submission/classroom 后端文件与接口，产出 `backend-core-audit.md`。
3. 新增轻量代码边界说明，例如后端主链路分类枚举/文档类，供测试锁住分类语义。
4. 对 controller/API 做低风险边界注释或辅助分类，不改变接口返回。
5. 运行 OpenSpec validation 和相关后端测试。
6. 下一轮再基于审计清单删除或合并具体旧账。
