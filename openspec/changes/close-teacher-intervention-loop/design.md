## Context

当前课堂端已经具备三个相关能力：`ClassroomService` 会生成课堂复盘建议，`ClassReviewFeedbackService` 会记录教师对建议的采纳、调整或忽略，学生后续提交和诊断分析也已经在同一作业内可查询。缺口在于这些信息没有被拼成一个“教师动作后的成效状态”：老师按下采纳后，系统仍无法解释这个动作是否带来改善、是否没有后续证据，或是否仍卡在同类错因。

这次改动面向教师、学生和开发者三类观察者：教师需要知道课堂动作是否值得继续；学生需要收到更合适的下一步；开发者需要把教师介入效果纳入 AI 质量闭环。

## Goals / Non-Goals

**Goals:**

- 从已有教师复盘反馈、证据标签、示例题目和后续提交中推导教师介入成效状态。
- 在课堂复盘建议中展示状态、证据、下一步动作和升级判断。
- 将教师介入闭环加入 AI 质量概览，让“教师已介入但仍无改善/无后续证据”成为可观察质量信号。
- 保持兼容，不破坏已有作业概览、复盘反馈接口和 AI 质量接口。

**Non-Goals:**

- 不新增课堂干预计划表或复杂工作流状态机。
- 不要求教师填写完整课后报告。
- 不自动修改学生推荐或诊断输出，只先暴露可验证状态。
- 不在本轮处理跨作业、跨班级的长期教师动作效果画像。

## Decisions

### 1. 先从现有数据推导，不新增表

教师复盘反馈已经包含 `suggestionKey`、`actionType`、`targetAbility`、`exampleProblemId`、`evidenceTags`、`teacherNote` 和 `createdAt`。本轮新增 `TeacherInterventionImpactAnalyzer`，直接基于这些字段与后续提交/分析推导状态。

备选方案是新增 `teacher_intervention_plan` 表，把每次干预作为显式对象持久化。这个方案更完整，但会引入迁移、状态同步和 UI 流程成本。当前更需要的是把闭环跑通，先用已有反馈生成可观察状态。

### 2. 只把采纳和调整视为已执行介入

`ACCEPTED` 和 `MODIFIED` 表示教师准备使用或已经调整了 AI 建议，系统 SHALL 尝试评估后续效果。`DISMISSED` 表示教师否定该建议，不计入介入成效和 AI 质量风险，但可以在建议上显示“已忽略”。

这样可以避免把教师明确忽略的建议误判为“介入无效”。

### 3. 成效状态保持少而可解释

本轮使用以下状态：

- `NO_FEEDBACK`：还没有教师动作。
- `DISMISSED`：教师忽略了建议，不进入成效评估。
- `WAITING_FOLLOWUP`：教师已采纳或调整，但没有后续提交。
- `IMPROVED`：教师动作后同题或同证据范围的后续提交通过。
- `SHIFTED`：后续提交仍失败，但错因不再命中原建议证据标签。
- `STILL_STUCK`：后续提交仍命中原证据标签，需要升级或降低颗粒度。

状态判断优先级为：无反馈/忽略 → 无后续 → 后续 AC → 同类错因 → 错因转移。

### 4. AI 质量概览消费聚合后的介入成效

AI 质量概览新增 `TEACHER_INTERVENTION_LOOP` 维度。它不直接复用自然语言建议，而消费结构化计数：已执行介入数、等待后续数、改善数、仍卡住数、需要升级数。这样后续可以把维度接到趋势、评测或教师端质量看板。

## Risks / Trade-offs

- [Risk] 教师采纳按钮不等于真实课堂已执行 → Mitigation：状态文案明确为“基于教师反馈和后续提交的观察信号”，不声称因果证明。
- [Risk] 同题后续提交不足以代表全班复盘效果 → Mitigation：本轮只做作业内最小闭环，后续再扩展到跨题/跨班级长期画像。
- [Risk] 建议 key 变化导致找不到历史反馈 → Mitigation：继续使用已有稳定 key 规则；本轮不重写 key 生成。
- [Risk] 旧反馈缺少 evidenceTags 或 exampleProblemId → Mitigation：缺少标签时退化为题目和提交时间判断，返回等待/转移而不是抛错。
