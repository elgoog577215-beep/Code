## Context

`main` 当前页面更接近教学工具：学生知道下一题，题目页反馈更收敛，教师端 AI 质量不会压过课堂过程。但它的 `/app` 入口较弱，公共题库没有独立路径，教师端仍把总览和单作业细节放在一起。

`origin/feature/login-ui-redesign` 的优点是产品入口清楚、题库有浏览筛选、教师总览可以进入单作业详情。但该分支也有明显风险：路由未完全适配 `/app` base，部分学生首页使用 localStorage/mock 数据，教师 KPI 有占位值，并且夹带后端、AI 和静态构建产物。

## Goals / Non-Goals

**Goals:**
- 以 `main` 为唯一实现基线，吸收 origin 的交互模式而不是合并代码历史。
- 补齐公共题库、角色入口、邀请码快捷入口和单作业详情页。
- 保持学生主流程和题目页反馈主流程稳定。
- 保持教师端低噪音，把 AI 质量继续作为摘要或折叠信息。

**Non-Goals:**
- 不实现登录认证系统。
- 不引入 origin 的 mock 数据、localStorage 伪进度或占位 KPI。
- 不合并 origin 的后端、AI 路由、评测和静态构建产物。
- 不重写 UI 框架或业务数据流。

## Decisions

### 决策 1：保留 `/app` base，所有新增路由使用 `/app/*`

origin 的新页面使用了裸路径，如 `/problems` 和 `/teacher/assignment/:id`。在当前 Vite `base=/app/` 下，这些路径容易在浏览器中落回欢迎页。本轮新增路由必须使用 `/app/problems` 和 `/app/teacher/assignment/:id`，旧路径只作为兼容重定向。

### 决策 2：公共题库使用真实 API，失败时显示空状态

origin 的公共题库使用 mock fallback。本项目是教学工具，不能让 mock 数据冒充真实题库。公共题库页面只消费 `api.problemCatalog()`；失败或为空时展示可理解的空状态。

### 决策 3：教师端拆分为总览与单作业详情

`/app/teacher` 保持总览：作业选择、课堂摘要、需关注学生、高频问题。`/app/teacher/assignment/:id` 承载单作业详情：邀请码、KPI、问题分布、学生列表、教师校正。这样可以吸收 origin 的任务拆页优点，同时保留 main 的课堂过程优先级。

### 决策 4：不复制 origin 样式，抽取交互节奏后用 main 视觉系统重写

origin 的视觉有更强入口感，但整体偏“产品首页”。本轮使用 `main` 当前低噪音视觉系统，新增少量通用样式：角色入口、题库筛选、紧凑作业列表、单作业详情布局。

## Risks / Trade-offs

- 路由新增可能影响旧入口路径 -> 保留 legacy redirect，并覆盖 `/app` 与非 `/app` 常用路径。
- 教师页拆分可能造成重复逻辑 -> 先做展示层小拆分，复用现有 API 和格式化工具，不改变数据流。
- 题库页新增筛选可能增加样式复杂度 -> 保持搜索和难度筛选为唯一筛选维度。
- 当前工作区已有大量未提交后端/AI/OpenSpec 改动 -> 本轮只修改前端源代码与本变更目录，提交时仅选择相关文件。
