## Context

现有推荐事件已经记录 `EXPOSED / CLICKED / ENTERED_PROBLEM / SUBMITTED`，但行动效果分析只比较 `focusTags` 与后续诊断的首个粗细标签，并把后续 `ACCEPTED` 直接视为完成。系统同时已经拥有逐问题诊断事实、规范化问题键、标准库知识/能力/易错点、测试点语义快照和教师证据分析，因此无需新增事实源，只需把稳定身份带入推荐行动并统一验收。

作业创建目前由客户端检查基本字段，后端保存 `DRAFT / ACTIVE`，尚未使用已有测试点语义和标准库质量数据。教师端已有作业、题目和学生证据下钻，但缺少从未兑现建议直接形成的短队列。

约束：

- 生产 PostgreSQL 由 Flyway 管理，Hibernate 只做 `validate`。
- 标准库是教学参考与稳定锚点，不是诊断答案库。
- 没有真实学生效果数据，本轮只能验证工程契约与开发者样本，不能宣称教学效果。
- 正式入口固定为 `https://tuotuzju.com/code/`。

## Goals / Non-Goals

**Goals:**

- 让推荐行动携带可追溯且可比较的来源问题、标准库锚点和测试点意图。
- 让后续提交结果按稳定优先级给出可解释的观察性状态。
- 让教师只看到需要处理的最小队列，并可回到现有证据页。
- 在发布作业前复用既有内容质量资产，阻止明显不可诊断的题目进入正式课堂。
- 用固定样本覆盖新旧数据兼容、恢复、持续、复发和证据不足。

**Non-Goals:**

- 不新增第二套推荐、诊断、知识库或教师分析系统。
- 不自动替教师做教学决策，不把相关性描述成因果效果。
- 不因警告阻止保存草稿，不要求一次性回填所有历史推荐事件。
- 不直接揭示隐藏测试数据或把测试点意图当成学生错因。

## Decisions

### 1. 推荐行动采用“稳定身份优先、旧标签兜底”的分层证据契约

扩展 `student_recommendation_events` 保存来源规范化问题键、诊断问题 ID、知识节点 code、能力 code、易错点 code、目标测试意图 code 集合，以及后续提交对应的规范化问题键和失败测试意图快照。事件仍是唯一行动账本。

判断优先级固定为：

1. 同一规范化问题键或诊断问题身份；
2. 同一正式易错点、能力或知识路径；
3. 同一失败测试点语义 code / intent；
4. 仅对旧事件使用规范化标签兜底。

选择它而不是继续增强文本相似度，是因为稳定键可审计、可回归，也不会把文案变化误认为问题恢复。

### 2. `ACCEPTED` 是强证据，但不覆盖尚未完成的诊断

后续提交通过且没有同一稳定问题时，状态为 `CONTRACT_FULFILLED`；未通过且仍命中稳定关注点时为 `UNRESOLVED_SAME_FOCUS`；已有判题但正式诊断尚未落库时为 `WAITING_DIAGNOSIS`。高风险或同一关注点连续未解决达到门槛时标记教师介入。

分析响应增加 `matchBasis`、来源锚点、后续失败意图和证据引用，前端只展示短状态，不自行重算。

### 3. 教师干预队列是既有聚合的投影视图

不新建人工维护的干预表。后端从推荐事件、行动证据分析、学生和作业关系生成最多一页的队列，按“高风险持续未解决 → 复发/同类持续 → 已行动但无后续提交”排序。每项保留学生、作业、题目、推荐 token、原因、状态和提交链接。

这样避免队列与正式诊断事实漂移；未来若需要“已处理”工作流，再单独增加明确状态表。

### 4. 作业质量检查分为 blocker 与 warning，并以后端为准

新增只读预检服务与 API。对每个题目检查：

- blocker：题目不存在、无测试用例、正式测试点缺少已审核语义、语义 code 重复、知识/能力引用断链、隐藏揭示策略冲突；
- warning：意图类型过少、缺少知识或能力映射、缺少可用于教学起步的说明等不影响判题但降低教学质量的项目。

`ACTIVE` 创建必须在同一事务内再次运行预检并拒绝 blocker；`DRAFT` 允许保存并返回/预览警告。客户端预览只用于提前反馈，不承担最终安全判断。

### 5. 回归样本直接驱动证据分析器

新增结构化 fixture，覆盖稳定问题命中、标准库路径命中、测试意图命中、标签兼容、通过恢复、等待诊断、无后续提交和高风险教师介入。测试读取 fixture 构造事件并断言结果与 `matchBasis`，避免样本只成为不可执行文档。

## Risks / Trade-offs

- [历史事件没有稳定锚点] → 保留标签兜底并明确 `LEGACY_TAG`，不伪造回填。
- [诊断异步导致短暂误判] → 已提交但无正式后续问题证据时保持 `WAITING_DIAGNOSIS`，完成后重新聚合。
- [同能力下不同问题被误认为未解决] → 匹配结果暴露层级；问题键/易错点优先于能力与知识层，低粒度匹配只作为弱证据。
- [预检查询增加创建延迟] → 批量读取所选题目、测试点与标准库关系；创建时只检查当前选择。
- [教师队列过载] → 限定范围、稳定排序、去重 token，并把已兑现项排除。
- [生产迁移失败] → 发布前备份并校验 PostgreSQL，保留旧镜像和数据卷，迁移采用只增列/索引的向前兼容方式。

## Migration Plan

1. 新增 Flyway 迁移扩展推荐事件字段与必要索引，旧列保持不变且新列允许空值。
2. 本地运行迁移校验、后端测试、前端 lint/build 和开发者 fixture。
3. 生产发布前备份 `.env` 与 PostgreSQL，验证备份可读，并为旧应用镜像加回滚标签。
4. 加载/构建新镜像并用 `--no-build` 启动；校验 Flyway history、数据库质量门禁、页面、readiness 和 API。
5. 若应用验证失败，恢复旧镜像；新增空列不影响旧应用读取。数据库迁移不做破坏性自动回滚。
