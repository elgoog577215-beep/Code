## Why

当前 `online-judge` 已经积累了判题、课堂作业、学生身份、外接模型诊断、学生双通道反馈、学习轨迹、教师看板、AI 质量评测等能力，但这些能力仍然像一组叠加在 OJ 上的功能块，而不是一个清晰的教育 agent 核心系统。

真实学校使用场景中，学生并不需要看到复杂诊断报告，也不需要理解工程状态；老师也不需要被 AI runtime、fallback、smoke、profile 等术语淹没。系统真正要完成的是：把判题事实转化为学生可执行的下一步，把学生连续提交转化为学习轨迹，把班级提交分布转化为教师课堂判断。

本变更不是继续修补现有 OJ + AI 报告结构，而是重新定义主链路：判题事实层、学生 AI 快反馈层、学习轨迹层、教师洞察层。旧代码只作为可复用资产进入新架构；职责混乱的大对象、学生端废话展示、本地规则冒充 AI、教师端工程状态堆叠、从大报告里抽前端信息的模式，都不再主导新链路。

## What Changes

- 建立教育 agent 核心架构规划，把系统拆为四层：
  - 判题事实层：只负责 verdict、测试点、错误、首个失败点、代码证据。
  - 学生 AI 快反馈层：只负责修正建议、提升建议、追问、AI 状态，且学生可见建议必须来自模型结构化返回。
  - 学习轨迹层：记录学生多次提交的进步、回退、复发、AI 提示后的变化。
  - 教师洞察层：展示需关注学生、高频问题、共性误区、教师干预和 AI 质量证据。
- 明确旧代码复用策略：判题、题目、作业、学生、测试点、证据构建、安全校验、部分轨迹分析可以保留为底层资产；万能 `SubmissionAnalysis` 模式、从大报告抽学生反馈、本地规则生成学生建议、教师端工程状态展示退出主链路。
- 将验收体系改为三层：
  - 布尔门禁：判断结构、安全、状态和责任边界是否成立。
  - 质量 rubric：判断 AI 指导、学生体验、教师洞察是否有教学价值。
  - 观测指标：记录真实课堂使用中的速度、成功率、转化和复发趋势，初期不强行设阈值。
- 规划迁移路径：先定义目标架构和接口契约，再重建学生提交后的 AI 快反馈链路，再重建学习轨迹和教师洞察，最后清理旧 analysis 大对象和旧 UI 入口。
- 不在本规划阶段写业务代码；本变更先产出 proposal、design、tasks、spec，作为后续 implementation 的约束。

## Capabilities

### New Capabilities

- `education-agent-core-architecture`: 系统 SHALL 以教育 agent 核心架构组织判题事实、学生 AI 快反馈、学习轨迹和教师洞察，并提供可判断的迁移、验收和测试边界。

### Modified Capabilities

- 后续实施阶段将会修改学生反馈、学习轨迹、教师洞察和旧 analysis 相关能力；本规划阶段只定义目标态和实施约束。

## Impact

- OpenSpec：新增 `rebuild-education-agent-core` 的 proposal、design、tasks 和规格。
- 后端规划影响：后续将新增或重组判题事实、学生 AI 快反馈、学习轨迹、教师洞察的接口与服务边界。
- 前端规划影响：后续学生端以作业/做题/结果反馈为主链路，教师端以课堂判断为主链路，减少解释性废话和工程术语。
- 评测规划影响：后续测试不只看接口返回，还要覆盖模型结构化输出、反馈安全、rubric 质量、观测指标采集和浏览器主流程。
- 非影响范围：本规划阶段不改数据库、不改业务源码、不删除旧文件、不提交 API key、不推送。
