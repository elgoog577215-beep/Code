## Context

现有正式提交链路为 `JudgeService → StudentAiFeedbackAsyncService → StudentAiFeedbackService → SubmissionAnalysisService → DiagnosticAgentService → AiReportService`。外部 AI 部分已经遵守“先自由诊断 issues，再逐问题挂接标准库，最后生成建议”，但所有 issue 挂接仍在同一线程中串行执行，且整个异步任务只有 `GENERATING / READY / FAILED` 三种粗粒度状态。一次包含多个问题的真实提交会把每个问题的多轮导航依次排到关键路径上；任何进程重启或后段失败都可能重复已经成功的模型调用。

本次变更必须保持既有业务事实：标准库服务 AI 而非约束 AI；一个提交只有一个核心诊断判断；问题与建议数量由证据决定；学生历史结果和教师统计继续以持久化诊断事实为准。并行仅发生在核心诊断之后互不改变诊断结论的分支。

## Goals / Non-Goals

**Goals:**

- 把每次提交的完整 AI 诊断变成数据库可观察、可恢复、可审计的阶段工作流。
- 只执行一个核心诊断调用，产生本次提交唯一的规范 `issues[]`。
- 让不同 issue 的标准库挂接并行执行，同时保留每个 issue 内部必要的多轮顺序导航。
- 让学生反馈与教师洞察并行生成，并强制引用同一 `issues[]` 和证据集合。
- 让正式结果保存后的事实、生命周期、统计、成长候选和质量记录以幂等投影并行执行。
- 支持只重试失败阶段与进程重启后续跑，成功阶段直接复用。
- 为学生提供学习化阶段进度，不泄漏基础设施细节。

**Non-Goals:**

- 不增加多诊断 Agent、投票 Agent、复核 Agent 或推荐 Agent。
- 不增加普通、快速、降级质量路径，不使用规则结果伪装模型成功。
- 不设置每次提交或每个 issue 的 AI 调用次数上限；物理并发度只用于保护服务稳定性，不改变业务完整度。
- 不更换默认模型，不重写标准库内容与树结构，不调整问题生命周期定义、教师统计阈值、学生登录和教师鉴权。
- 不在本次变更中引入 Flyway/Liquibase；新增 JPA 表保持与当前部署方式兼容，后续生产化迁移单独治理。

## Decisions

### 1. 使用持久化 DAG，而不是继续扩展单个 `@Async`

新增 `AiDiagnosisRun` 与 `AiDiagnosisStageRun`。Run 记录提交、生成键、正式版本、总状态、当前阶段、核心输出版本和起止时间；Stage 以 `runId + stageKey` 唯一，记录阶段类型、issue ID、状态、尝试次数、模型、prompt 版本、输入摘要、输出 JSON、失败原因和耗时。

选择持久化阶段而非仅用 `CompletableFuture` 内存状态，是因为恢复、幂等和真实运行审计都需要跨进程证据。现有 `SubmissionAnalysis`、`StudentAiFeedbackRevision` 仍作为业务读模型；新表是编排事实，不替代历史反馈版本。

### 2. 正式版本按生成键追加，不覆盖历史结果

每次新的正式生成使用 `StudentAiFeedback.generationKey` 创建一个 Run，并为同一 submission 分配递增版本。阶段重试更新同一个 Run 的同一 Stage；用户明确发起全量重新诊断时才创建新 Run。Run 完成后标记为该 submission 的最新正式版本，但旧 Run 和 Stage 永久保留。

### 3. 一个核心诊断阶段是唯一问题事实源

`CORE_DIAGNOSIS` 只调用现有自由诊断协议，输入完整题目、完整代码、判题事实、学习历史与教师修正。合法输出保存为规范 `issues[]`。后续 `ISSUE_ATTACHMENT:*`、`STUDENT_OUTPUT` 和 `TEACHER_OUTPUT` 均只能消费保存的 `issues[]`，不得重新增删问题或形成第二套诊断结论。

### 4. 并行边界按依赖关系划分

工作流顺序为：

1. `CORE_DIAGNOSIS`；
2. 所有 `ISSUE_ATTACHMENT:<issueId>` 并行；
3. `STUDENT_OUTPUT` 与 `TEACHER_OUTPUT` 并行；
4. 证据与安全校验后保存正式分析和反馈版本；
5. `FACT_LIFECYCLE_PROJECTION`、`TEACHER_ANALYTICS_PROJECTION`、`LIBRARY_GROWTH_PROJECTION`、`RUNTIME_QUALITY_PROJECTION` 并行。

不同 issue 的挂接互不依赖，因此使用专用有界线程池并行；同一 issue 内的大章节、小章节、知识点和诊断层导航仍顺序执行。线程池并发度是部署资源配置，不是业务调用上限；排队的 issue 必须最终执行。

### 5. 成功阶段通过保存输出复用，失败阶段独立重试

阶段执行采用原子 claim：只有 `PENDING / RETRYABLE_FAILED` 可进入 `RUNNING`；`SUCCEEDED` 直接反序列化保存输出；其他节点看到 `RUNNING` 时不重复调用。异常按现有失败分类器标记为可重试或终态失败。重启恢复器扫描未完成 Run，把遗留 `RUNNING` 阶段改为可重试并重新入队。

不使用“整体失败后从头再来”的方案，因为它会重复核心诊断和已完成 issue 的多轮调用。也不删除最大导航轮数的结构安全门槛：该轮数限制用于防止状态机无限循环，不是 AI 总调用次数或问题数量限制；到达轮次时保留 `PARTIAL` 结果。

### 6. 学生输出与教师洞察是同事实的两个视图

学生输出沿用现有 advice 契约与安全校验；教师洞察新增轻量结构化输出，只允许总结 `issues[]` 的教学观察、证据和关注顺序，不得创建新 issue。两个分支并行运行，均保存阶段输出。教师洞察失败时必须显示为待重试，不能用模板伪造；已经成功的学生反馈可继续查看，但整个 Run 只有在所有必需 AI 阶段成功后才标记 `COMPLETED`。

### 7. 正式保存与派生投影分离

正式分析、反馈修订和规范诊断问题在同一收口步骤保存；后续投影按 Stage 独立、幂等执行。事实/生命周期投影是教师统计的来源，教师分析不直接读取模型临时输出。成长候选、推荐回填或质量记录失败只降低对应投影状态，不回滚正式诊断，也不把学生反馈改为失败。

### 8. API 返回稳定的学习化进度

`StudentAiFeedbackLookupResponse` 增加 `diagnosisProgress`，只暴露阶段代码、学习化状态、已完成/总数量、是否重试和更新时间。后端将内部阶段映射为“理解题目与证据、匹配知识路径、生成完整建议、核验证据与安全、完成/正在重试”；失败原因只返回可行动的学生文案，provider、key、token、HTTP 状态和模型内部错误仅保留在教师/运维记录中。

## Risks / Trade-offs

- [并行调用触发 provider 限流] → 使用独立有界执行器、可配置并发度和现有模型池重试；所有 issue 都保留在队列中，不以丢弃或调用上限换吞吐。
- [共享 RuntimePlan 在线程间产生竞态] → 并行阶段只共享不可变 brief、prompt 与规范 issues；每个 issue 使用独立导航局部变量，并在线程内激活调用上下文；聚合时按原 issue 顺序确定性排序。
- [阶段表与旧反馈状态短期不一致] → Run 为编排事实，旧反馈为兼容读模型；每次查询优先返回 Run 进度，完成后仍以保存的反馈修订为展示内容，并增加一致性测试。
- [进程在外部调用完成、阶段落库前崩溃导致重复调用] → 使用阶段 attempt 和请求指纹审计；无法对不支持幂等键的 provider 做严格 exactly-once，但恢复后只重试未落库阶段，并保留重复尝试记录。
- [教师洞察新增一次模型调用增加总消耗] → 用户明确要求完整路径且不限制调用次数；通过与学生输出并行降低关键路径耗时，并用真实样本比较完整性和总延迟。
- [现有 JPA 自动建表不满足生产迁移治理] → 本轮保持现有项目约束并添加索引/兼容启动测试；正式生产发布前另立数据库迁移治理变更。

## Migration Plan

1. 部署新增表、索引和默认执行器配置；旧记录不强制回填 Run。
2. 新提交开始创建 Run；没有 Run 的历史提交继续使用旧反馈/分析读取逻辑。
3. 启动恢复器只处理新表中未完成的 Run，不触碰旧 `READY` 历史反馈。
4. 观察阶段失败率、排队时长与 provider 限流后逐步调整物理并发度。
5. 回滚时关闭新工作流开关即可恢复旧异步入口；新表和历史 Run 保留，不做破坏性删除。

## Open Questions

- 生产数据库迁移工具与发布窗口不在本次范围内，后续需单独确定 Flyway/Liquibase 方案。
- 教师洞察的最终独立 UI 入口不在本次范围内；本轮先持久化并合入现有分析读模型，教师看板继续消费规范事实与投影。
