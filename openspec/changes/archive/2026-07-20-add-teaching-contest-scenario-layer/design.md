## Context

生产 V6 共有 737 个启用知识节点、140 个启用能力点、381 个易错点和 121 个提升点，结构已经能支持 AI 逐层导航，但 27 个能力点没有提升路径，其中 17 个是二分、DP、图、搜索、前缀差分、贪心、模拟、排序和映射结构等正式能力，另 10 个是历史兼容标签。现有诊断层只返回能力、易错和提升条目，缺少把能力转化为课堂形成性任务与竞赛约束任务的明确合同。

高中信息技术课程标准要求学生从实际问题出发经历算法设计、程序实现、调试运行和评价改进，并允许评价设置基本项与奖励项；CCF 的 NOI 大纲与官方题目知识构成分析又以知识难度、数据规模、部分分和算法应用检验竞赛能力。因此本轮要增加的是二者之间的应用桥梁，而不是继续复制同义知识点。

约束：

- 保持统一知识树和高中术语优先，不新增课堂/竞赛平行知识库。
- 标准库服务 AI 的命名、检索和迁移语境，不代替自由诊断或判题证据。
- 没有真实学生学习数据，只能验证结构、内容、读取链路和开发者回归，不宣称教学效果。
- 生产数据库由 Flyway 管理，发布必须可备份、可回滚且不触碰业务数据。

## Goals / Non-Goals

**Goals:**

- 建立显式、可审计、可被 AI 和教师真实读取的课堂—竞赛应用场景层。
- 用 17 组迁移对覆盖当前全部正式无提升路径能力，并让每个场景具备可观察证据和验收标准。
- 让最终建议生成读取相关场景，同时保持包体有界和问题证据优先。
- 通过迁移内门禁与快照审计保证场景成对、引用闭合、兼容快照同步和业务表不变。

**Non-Goals:**

- 不新增教学决策 Agent、推荐引擎或学生画像推断。
- 不根据开发者样本宣称掌握度、提分或迁移效果。
- 不把场景 code 变成学生诊断必须输出的新 ID。
- 不一次覆盖全部 737 个节点；首批聚焦生产已确认的 17 个正式能力缺口。

## Decisions

### 1. 新建应用场景表，而不是继续膨胀提升点文本

新增 `ai_standard_application_scenarios`。提升点回答“下一步练什么”，场景回答“在什么任务中练、观察什么、怎样验收、如何迁移”；两者职责不同。把所有内容压进 `practice_strategy` 会造成字段混义，也无法表达课堂—竞赛成对关系。

表以稳定 `code` 唯一标识场景，以 `transfer_pair_code + context_type` 保证每对恰好一个课堂端和一个竞赛端。场景通过字符串 code 引用知识点与能力点，数据库外键保证主引用存在，迁移门禁解析换行列表校验易错点和提升点属于同一能力。

备选方案是 JSONB 场景文档。它扩展灵活，但核心质量字段难以建立非空、长度和引用门禁，不利于教师治理和 SQL 审计，因此采用显式列。

### 2. 场景作为能力点的应用子层，不改变知识树层级

API 结构保持：

`知识点 → 能力点 → 易错点 / 提升点 / 应用场景`

课堂和竞赛差异存入 `context_type`、`learning_phase`、`constraint_profile` 和 `transfer_note`。两端复用同一个知识点和能力点，避免出现“高中二分”和“竞赛二分”两棵树。

### 3. 首批覆盖全部 17 个正式无提升路径能力

每个能力新增一条人工审校提升点，并用该提升点组织一组课堂—竞赛迁移对，共 17 条提升点、34 个场景。兼容能力继续只服务旧标签映射，不接收新增正式内容。迁移结束后“无提升路径能力”允许剩余 10 个，但它们必须全部是 `SK_COMPAT_%`。

课堂端使用真实活动结构：任务情境、学生产物、形成性证据、教师追问和成功标准。竞赛端使用真实约束结构：题面抽象、规模梯度、失败类型、边界/复杂度验证和部分分到完整解的迁移。

### 4. 教师导航与 AI 最终包使用同一场景来源

新增 JPA 实体与仓储。`AiStandardLibraryService.expandDiagnosticLayer` 按能力嵌入场景，供教师端和逐层导航读取；`StandardLibraryNavigationPackBuilder` 根据已选能力和知识点再次有界查询，把场景同时放到 `knowledgeGroups.skillUnits[].applicationScenarios` 和顶层兼容列表。

最终包最多 12 个场景，按 `sort_order, code` 稳定去重。场景查询失败或为空不改变挂接状态。提示词明确场景只用于生成检查动作和迁移建议，合法诊断 ID 仍只来自知识点、能力点、易错点和提升点。

备选方案是只扩展教师 API。这样数据不会进入学生建议主链，无法证明场景层真正服务 AI，因此拒绝。

### 5. Flyway V7 同步规范提升点与兼容结构

V7 先建表和索引，再写 17 条规范提升点，同步 `ai_standard_library_items` 和 `ai_standard_library_legacy_mappings`，最后写 34 个场景。SQL 使用稳定 code 和 `ON CONFLICT`，但 Flyway 仍只执行一次；幂等片段验证用于确认内容脚本没有数量漂移。

迁移内 `DO` 门禁检查：

- 17 条新增正式提升点和 34 个场景数量准确；
- 17 个迁移对各有 `CLASSROOM` 与 `CONTEST`；
- 知识点、能力、易错、提升引用全部存在且同属一个能力；
- 正式能力全部至少一条提升路径，剩余缺口全部是兼容能力；
- 课堂端和竞赛端关键质量字段非空且文本不是模板占位；
- 规范提升点、兼容快照和 legacy mapping 一致。

## Risks / Trade-offs

- [场景字段较多，人工维护成本上升] → 首批用 17 个核心迁移对验证字段价值；质量门禁只要求真正用于观察和验收的字段，避免为了字段数量填空话。
- [场景包增加模型上下文] → 只按已选能力查询并限制 12 条；不在自由诊断前倾倒全库。
- [官方标准没有逐条规定本地教学活动] → `source_framework` 区分课程标准、NOI 大纲和本地教学推导，`review_status` 明确 `SOURCE_ANCHORED` 与 `INFERRED_REVIEWED`。
- [新表只被教师端读取，AI 链路名义接入] → 为最终 `StandardLibraryPack` 和提示词增加专门回归测试，验证 JSON 中确实出现相关场景且无关场景不出现。
- [生产磁盘空间紧张] → 发布前审计镜像与构建缓存，只做范围明确的应用镜像构建；禁止广域 prune，保留数据库 Volume 和可用回滚镜像。

## Migration Plan

1. 在生产 V6 上执行只读缺口审计并保存关键表计数。
2. 用生产 PostgreSQL dump 建立本地临时数据库，验证 V7、引用门禁、业务表不变和二次内容执行数量稳定。
3. 运行 V1→V7 空库迁移、定向测试、完整后端测试、OpenSpec 校验和静态差异检查。
4. 提交并推送代码；生产发布前创建 PostgreSQL 备份、校验 SHA-256、保留当前应用镜像标签。
5. 安全构建新应用镜像，只替换 app 容器；Flyway 自动执行 V7。
6. 验证 Flyway、34 个场景、17 个迁移对、正式能力提升路径、导航 API、AI prompt 包、核心业务表计数和 HTTP/readiness。
7. 若失败，停止应用、恢复备份与旧镜像后重新启动；不执行 `down -v` 或任何 Volume 清理。
