## Context

当前提交主链路已经先写入 `submissions`，随后保存测试点、完整 `submission_analyses`、学生可见 `student_ai_feedbacks` 和 `student_ai_feedback_events`。问题不在于完全没有数据库记录，而在于四个断点：学生题目页只加载历史摘要但不恢复最近提交和反馈；课堂提交允许只带作业 ID 而没有学生 ID；分析结果主要以整段 JSON 保存，教师统计需要反复解析；教师前端还会根据标签关键词推测知识路径并混用提交次数、涉及学生数和在册人数。

现有教师主线已经确定为“班级 -> 作业 -> 题目”的递进分析，AI 信息只负责客观归因、路径和证据；现有学生反馈已经区分正式、临时、推断和未归类知识路径。本设计延续这些边界，不重做 AI 诊断 Agent，也不另建一套学生画像真源。

## Goals / Non-Goals

**Goals:**

- 让每次提交、评测、AI 反馈、反馈查看和同题后续提交都能被持久化、恢复和追溯。
- 保证课堂提交一定绑定合法学生、班级、作业和题目上下文。
- 让学生重新进入题目时看到当前范围内最近一次结果，并能重新打开历史结果与建议。
- 保留完整诊断快照，同时生成适合统计的逐问题事实，避免教师端猜测路径或临时解析全部 JSON。
- 统一教师端指标口径，使班级、作业、题目和学生近期状态都能下钻到原始提交证据。
- 通过幂等回填兼容历史数据，并明确呈现无法可靠补齐的缺失状态。

**Non-Goals:**

- 不改变现有 issue-first AI 诊断、标准库逐层导航和学生建议生成逻辑。
- 不用本地规则在 AI 失败时伪造诊断内容或知识路径。
- 不把学生归类为固定能力类型，不生成不可解释的综合分数或长期标签。
- 不把“查看反馈后的改善”解释为 AI 反馈造成的因果结果。
- 不在本次变更中完成 H2 到 Postgres 的生产迁移，但新增结构必须同时兼容两者。
- 不把教师分析主线改成教学动作推荐系统。

## Decisions

### 1. 课堂提交上下文由后端解析，前端 ID 只作为请求意图

课堂作业提交必须从学生令牌解析当前学生，再校验学生所属班级、作业所属班级、题目是否属于该作业。请求中的 `studentProfileId` 只能与令牌身份一致，不能决定最终归属；请求中的 `assignmentId` 存在时，任一关系不合法都拒绝提交。公共题库继续允许登录学生提交，也允许匿名练习，但匿名记录不进入教师课堂统计和学生画像。

选择该方案而不是仅给 DTO 字段增加非空校验，是因为字段非空仍无法阻止伪造其他学生 ID、跨班作业或不属于作业的题目。

### 2. 学生历史按使用场景分区，而不是只按题目查询

课堂题目历史使用“学生 + 作业 + 题目”作为范围；公共题库历史使用“学生 + 无作业 + 题目”，匿名公共练习只在匿名范围内读取。题目页首次加载时先取得范围内历史摘要，若存在记录，再并行读取最近提交详情和对应学生反馈。只有反馈处于生成中或过期生成状态时继续轮询；读取已成功或已失败的历史记录不得重新调用模型。

选择显式范围参数而不是继续复用“学生 + 题目”，是为了防止同一道题在不同作业中复用时串历史，也避免课堂结果与公共练习混在一起。

### 3. 保留当前快照接口，同时增加追加式反馈版本

`student_ai_feedbacks` 继续作为每个提交的当前展示快照，保持现有 API 兼容。新增反馈版本记录，每次进入生成终态时追加保存完整学生可见反馈、状态、失败原因、模型/提示词/结构版本、生成时间和关联分析版本；重新生成不得覆盖旧版本。当前快照只指向或复制最新可展示版本。

选择“当前快照 + 追加版本”而不是直接把现有表改成多行，是为了避免破坏现有按 `submission_id` 唯一查询，同时满足审计和历史对比。

### 4. 完整 JSON 是审计真源，逐问题事实是统计真源

完整 `submission_analyses.report_json` 与反馈 JSON 保留不变。新增逐问题诊断事实，每条记录至少包含提交、分析版本、稳定问题 ID、问题层级、是否主要问题、知识节点、能力点、易错点或提升点、完整 breadcrumb、路径状态、标准库归属、证据引用、置信度和生成时间。唯一键以“分析版本 + 稳定问题 ID + 事实类型”为基础，保证同一分析重放不会重复计数。

教师校正不覆盖原始 AI 事实，而是作为校正层叠加；教师端默认展示有效事实，同时保留原始值和校正来源供追溯。

选择结构化事实而不是直接在教师请求中解析所有 JSON，是为了稳定统计口径、支持索引和回填，并避免前后端各自实现一套标签解释。

### 5. 学习痕迹保存事实，学生近期状态按需派生

反馈查看事件继续使用追加式事件记录，并关联提交和反馈版本。系统按同一学生、同一题目、时间顺序把查看后的下一次提交作为观察样本，比较评测阶段和规范化问题集合，得到：等待后续、查看后改善、仍卡同类问题、问题转移、回退、评测阶段变化或暂无明确变化。

学生近期状态从限定时间窗口和独立提交事实派生，输出证据数量、涉及题目数、重复情况、最近变化和置信边界，不在 `student_profiles` 上写入永久的“薄弱学生”标签。

### 6. 教师指标全部由后端按明确分母聚合

教师聚合服务直接读取在册学生、提交、诊断事实、校正和反馈事件，返回已经计算好的路径桶、计数、比例和证据样本。前端只负责展示和粒度切换，不再根据中文关键词推断章节、知识点或能力点。

核心口径固定为：

- 在册人数：班级学生表中的有效学生数。
- 提交人数：当前范围内至少一次合法课堂提交的去重学生数。
- 尝试次数：当前范围内合法提交总数。
- 学生通过率：至少一次通过的学生数 / 提交人数。
- 尝试通过率：通过提交数 / 尝试次数。
- 错误出现次数：去重后的“提交 + 稳定问题 ID”数量。
- 涉及学生数：命中该问题的去重学生数。
- 重复错误学生数：该学生在至少两次独立提交中命中同一规范化问题。
- 恢复率：存在可比较后续提交的样本中，后续不再命中同一问题的学生数 / 可比较学生数。
- 数据完整率：具备合法身份、评测结果、诊断状态和可解释归因的提交数 / 合法提交数，并分别暴露缺失原因。

### 7. 回填采用幂等投影，未知信息不猜测

迁移任务从现有提交、分析、反馈和事件读取历史数据：能够从学生令牌历史关系、正式标准库 ID 或后端 breadcrumb 可靠恢复的字段才写入；缺少学生身份的课堂提交标记为身份缺失，缺少可靠路径的诊断事实标记为 `UNCLASSIFIED` 或历史 `INFERRED`。回填记录批次、游标、成功数、跳过数和失败原因，可重复运行而不增加计数。

### 8. 数据读取分阶段切换并保留回滚开关

先新增表与双写，再回填历史数据并对比新旧聚合，最后把学生历史和教师看板切换到新接口。切换期间保留旧读取路径作为只读回滚开关；回滚只切换读取，不删除新事实和版本记录。

## Risks / Trade-offs

- [历史课堂提交缺少学生身份，无法可靠归属] → 保留记录并标记身份缺失，排除出学生画像和去重学生统计，只计入数据完整性告警；不得按姓名或时间猜测绑定。
- [完整 JSON 与逐问题事实发生漂移] → 事实投影与分析保存使用同一事务边界或可靠重试，记录分析版本和投影状态，并提供一致性检查。
- [新增逐问题事实增加写入量] → 使用批量写入和必要索引；教师查询只读取结构化事实，不扫描源码和完整 JSON。
- [同一问题的模型 ID 不稳定导致重复统计] → 优先使用稳定 issue ID 和正式标准库 ID；库外问题使用规范化父路径、类型和代码生成可重放指纹，并保留来源状态。
- [恢复率样本过少造成误导] → 同时返回分子、分母和证据数；样本不足时显示“证据不足”，不展示孤立百分比。
- [版本表无限增长] → 首期保留全部版本以满足审计；上线前根据学校数据保留政策确定归档周期，不在业务层直接删除。
- [课堂提交校验上线后暴露旧前端遗漏] → 先让前端稳定携带作业上下文并完成端到端测试，再启用后端强校验；错误信息明确指出身份、作业或题目关系问题。

## Migration Plan

1. 新增反馈版本、逐问题诊断事实、投影状态和必要索引，保持现有表与接口可用。
2. 在提交完成时同步创建当前反馈状态；在分析和反馈进入终态时双写版本与逐问题事实。
3. 新增带作业范围的学生历史查询，并完成学生题目页最近结果恢复和历史入口。
4. 运行幂等历史回填，输出身份缺失、诊断缺失、路径未归类和投影失败统计。
5. 新增教师后端聚合响应，与现有概览结果做样本级、班级级口径对比。
6. 切换教师看板到后端路径桶和精确分母，删除前端关键词路径推断。
7. 通过功能开关启用课堂提交强校验；观察拒绝原因后再移除旧宽松路径。
8. 回滚时关闭新读取和强校验开关，恢复旧只读接口；保留新增表和双写数据，待修复后重新启用。

## Open Questions

- 正式学校部署前需确认反馈版本和学习事件的保留年限、学生离校后的匿名化规则及教师可见范围。
- 当前历史匿名课堂提交是否需要人工认领，应由学校数据治理决定；系统默认不自动绑定。
