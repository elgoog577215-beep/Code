## Context

当前系统已有完整的推荐事实链：`StudentRecommendationService` 聚合学生画像、诊断事实、问题生命周期和教学动作，返回排序后的建议；`StudentRecommendationEventService` 记录曝光、点击、进入题目和再次提交；推荐效果已经进入教师与 AI 质量分析。缺口集中在学生首页：页面没有请求推荐接口，也没有把推荐转换成清晰的学习行动，导致强后端能力停留在隐藏状态。

本变更跨越学生首页、API 契约、事件追踪、双语文案、样式和测试，但不改变诊断事实源与推荐算法。

## Goals / Non-Goals

**Goals:**

- 学生登录后第一屏即可看到当前最高优先级的下一步学习行动。
- 行动同时回答“做什么、为什么、完成标准是什么”，并能回到真实题目继续学习。
- 推荐令牌贯穿点击、进入题目与再次提交，继续使用现有推荐效果证据链。
- 对加载、空、失败、不可跳转和低证据状态做诚实表达。
- 中英文、移动端、键盘和屏幕阅读器均可用。

**Non-Goals:**

- 不新增第二套推荐 Agent、学生画像或学习事实表。
- 不改变核心 AI 诊断流程、标准库协议或模型供应商。
- 不在本阶段自动替教师布置任务，也不把推荐描述为强制教学决策。
- 不以推荐接口失败推断学生已经掌握。

## Decisions

### 1. 复用现有推荐接口作为唯一事实源

学生首页直接调用 `/api/student/profile/{id}/recommendations`，不在前端重新根据提交、标签或通过率推断下一步。这样推荐策略、证据、事件和教师效果分析保持同源。

替代方案是前端从 `reviewCards` 拼接建议；该方式无法保留策略、优先级、完成信号和推荐令牌，因此不采用。

### 2. 单一主行动优先，其他建议折叠为候选

页面按后端返回顺序选取第一条为主行动，突出一个明确入口；其余建议作为“之后可以做”保留，避免多个同权按钮制造选择负担。前端不重新计算优先级。

### 3. 路由由推荐对象确定并保留令牌

有 `problemId` 时进入对应作业题或公共题，URL 携带 `studentProfileId` 与 `recommendationToken`；只有 `assignmentId` 时进入作业；无可执行目标时展示建议但不伪造按钮。点击主行动前记录 `CLICKED`，题目页继续记录 `ENTERED_PROBLEM`，提交时继续携带令牌。

### 4. 完成信号是一等信息

`expectedCompletionSignal` 直接展示为“完成标准”，`learningHypothesis` 作为系统判断依据的补充；缺少字段时不生成模板式假完成标准。风险与回退动作仅在后端提供时展示。

### 5. 失败与空状态分离

空建议表示当前没有可靠的个性化行动，页面引导学生继续当前作业或公共练习；请求失败表示暂时无法读取，提供重试入口。两者不得共用“已掌握”文案。

## Risks / Trade-offs

- [已有推荐文案质量不一致] → 前端只负责清晰编排，不改写事实；通过契约测试保证关键字段和空缺状态可读，后续再以真实行为证据调优推荐算法。
- [学生首页额外请求影响首屏] → 推荐与题库、作业并行加载，独立骨架和错误边界，不阻塞原有入口。
- [推荐目标可能缺少作业上下文] → 公共题使用公共路由，作业题仅在 `assignmentId` 存在时构造作业路由；无法安全构造时不展示伪入口。
- [首页信息密度增加] → 只突出一个主行动，其余建议限制为简洁次级列表。
- [运行数据文件已有本地改动] → 实施和提交显式排除 `onlinejudge.mv.db` 与 `run.pid`。

## Migration Plan

1. 先补充前端行为契约测试，锁定推荐加载、主行动、令牌路由和失败状态。
2. 接入现有 API 并实现学生首页区域、双语文案和响应式样式。
3. 运行前端类型检查、测试、构建和浏览器验收。
4. 无数据库迁移；发布失败时可回退前端提交，既有推荐接口和事件不受影响。

## Open Questions

- 后续是否把“完成标准”升级为学生可勾选/回答的验证动作，应根据本阶段点击、进入题目、再次提交和问题恢复数据决定。
- 教师是否需要直接向学生置顶某个行动，属于后续教师干预闭环，不在本变更内提前设计。
