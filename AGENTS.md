# Project Agent Guide

本项目默认中文协作。目标不是多写流程，而是把在线判题系统的 AI 能力做成可验证、可迭代、可沉淀的教育 agent。

## Core Loop

```text
读上下文 -> 判断当前能力瓶颈 -> 选择最小但有结构增益的改动 -> 实现 -> 评测 -> 沉淀
```

- AI 核心能力优化必须同时考虑：错因分析、学习状态、提示安全、教师可解释性、评测闭环。
- 遇到“继续优化 AI / 改变与革新 / 更像教育 agent / 提升颗粒度”这类请求时，不要只改提示词文案；先判断是否需要新增结构化状态、证据链或评测目标。
- 改动优先保持兼容：先新增字段、信号、fixture、评测门槛，再逐步让前端和教师端使用它们。

## AI Innovation Heuristic

这条经验来自一次关键迭代：用户只说“执行下一部分优化，要做一定程度上的改变与革新”，最终落点不是继续微调单次提交错因分析，而是把 AI 从“单次提交诊断”推进到“多次提交学习轨迹分析”。

以后遇到类似模糊但高层级的 AI 革新请求时，按下面方式判断：

1. 先看当前能力是否已经在旧维度接近饱和。
   如果单次提交的标签、提示、安全、评测已经稳定，再继续堆规则或提示词，收益会变小。

2. 问一个更深的问题：教育场景里还缺哪个关键观察变量？
   单次代码只能回答“这次哪里错”，但老师真正关心的是“学生是在进步、回退、反复卡住，还是通过后该复盘”。这说明缺的是时间维度和学习轨迹。

3. 把“革新”翻译成 agent 状态空间升级。
   不是让一句提示更聪明，而是新增系统能稳定识别的状态，例如：
   - 编译错误是否已经修好
   - RE 是否推进到 WA
   - 是否多次卡在同类错误
   - 是否从失败推进到 AC
   - 是否从 AC 回退到失败

4. 把新状态做成结构化输出，而不是藏在自然语言里。
   优先新增类似 `learningTrajectorySignal` 的字段，让后端、前端、教师端、评测集都能复用。

5. 同步建立评测闭环。
   新能力必须有 fixtures 和断言验证，尤其要覆盖真实或半真实的多次提交链，而不是只看单条错误代码。

6. 避免帮倒忙。
   不要一上来重写整条 AI 管线；先做可回退、可观察、兼容旧结果的增量结构升级，并跑旧评测防止原能力下降。

## Why Trajectory Beat Prompt Tweaks

这次没有优先“优化提示词模板”，原因是：

- 提示词优化容易提升表面表达，但不一定增加系统可观测状态。
- 当前问题不是 AI 不会说，而是系统没有明确表示“学生学习过程”的结构。
- 教育 agent 的关键不是单次回答更像老师，而是能根据连续提交调整下一步教学动作。
- 多次提交轨迹可以直接服务教师看板、学生复盘、个性化推荐和后续 coach agent。

因此，真正的高杠杆改动是：

```text
单次错因标签
-> 错因标签 + 学习轨迹 phase + 证据引用 + 老师介入判断 + 轨迹评测集
```

## Future AI Upgrade Pattern

后续继续优化 AI 时，优先寻找新的“教育状态变量”或“反馈闭环”，例如：

- 学生是否理解了提示后的追问
- 同一细粒度错因是否跨题重复
- coach 对话是否改善了下一次提交
- 哪类错误需要老师介入而不是继续自动提示
- AC 后是否完成了复盘迁移

每次新增能力都要回答：

- 新增了哪个可观测状态？
- 这个状态对学生或老师有什么教学价值？
- 输出结构在哪里？
- 证据引用在哪里？
- 用什么 fixture 或测试验证？
- 是否保持旧能力不下降？

## Verification Defaults

AI 能力改动至少跑相关后端测试和诊断评测；涉及前端类型时跑 `npm run typecheck`。不要为了验证刷新大量静态构建产物，除非本轮任务确实需要前端构建交付。
