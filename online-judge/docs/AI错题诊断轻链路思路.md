# AI 错题诊断轻链路思路

## 1. 核心判断

这套系统的目标不是把 AI 变成一个被规则绑死的填表机器，而是让外部大模型像一个真正懂信息学教学的老师一样，完成：

- 读题。
- 读代码。
- 看判题结果。
- 对照细颗粒知识树。
- 找出当前最主要的问题。
- 用学生能听懂的话给出分析和建议。

标准库的作用是服务 AI，而不是制约 AI。

它应该提供知识坐标、易错点名称、能力点边界和标准化表达，让 AI 的判断更细、更稳、更可追踪；但不能要求 AI 只能在标准库里“选答案”。如果标准库没有覆盖真实问题，AI 应该可以明确说“未命中”，并给出自己的判断。

## 2. 轻链路总览

理想链路可以简化为两段：

```text
学生提交
-> 搜索定位
-> 完整诊断与建议
-> 学生反馈
```

其中：

- 搜索定位：帮助 AI 找到最相关的知识点、能力点、易错点。
- 完整诊断与建议：让 AI 基于题目、代码、判题结果和候选知识坐标，自由完成有深度的教学分析。

这条链路不追求 Agent 数量多，也不追求流程复杂，而是追求每一步都真的增加质量。

## 3. 搜索定位阶段

### 3.1 这一阶段要做什么

搜索定位阶段负责从标准库中找到“可能相关”的条目。

输入包括：

- 题目描述。
- 学生代码。
- 编程语言。
- 判题结果。
- 编译错误、运行错误、超时信息。
- 样例差异或测试点概况。
- 本地规则能提取出的简单信号。

输出包括：

- 候选知识点。
- 候选能力点。
- 候选易错点。
- 候选提高方向。
- 每个候选为什么被召回。

### 3.2 这一阶段不应该做什么

搜索定位阶段不直接给学生建议。

它不应该替诊断 Agent 下最终结论，也不应该说：

- 学生一定错在这里。
- 必须这样修改。
- 这个条目就是最终答案。

它只提供候选范围，让下一步诊断更聚焦。

### 3.3 标准库命中状态

搜索定位需要允许三种状态：

```text
HIT      标准库命中，候选条目能覆盖主要问题
PARTIAL  半命中，标准库覆盖了一部分，但不完整
MISS     未命中，标准库没有覆盖当前真实问题
```

这三个状态不是学生看到的最终诊断，而是系统内部判断标准库覆盖度的信号。

它们的意义是：

- HIT：可以放心让诊断围绕候选条目展开。
- PARTIAL：诊断要允许 AI 补充标准库之外的判断。
- MISS：不能强迫 AI 套标准库，应记录为标准库成长线索。

## 4. 完整诊断与建议阶段

### 4.1 这一阶段要做什么

诊断阶段是真正面向学生产生价值的阶段。

它需要整体理解：

- 题目到底要求什么。
- 学生代码想怎么做。
- 当前行为和题目要求差在哪里。
- 哪些是阻塞 AC 的基础层问题。
- 哪些是修好之后值得提升的提高层问题。
- 学生下一步最应该做什么。

### 4.2 这一阶段应该怎样使用标准库

标准库只提供坐标，不提供死答案。

诊断 Agent 应该：

- 优先参考搜索定位给出的候选条目。
- 能把命中的问题绑定到标准库 ID。
- 能说明证据来自代码、题目还是判题结果。
- 如果候选条目不合适，可以明确不采用。
- 如果发现标准库没有覆盖的问题，可以输出“库外问题”。

这样既能保持标准化，又不会限制 AI 的判断能力。

### 4.3 学生反馈应该长什么样

学生看到的反馈不应该是数据库字段拼出来的文本。

它应该更像一个老师的讲解：

```text
基础层

1. 主要问题：输入结构读错了
你现在只读了一个 n，但题目第一行其实有 n 和 q 两个数。
这会导致后面的数组和查询行整体错位，所以程序不是小边界问题，而是输入模型没有对齐题目。

建议你先不要急着改算法，先把题目的输入分成三层：
第一行 n、q；第二行 n 个数；后面 q 行查询。
然后检查你的代码是不是每一层都读到了。

提高层

1. 可以进一步思考：多次区间查询怎样更快
当输入里有很多个 l、r 查询时，每次都重新求和会变慢。
修好输入之后，可以考虑先预处理前缀和，让每次查询只用常数时间得到答案。

下一步

先用一个很小的样例手推输入读取过程，确认每一行输入都被正确消费。
```

这里可以有结构，但不能变成生硬填空。

## 5. 后端保障阶段

轻链路不是完全不校验，而是只校验真正必要的东西。

后端应该检查：

- 是否直接给出完整可复制答案。
- 是否引用了不存在的标准库 ID。
- 是否引用了不存在的证据。
- 是否把判题结果说反了。
- 是否在未 AC 时完全没有基础层问题。
- 是否生成了明显空泛的建议。

后端不应该强行改写 AI 的自然语言表达。

如果 AI 已经写出一段清楚、自然、有证据的解释，后端应该保留它，而不是拆成很多小字段再重新拼接。

## 6. 标准库成长阶段

标准库成长不是主诊断链路的一部分，它应该在后台异步发生。

触发来源包括：

- 搜索定位 MISS。
- 搜索定位 PARTIAL。
- 诊断 Agent 发现候选库外问题。
- 多个学生在同类问题上反复出现库外判断。
- 教师手动标记某次诊断很有价值。

成长 Agent 可以提出：

- 新增知识点。
- 新增易错点。
- 拆分过粗条目。
- 合并重复条目。
- 修改条目名称或学生解释。

但它不应该无审查地直接改坏主库。

更稳的方式是：

```text
诊断结果
-> 成长候选
-> 自动去重和质量检查
-> 教师或管理员确认
-> 写入标准库
```

如果后续确实要给成长 Agent 更大权限，也应先有版本记录、回滚机制和变更说明。

## 7. 为什么要从复杂 Agent 链路收回来

之前的复杂链路容易产生几个问题：

- 阶段太多，每一步都可能损失信息。
- 结构化字段太细，导致学生反馈像填表。
- 标准库被当成约束，而不是参考坐标。
- 后端校验太硬，可能把 AI 的好判断打掉。
- 运行链路难以排查，模型能力和系统问题混在一起。

轻链路的核心是：

```text
让系统负责提供事实和边界
让标准库负责提供知识坐标
让大模型负责真正分析和表达
让后端负责底线校验
让成长机制负责长期沉淀
```

## 8. 之后开发的方向

下一步不应该急着继续堆 Agent。

更值得优先验证的是：

1. 单提示词直连强模型，能不能给出明显更自然、更准确的诊断。
2. 如果能，说明当前复杂链路主要问题在系统设计，而不是模型能力。
3. 再把单提示词里有效的部分沉淀成正式轻链路。
4. 最后才考虑哪些地方需要重新引入结构化、检索、校验和成长机制。

因此，对照实验应该单独进行，不写进本文档本身。
