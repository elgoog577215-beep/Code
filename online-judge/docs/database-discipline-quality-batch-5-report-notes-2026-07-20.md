# 第五批学科数据库升级证据说明

## 结论

第五批不再只是“把知识点和提升点写得更细”，而是新增了一层可被教师与 AI 真实读取的应用合同：

```text
知识点
└── 能力点
    ├── 易错点
    ├── 提升点
    └── 应用场景
        ├── 课堂：怎样形成能力、观察什么、教师怎样检查
        └── 竞赛：怎样读约束、验证边界、从部分分迁移到完整解
```

本批覆盖生产 V6 中全部 17 个正式无提升路径能力，新增 17 条提升路径和 17 组课堂—竞赛迁移对，共 34 个应用场景。迁移后正式能力缺口为 0；剩余 10 个无提升路径能力全部是历史 `SK_COMPAT_*`，没有用兼容标签承载新教学内容。

## 为什么这一批更贴近教学与竞赛

第四批主要解决“标准覆盖范围、知识点文本和具体易错行为”。第五批继续向下走一步，明确一项能力在真实任务中的完整闭环：

- 课堂端保存任务情境、学生产物、可观察证据、常见失败、教师追问、学生自检和成功标准。
- 竞赛端保存题面抽象、数据范围、复杂度预算、边界类型、部分分梯度和赛后最小反例复盘。
- 两端复用同一个知识点和能力点，用 `transfer_pair_code` 显式成对，不建立“高中库”和“竞赛库”两棵树。
- AI 只在当前问题已经被代码和判题证据支持后，才使用场景细化检查问题与提高建议；场景中的常见失败不能反推学生真的犯了该错。

场景分布不是同一模板复制 34 次：包含 6 个概念发现、11 个引导练习、6 个竞赛读题、9 个实现任务和 2 个赛后复盘；难度包含 8 个基础、14 个中等和 12 个进阶场景。

## 一个分支的升级前后对比：答案二分

### V6 之前

`ALGO.BINARY.ANSWER.check_函数 → SK_BINARY_ANSWER_CHECK` 已经能描述“写出单调可行性检查”，也已有两个细颗粒易错点：

- `MP_BINARY_CHECK_EQUAL_CASE_REJECTED`：临界值恰好满足时被误判为不可行。
- `MP_BINARY_LOWER_BOUND_TOO_LOOSE`：答案下界只写 0 或负无穷，脱离题意。

但这个能力没有正式提升路径，也没有说明教师怎样观察学生是否真正理解、竞赛中怎样把理解迁移到大范围约束。AI 可以命名问题，却缺少可直接检查的训练语境。

### V7 之后

新增提升点 `IP_DQ5_BINARY_FEASIBILITY_BOUNDARY_TABLE`：

- 先选不可行值、临界值和可行值，手算 `check`。
- 再证明真值只发生一次切换。
- 最后从题意推出上下界、等号归属和搜索方向。

课堂场景 `SC_DQ5_BINARY_ANSWER_CLASSROOM` 不让学生先抄二分模板，而是让学生为三个连续候选值填写“必要条件—可切段数—check 真值—等号归属”表。教师观察的是学生能否口述单调方向、能否解释临界值两侧为何改变，而不是代码长得像不像模板。

竞赛场景 `SC_DQ5_BINARY_ANSWER_CONTEST` 把同一能力放入 `n≤2×10^5、值域≤10^9` 的最小化最大负载任务。学生必须先比较枚举与“线性 check + 二分”的操作量，再写出候选答案语义、必要下界、充分上界，并用 `x-1 / x / x+1` 自测。这样课堂中的边界表被迁移为竞赛中的复杂度与边界合同。

这条分支现在形成：

```text
判题与代码证据
→ 二分 check 能力
→ 临界等号 / 下界过松易错点
→ 可行性边界表提升点
→ 课堂观察任务 + 竞赛约束任务
→ AI 生成可检查的基础修复与提高建议
```

## 17 组真实覆盖

本批覆盖答案二分、二分模型、结构化 DP、计数 DP、图松弛、拓扑排序、贪心、前缀区间、二维前缀差分、BFS 层次、搜索状态、模拟同步、排序比较、滑动窗口、图建模、哈希频次和 set/map/排序选择。每组都绑定生产库中的真实能力、真实易错点和本批提升点。

其中算法领域 14 组，数据结构领域 3 组。没有新增同义知识节点或能力点，数量增长全部发生在“改进动作”和“应用语境”层。

## AI 链路的实际接入

- 教师展开知识点诊断层时，每个能力点返回自己直接关联的课堂与竞赛场景。
- 最终 `StandardLibraryPack` 升级为 v2，在能力点内嵌场景，并提供最多 12 条的有界顶层场景列表。
- 单能力最多向嵌套结构提供 2 条场景，正好保留一组课堂—竞赛迁移对。
- 场景查询失败不改变已有标准库挂接状态，也不阻断其他问题。
- prompt 明确场景 code 不是合法诊断 ID，`commonFailure` 不是当前提交证据；题目、学生代码、判题事实和 `evidenceRefs` 始终优先。

## 隔离验证结果

当前生产 V6 备份已通过 SHA-256 校验，并在本机隔离 PostgreSQL 16 恢复。V7 原始 SQL 连续执行两次后仍为：

| 指标 | V6 | V7 |
|---|---:|---:|
| 启用提升点 | 121 | 138 |
| 正式无提升路径能力 | 17 | 0 |
| 兼容无提升路径能力 | 10 | 10 |
| 应用场景 | 0 | 34 |
| 课堂—竞赛迁移对 | 0 | 17 |
| 结构阻断指标 | 0 | 0 |

题目 16、测试用例 45、提交 53、分析 34、诊断事实 85、学生反馈 53，迁移前后完全一致。真实 Flyway 从 V6 升到 V7 后，Hibernate schema validation 通过；空库 V1→V7、重复启动、显式基线和 Schema 漂移阻断通过；完整后端回归 660 个测试通过，失败和错误均为 0，跳过 9 个；OpenSpec 严格校验和全量 109 项校验通过。

## 证据边界

本批证明的是：

- 数据库已经能表达教学形成、竞赛迁移和可观察验收。
- AI 与教师导航确实能读取这些场景。
- 正式能力的提升路径缺口已经闭合。
- 迁移可重复、可审计，并且没有改变业务事实。

本批不能证明学生已经学得更好，也不能证明竞赛分数提高。34 个场景仍需要真实课堂观察、教师校正和后续提交行为来验证有效性。下一步不宜立即继续平均铺更多场景，而应先观察哪些场景被 AI 使用、教师是否认可、学生后续是否出现可比较变化，再决定第六批优先级。
