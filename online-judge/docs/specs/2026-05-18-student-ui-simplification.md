# Spec: 学生端 UI 简化与适配

## Why
学生端是校内课堂工具，不是产品介绍页。学生进入页面后只需要完成一条路径：输入邀请码，确认身份，选择题目，提交代码，查看结果。当前页面仍有解释性文案、侧栏优先级偏高、题目卡片信息偏多、移动端首屏路径不够短的问题。

## Goals
- 入口页只保留邀请码输入和进入作业操作。
- 未确认身份时，身份确认优先于题目列表。
- 已确认身份后，题目列表优先于学生信息和提交记录。
- 题目卡片只保留题号、题目名、必要状态和开始按钮。
- 提交记录只显示完成数、提交次数、最近状态和需要继续关注的信息。
- 手机端按单列任务流展示，平板/桌面端保持主内容与辅助信息分区。

## Non-goals
- 不改后端 API、数据库、评测逻辑。
- 不新增学习功能、AI 功能、排行榜或营销展示。
- 不改变题目页代码提交流程。

## Scenarios
### Scenario: 学生输入邀请码
GIVEN 学生打开学生端且未进入作业  
WHEN 输入老师提供的邀请码  
THEN 页面进入对应课堂作业，不出现多余说明。

### Scenario: 学生未确认身份
GIVEN 学生已进入作业但没有确认身份  
WHEN 查看页面  
THEN 先看到身份确认表单，再看到题目列表；开始按钮保持不可用或明确提示先确认身份。

### Scenario: 学生已确认身份
GIVEN 学生身份已确认  
WHEN 查看作业页  
THEN 先看到题目列表和开始按钮；学生信息、提交记录作为辅助信息显示。

### Scenario: 手机端使用
GIVEN 学生使用手机访问  
WHEN 打开学生端  
THEN 页面单列展示，按钮高度适合触控，没有横向溢出，首屏路径清楚。

## Impact
- Frontend: 修改 `StudentPage.tsx` 和学生端 CSS。
- Backend: 无。
- Database: 无。
- AI / standard data: 无。
- Deployment: 仅前端构建产物变化。

## Risks
- 身份确认与题目列表顺序调整后，需确保学生未确认身份时不能误以为可以直接开始。
- 删除说明文案后，错误提示必须仍然明确。
- 移动端压缩信息密度后，题目名和状态标签不能溢出。
