# AI 标准库质量报告（V5 前置分析）

生成日期：2026-06-19

## 1. 报告目的

本报告用于回答一个核心问题：当前 AI 标准库是否已经足够支撑高中信息学代码诊断，如果还不够，下一轮应该优先补哪里。

本轮不评价外部大模型调用质量，不设计 RAG，也不改变学生端展示。报告只聚焦标准库本身：

- 知识树覆盖是否完整。
- 能力点和易错点是否足够细。
- 哪些章节主要依赖自动生成，缺少高质量手写条目。
- 下一轮扩张应该先补哪些高价值薄弱区。

## 2. 当前结构结论

当前标准库已经从旧的“基础层/提高层条目”转向更合理的结构：

```text
知识树 -> 能力点 -> 易错点
```

这个方向是正确的。标准库不应该替 AI 写死诊断结论，而应该提供细颗粒知识框架和标准化错因名称，让 AI 根据题目、代码、判题结果自己分析，并把结果落回标准库 ID。

当前核心文件：

- 知识树：`src/main/java/com/onlinejudge/learning/knowledge/application/InformaticsKnowledgeSeedCatalog.java`
- 标准库：`src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibrarySeedCatalog.java`
- 质量测试：`src/test/java/com/onlinejudge/learning/standardlibrary/AiStandardLibrarySeederTest.java`

## 5. 当前主要收益

### 5.1 知识树已经有了高中信息学骨架

当前知识树覆盖了程序基础、数据结构、算法思想、数学建模、调试工程、竞赛策略六个领域。它已经不是零散错因库，而是一个可以继续沉淀的学科框架。

### 5.2 标准库已经具备全量兜底

每个细知识点都有自动生成的能力点和易错点，因此不会出现某个知识点完全没有标准库支撑的情况。

这对后续召回有价值：哪怕某个知识点还没有手写高质量条目，系统也可以先召回一个基础版本，避免 AI 完全没有标准语言可用。

### 5.3 高质量样本方向是对的

已经存在一批比较好的手写样本，例如：

- 循环边界。
- 数组下标。
- 输入输出结构。
- 二分答案。
- DP 状态。
- 图建模。
- 函数返回。
- 矩阵行列。
- 并查集。
- 拓扑排序。
- 位运算。
- 读题建模。

这些条目的共同特点是：不是泛泛地说“掌握某知识点”，而是把学生真实会错的动作拆出来。

## 6. 当前主要问题

### 6.1 自动生成条目占比过高

当前总条目 1213 条，其中自动生成能力点和易错点合计 1112 条，占主体。

自动生成条目的作用是兜底覆盖，但它的表达仍偏概括。它可以告诉 AI “这里有这个知识点”，但不一定能帮助 AI 精准区分学生到底错在什么具体动作上。

例如“字符串匹配基础”自动生成条目可以覆盖主题，但它还不够细到：

- 查找返回 -1 后直接当下标使用。
- 子串长度和右端位置混淆。
- 回文判断只比较一半但边界错。
- 统计重叠出现次数时跳步错误。

### 6.2 一些高频主题缺少手写条目

统计显示，下面这些主题当前手写关联为 0 或明显偏少：

| 主题 | 问题 |
| --- | --- |
| `BASIC.STRING.CHAR` 字符处理 | 字符数字转换、大小写、特殊字符处理还不够细 |
| `BASIC.STRING.MATCH` 字符串匹配基础 | 查找、统计、前后缀、回文的错因未充分展开 |
| `BASIC.STRING.BUILD` 字符串构造 | 拼接效率、结果顺序、删除替换容易出错 |
| `BASIC.ARRAY.TRAVERSE` 数组遍历 | 同步遍历、逆序遍历、局部遍历缺少典型错因 |
| `BASIC.ARRAY.UPDATE` 数组更新 | 原地覆盖、临时数组、交换、累计更新仍需加强 |
| `BASIC.BRANCH.IF` if 条件 | 多分支归属、条件顺序、默认分支仍偏粗 |
| `BASIC.EXPR.LOGIC` 逻辑表达式 | 与或非、短路、德摩根、条件组合需要补细 |
| `ALGO.SIM.STATE` 模拟状态设计 | 状态变量选择和同步是学生高频错点 |
| `ALGO.SIM.PROCESS` 流程模拟 | 事件顺序、规则优先级、终止条件需要细化 |
| `ALGO.SIM.CORNER` 模拟边界 | 空状态、首尾、同一时刻冲突需要加强 |
| `ALGO.ENUM.LOOP` 枚举设计 | 枚举对象、范围、去重、组合排列需要加强 |
| `ALGO.ENUM.PRUNE` 剪枝 | 剪枝条件写错会导致漏解，需单独展开 |
| `ALGO.BINARY.INDEX` 位置二分 | lower_bound、upper_bound、返回边界仍需加强 |
| `ALGO.PREFIX.DIFF` 一维差分 | 边界加减、还原、离线处理错因不足 |
| `ALGO.PREFIX.MATRIX` 二维前缀和 | 容斥、坐标偏移、补零边界错因不足 |
| `ALGO.DP.CLASSIC` 经典 DP | 背包、线性、区间、树形 DP 仍太粗 |
| `ALGO.GREEDY.INTERVAL` 区间贪心 | 排序依据和边界重叠需要补 |
| `ALGO.GREEDY.STRUCTURE` 结构化贪心 | 优先队列、延迟选择、资源分配需要补 |
| `ALGO.GRAPH.MST` 最小生成树 | Kruskal、连通性、边排序错因需要补 |

### 6.3 错因类型分布不够均衡

按自动分类统计，当前 556 个细知识点对应的错因类型大致为：

| 错因类型 | 数量 |
| --- | ---: |
| MODELING 建模依据不清 | 160 |
| CONCEPT 适用条件混用 | 135 |
| BOUNDARY 边界或下标偏移 | 110 |
| STATE 状态定义不完整 | 70 |
| COMPLEXITY 规模估算不足 | 25 |
| IO_FORMAT 输入输出映射错误 | 21 |
| VALUE_RANGE 数值范围估算不足 | 15 |
| RUNTIME 运行出口或资源保护不足 | 15 |
| SYNTAX 语法规则未落实 | 5 |

问题不在于这些分类本身错误，而在于 `MODELING` 和 `CONCEPT` 容易变成大筐。后续要把一部分概念型错因继续细拆成更可诊断的具体动作。

例如：

```text
CONCEPT: 适用条件混用
```

应该继续拆成：

- 把有序性当成天然存在。
- 把单调性当成样例规律。
- 把局部最优当成全局最优。
- 把连通图算法用于非连通图。
- 把无权最短路和带权最短路混用。

### 6.4 标准库还没有质量报告机制

目前质量检查主要在测试里验证：

- 条目数不低于阈值。
- 引用的知识点存在。
- 易错点关联的能力点存在。
- 不出现明显空泛名称。

这很好，但还不够。下一步应该让系统能持续回答：

- 哪些章节缺少手写高质量条目。
- 哪些主题只有自动生成兜底。
- 哪些错因类型过度集中。
- 哪些能力点没有足够易错点。
- 哪些易错点太像泛化描述。

## 7. 下一轮补库优先级

### P0：模拟与读题建模

优先原因：高中信息学大量题目不是纯算法难，而是规则翻译和状态维护错。

建议补：

- 状态变量选错。
- 状态更新顺序错误。
- 多对象状态不同步。
- 事件优先级误读。
- 终止条件多一步或少一步。
- 同一时刻冲突处理错误。
- 空状态和首尾状态漏处理。

### P0：字符串与数组细节

优先原因：这是初高中学生最常提交的基础错误，AI 诊断必须说得很细。

建议补：

- 字符和数字互转。
- 字符串查找失败返回值。
- 子串起止位置和长度混淆。
- 回文双指针边界。
- 重叠子串统计。
- 拼接顺序错误。
- 原地更新覆盖旧值。
- 双数组同步遍历错位。

### P1：DP 经典类型

优先原因：DP 是提高层诊断的核心，不能只停留在“状态定义不清”。

建议补：

- 01 背包正序遍历导致重复取。
- 完全背包逆序遍历导致无法重复取。
- 计数 DP 初始项错误。
- 区间 DP 枚举长度顺序错误。
- 线性 DP 答案位置取错。
- 树形 DP 父子状态混淆。

### P1：二分、前缀差分、贪心

优先原因：这些是 CSP-J/S 和高中校内训练常见题型，错因具有明显模式。

建议补：

- 位置二分返回值含义混淆。
- `lower_bound` 与 `upper_bound` 混用。
- 差分右端 `r + 1` 漏减。
- 二维前缀和容斥符号写反。
- 贪心排序关键字错误。
- 区间重叠边界归属错误。

### P2：图论和数据结构

优先原因：这部分对提高层建议重要，但可以放在第二批扩张。

建议补：

- BFS 最短路入队时机错误。
- Dijkstra 负权误用。
- 拓扑排序环检测遗漏。
- 最小生成树连通性未检查。
- 栈队列空容器访问。
- map 默认值和不存在键处理。
- 堆排序方向和优先级定义错误。

### P2：调试与提交复盘

优先原因：这是教育 agent 和普通判题系统拉开差距的地方。

建议补：

- 只看样例不构造反例。
- 不会缩小错误输入。
- 调试输出影响格式。
- 变量轨迹没有跟循环同步记录。
- AC 后没有复盘复杂度和边界。

## 8. 建议新增的报告能力

下一步建议把这份静态报告沉淀成后端可生成的质量报告，至少包含：

```text
StandardLibraryQualityReport
├── totalItems
├── domainCoverage
│   ├── domainCode
│   ├── knowledgePointCount
│   ├── skillUnitCount
│   ├── mistakePointCount
│   ├── manualSkillCount
│   └── manualMistakeCount
├── weakTopics
│   ├── topicCode
│   ├── topicName
│   ├── knowledgePointCount
│   └── manualLinkCount
├── mistakeTypeDistribution
└── recommendations
```

先做内部服务和测试即可，不必马上做前端页面。

## 9. 扩张原则

后续补库应遵循以下原则：

1. 不为了数量而补。
2. 每个新增能力点必须是一个具体动作。
3. 每个新增易错点必须是学生真实会犯的具体错误。
4. 易错点不要写成完整建议，完整建议交给 AI 结合代码生成。
5. 标准库条目必须能被召回、能被模型引用、能被教师理解。
6. 优先补高频、可诊断、能影响 AI 输出质量的点。

## 10. 推荐下一步

建议按下面顺序推进：

1. 新增标准库质量报告服务和测试，把本报告中的统计自动化。
2. 按 P0 先补模拟、字符串、数组更新、读题建模。
3. 再补 P1 的 DP、二分、前缀差分、贪心。
4. 每补一批，都更新质量报告，观察薄弱项是否减少。
5. 等标准库质量稳定后，再进入召回和上下文组装。

当前结论很明确：标准库已经有骨架和规模，但下一阶段竞争力来自“手写高质量细颗粒条目占比”和“质量报告驱动的定向扩张”，而不是继续无差别堆数量。
