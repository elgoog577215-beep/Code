# 错误经验

## 2026-07-09 学生端展示契约字段不能在完整链路映射时丢失

- 背景：删除独立快反馈链路后，学生端 AI 弹窗仍保留“代码证据”和“知识路径”的前端渲染能力，但用户连续看到“暂无可跳转代码行”和知识点标签消失。
- 触发：`StudentAiFeedbackService` 改为调用完整诊断链路后，只映射了建议正文、标准库 ID 和证据 refs，遗漏了学生端 UI 已依赖的 `knowledgePath`，导致前端虽然仍读取 `item.knowledgePath`，但后端返回空字段。
- 结论：学生端反馈 API 是稳定展示和观测契约。以后替换 AI 生成链路、删除快链路或调整 mapper 时，必须逐项保留 `repairItems/improvementItems` 内的 `knowledgePath`、`evidenceRefs`、`evidenceSnippets`、标准库 ID、`libraryFit`、`latencyMs` 和安全/失败状态；不能只验证“有建议文字”。
- 复用：排查“前端标签/证据突然没了”时，先抓 `/student-ai-feedback` 响应确认字段是否为空，再查 `StudentAiFeedbackService` 从完整链路 DTO 到 `StudentAiFeedbackResponse` 的映射，不要先在前端硬补。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/application/StudentAiFeedbackService.java`、`online-judge/src/test/java/com/onlinejudge/submission/application/StudentAiFeedbackEventTest.java`、`online-judge/frontend/src/features/problem/ProblemPage.tsx`。

## 2026-07-09 学生端代码证据不能只依赖模型主动返回 code ref

- 背景：学生端 AI 反馈卡片显示“评测结果，暂无可跳转代码行”，但建议正文已经提到具体函数和代码表达式，用户期望能定位到对应代码行。
- 触发：前一次修复 `FREE_DIAGNOSIS` 时，缺证据 fallback 优先保留 `judge:first_failed_case`，后续 advice item 可能只有评测证据，没有 `code:line:N` 或 `code:range:A-B`，导致 `StudentAiFeedbackService` 无法生成 `evidenceSnippets`，前端只能降级显示“暂无可跳转代码行”。
- 结论：学生端最终 API 出口必须保证“能定位的建议尽量带代码证据”。不能只依赖模型主动返回 `code:*`；应优先复用 analysis 全局代码证据，其次用反馈文本里的真实代码标识符到提交源码中匹配行，再生成 `code:line:N` 和 snippet。
- 复用：以后看到前端出现“暂无可跳转代码行”，先查该 item 的 `evidenceRefs/evidenceSnippets`。如果正文含函数名、变量名或表达式但 refs 只有 `judge:*`，应修后端证据补全，不要在前端硬造跳转。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/application/StudentAiFeedbackService.java`、`online-judge/src/test/java/com/onlinejudge/submission/application/StudentAiFeedbackEventTest.java`。

## 2026-07-09 AI 链路可用性要区分模型额度、字段漂移和证据锚点

- 背景：把学生端 AI 生成切到完整外部模型链路后，用户连续看到 AI 反馈不可用，且更换大模型后仍有题目失败。
- 触发：`Qwen/Qwen3-235B-A22B-Instruct-2507` 先从 0/3 修到 2/3，随后 ModelScope 返回今日配额 429；`Qwen/Qwen3-Coder-30B-A3B-Instruct` 能调用但 `output-format` 卡在 `FREE_DIAGNOSIS`。
- 过程：逐层拆开 live eval 报告和 rawPreview 后确认，模型本身已经能识别 `print(..., end=" ")` 导致尾随空格；失败点是内部 DTO 只接收严格字段名，真实模型会输出 `evidenceRef`、`nextStepPlan.action`、`diagnosisIssues/name/reason/impact` 等别名，且证据 fallback 只硬编码 `judge:first_failed_case`，无法匹配当前 prompt 中的 `judge:first_failed_case:1`。
- 结论：AI 链路“不能用”不能直接归因于模型小或 prompt 差。外部模型链路要先判断三类问题：provider 额度/限流、模型 JSON 字段漂移、证据引用锚点不一致。可恢复的字段别名应在 Jackson 反序列化前修复；证据 fallback 必须从本次 prompt 的 ordered evidence refs 中选择真实存在的 ref。
- 复用：以后看到 `FREE_DIAGNOSIS` 或 `ADVICE_GENERATION` 失败，先查 rawPreview 是否已有高质量诊断内容；若内容好但状态失败，优先补 normalizer/alias/evidence fallback，而不是盲目换模型。模型选择上，235B 适合长期默认，额度耗尽时应用明确展示 429/配额耗尽；30B Coder 可作为当日可用性兜底，但要用 live eval 证明完成率。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/application/AiReportService.java`、`online-judge/src/main/java/com/onlinejudge/submission/application/FreeDiagnosisOutput.java`、`online-judge/src/main/java/com/onlinejudge/submission/application/AdviceGenerationOutput.java`、`online-judge/src/test/java/com/onlinejudge/submission/application/AiReportServiceAdviceGenerationRuntimeTest.java`。

## 2026-07-09 advice 安全重写不能只依赖模型自我纠错

- 背景：五道高级长代码 live 评测中，字段别名和建议数量已修复后，仍有两题在 `ADVICE_GENERATION` 第二次安全重写后失败。
- 错误：只靠 prompt 和二次请求要求模型“不要泄露答案”，模型仍会输出学生可见触发词，例如“将边的结束时间设为当前时间”“最终答案”，导致 validator 以 `SAFETY_RISK` 拒收整题。
- 结论：validator 不能放松；应在 `AdviceGenerationOutputNormalizer` 这个学生可见文案汇合点做受控清洗，去掉标题重复、直给修复语气和已知安全触发词，再交给 validator 严格校验。
- 复用：以后 live 报告出现 `studentReport/basicLayerAdvice contains answer leak`，先从 trace 精确定位触发字段和触发词，再把可恢复的表达清洗放到 normalizer；完整代码、参考答案等高风险内容仍应由 validator 拦截。

## 2026-07-09 真实模型评测不能让标准库挂接调用数随 issue 和目录深度放大

- 背景：正式标准库 seed 在 H2 中启用后，`LIBRARY_EMPTY` 消失，但五题 live 评测一题就可能因每个 issue 多轮标准库挂接触发外部模型 429；后续模型还暴露 `code:L118`、`visibleCaseFacts`、空 anchor evidenceRefs 等结构漂移。
- 错误：只修空库或只换模型，会把问题从空库变成调用爆炸、证据别名误杀或 advice anchor 空证据失败。Maven live 命令通过也不能代表业务达标，必须继续解析 report 和 trace。
- 结论：标准库挂接应有调用预算：默认只对首个 issue 做导航，其他 issue 仍进入 advice；轮次耗尽但已有目录路径时记 `PARTIAL`，不是整题失败。EvidenceRefSupport 要吸收真实模型常见证据别名，Advice validator 对可恢复的空 anchor evidenceRefs 可从 primary evidence 补齐。
- 复用：以后五题 live 验收必须看 `MODEL_COMPLETED`、`FREE_DIAGNOSIS:OK`、`LAYERED_ATTACHMENT`、`ADVICE_GENERATION:OK`、基础/提高建议数和安全触发词；不能只看 Maven `BUILD SUCCESS`。

## 2026-07-09 全量测试依赖历史内容时必须显式开启 seed

- 背景：项目已决定生产默认关闭 `app.content-seed.enabled`，正式内容应来自数据库。但部分旧集成测试仍依赖知识树、标准库和教师端管理列表的历史 seed 内容。
- 错误：只跑局部 AI 链路测试会通过；一跑 `./mvnw -Dskip.frontend=true test`，依赖历史内容的测试会因为 H2 空库返回空树、空标准库列表或找不到相似条目而失败。
- 结论：测试如果验证“历史 seed 内容本身”或基于这些内容的管理接口，应在 `@TestPropertySource` 中显式设置 `app.content-seed.enabled=true`；测试如果验证新运行时主链路，则应自建最小 fixture，不能默认依赖启动 seed。
- 复用：以后看到知识树根节点为空、标准库列表为空、成长候选无法继承相似条目的 skillUnit，先检查测试是否应该显式开启 seed，而不是重新打开生产默认 seed。

## 2026-07-08 真实模型结构漂移不能只靠 prompt 约束

- 背景：高级长代码真实评测中，模型虽然理解了问题，但多次没有严格遵守完整 JSON schema：`outOfLibraryFindings` 返回字符串数组，`basicLayerAdvice` / `improvementLayerAdvice` 只返回 `text + evidenceRefs`，证据引用还会写成 `code:line_160`、`code:Line-160`、`judge:case_1_output` 等变体。
- 错误：只加强 prompt 或只让 validator 拒绝，会导致“信息足够但结构略偏”的真实输出被整体判为失败，最终学生端没有任何 advice。这会掩盖真正目标：保留可校验证据上的多条建议。
- 结论：对学生可见 AI 链路，应把 schema 分成硬字段和可恢复字段。证据不可校验、安全风险、空内容属于硬失败；`text-only advice`、字符串型库外发现、大小写/短横线证据别名、缺省 `confidence` 或 `nextStepPlan` 属于可恢复结构，应在 normalizer 中补齐或规范化。
- 复用：以后新增 AI 输出 schema 时，同步增加真实模型结构漂移测试；validator 负责拦截危险或无法确认的内容，normalizer 负责把可确认内容整理成内部稳定结构。

## 2026-07-08 外部模型测试必须先确认实际生效的 key 和模型

- 背景：五题 live 初次报告显示 ModelScope `401 Authentication failed`，但根因不是 `.env` 的 `MODELSCOPE_API_KEY` 失效，而是 shell 中已有的 `AI_EVAL_API_KEY` 优先级更高并覆盖了 `.env`。显式设置 `AI_EVAL_API_KEY="$MODELSCOPE_API_KEY"` 后，外部 API smoke 返回 200。
- 错误：只检查 `.env` 里 key 长度相同，不等于确认测试进程实际使用了那个 key。遇到 401/429 时，如果不先打印“变量名 + 长度 + 生效优先级”，容易把配置优先级问题误判成服务端不可用。
- 结论：真实模型评测前必须确认最终传入 Spring 测试的 `AI_EVAL_API_KEY`、`AI_EVAL_MODEL` 和 base URL；ModelScope 单模型 429 时，先用 `/models` 和小 token smoke 选可用模型，再跑长代码样本。
- 复用：后续跑真实 AI 评测，优先命令形态为 `AI_EVAL_API_KEY="$MODELSCOPE_API_KEY" AI_EVAL_MODEL='...' ./mvnw ...`，避免 shell 中旧 eval key 抢优先级。
