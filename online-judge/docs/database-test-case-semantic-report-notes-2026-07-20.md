# 第六批数据库升级：测试点语义层

## 结论

这一轮找到的增长点不是继续横向加知识点，而是补上此前一直缺失的“判题证据解释层”：

```text
题目
└── 测试点
    ├── 原始事实：输入、期望输出、公开性、顺序
    └── 语义事实：评测意图、学习目标、竞赛角色、揭示策略
        └── 知识点 → 能力点
```

升级前，数据库能告诉 AI “第几个测试点没通过”，却不能告诉它“这个点在测什么”。升级后，16 道正式题的 45 个测试点全部完成逐点人工审校，覆盖 7 类评测意图、29 个知识点和 26 个能力点；60 条历史判题结果也全部绑定到确定的测试点，并保存语义快照。

这不是新增一条 AI 旁路。主链仍然是：

```text
判题事实
→ AI 自由诊断
→ 标准库挂接
→ 基础 / 提高建议
→ 数据库存储
```

测试点语义只是第一步里的中性事实。它帮助模型理解失败点的目的，但不能替代代码证据，也不能直接证明学生犯了某种错误。

## 为什么这是新的增长点

前几轮已经把“知识点—能力点—易错点—提升点—课堂/竞赛场景”做得更细，但判题层仍只保存输入输出。于是出现一个结构断层：

- 标准库知道“有哪些能力和易错点”。
- 场景库知道“课堂怎样形成、竞赛怎样迁移”。
- 判题结果却只知道“过或没过”，不知道每个测试点承担什么验证任务。

继续扩标准库只能让候选更多，不能让 AI 获得更好的当前提交证据。测试点语义层把评测设计本身变成数据库中的一等对象，让诊断从“看到失败编号后猜原因”升级为“知道失败点的中性目的，再回到代码中找证据”。

这一设计吸收了三个外部依据：

- ACM/IEEE-CS CS2023 把测试计划、测试数据、边界与角落情况视为独立的软件工程能力。
- ICPC/Kattis Problem Package Format 明确区分样例与隐藏测试数据，并支持描述性测试组织和测试组角色。
- EEF 的反馈框架强调反馈要连接学习目标与当前证据之间的缺口。

这些依据只决定结构原则；45 个测试点的具体语义仍由当前题目、输入约束、标准库路径和教学目标逐条人工审校。

## 一个真实分支：质数判断

### V7 之前

`质数判断` 有五个测试点。第 4 个隐藏点是：

```text
999983 → YES
```

数据库只知道它是隐藏的第 4 个测试点。因为隐藏数据不能传给 AI，模型既看不到 `999983`，也不知道这个点是为了检查定义边界、一般正确性还是复杂度。最终只能根据代码和“某个隐藏点失败”自行猜测。

### V8 之后

同一个测试点现在拥有稳定语义：

| 字段 | 内容 |
|---|---|
| `semantic_code` | `TCI_PRIME_LARGE` |
| 评测意图 | `PERFORMANCE` |
| 意图标题 | 大质数复杂度 |
| 学习目标 | 从因子成对性质推出平方根试除上界，并估算判定复杂度 |
| 竞赛角色 | `COMPLEXITY_STRESS` |
| 揭示策略 | `AI_GENERALIZED` |
| 标准路径 | `MATH.NUMBER.PRIME.试除法 → SK_PRIME_TRIAL_DIVISION` |

AI 可以收到的只是泛化事实：“这是接近规模上界的复杂度压力点，用于检查不必要的操作量。”它仍然看不到 `999983`、`YES` 或任何可反推隐藏答案的数值。

因此诊断逻辑变成：

```text
隐藏复杂度测试点失败
→ 这是一个中性线索，不是错误结论
→ 模型检查学生代码是否线性试除、循环上界是否合理
→ 只有找到具体代码行或运行行为证据，才形成自由诊断 issue
→ 后端再把 issue 挂接到试除法能力与相关标准库内容
```

这是真正的学科升级：不是给 AI 预塞标准答案，而是让它知道“评测者为什么设计这个点”，同时保留独立阅读代码和举证的责任。

## 45 个测试点现在覆盖什么

| 评测意图 | 数量 | 主要用途 |
|---|---:|---|
| `REPRESENTATIVE` | 17 | 代表性正常路径与基础算法合同 |
| `BOUNDARY` | 10 | 最小值、端点、空区间与定义边界 |
| `EDGE_CASE` | 8 | 非典型但合法的正确性陷阱 |
| `STRUCTURAL` | 7 | 数据结构、图关系、状态转移与输入结构 |
| `STATE_SPACE` | 1 | 判重键与状态维度是否完整 |
| `SCALE` | 1 | 数据规模增长下的实现稳定性 |
| `PERFORMANCE` | 1 | 复杂度上界与不必要操作量 |

23 个公开测试点采用 `PUBLIC_EXAMPLE`，22 个隐藏测试点采用 `AI_GENERALIZED`。每道题至少包含两类互补评测意图，避免把三五个测试点写成同一目标的数字替换。

## 数据库与 AI 链路改变

- Flyway V8 为 `test_cases` 增加稳定语义 code、评测意图、学习目标、竞赛角色、揭示策略、知识点、能力点、审核状态、来源和版本。
- `submission_case_results` 保存 `test_case_id` 与完整语义快照，后续修改测试点时不会篡改历史判题语境。
- 新判题会直接写入测试点身份与语义快照；V8 对已有 60 条判题结果做确定性回填，60/60 均成功映射。
- `free-diagnosis-v2` 只接收自然语言测试意图和 `judge:test-intent:*` 证据引用，不接收知识点 code 或能力点 code，避免自由诊断被标准库先入为主。
- `diagnosis-report-v4` 明确要求：测试意图只能帮助理解失败点，结论仍必须由代码、编译、运行或判题事实支持。
- readiness 与部署脚本新增测试点语义门禁；缺字段、重复 code、非法枚举、隐藏原值泄露、跨知识—能力路径、单题意图单一或历史快照缺失都会阻断发布。

## 证据边界

当前仍没有可用于教学效果评估的真实新学生数据，因此本轮能证明的是：

- 评测意图已经成为结构化、可审计、可回溯的数据库事实。
- AI 确实能安全消费泛化测试意图，同时看不到隐藏原值和标准库 ID。
- 历史与新判题都能保留当时的测试点语境。
- 迁移、约束、质量门禁和回归链路稳定。

本轮不能证明学生学得更好，也不能证明竞赛成绩提高。下一阶段应观察真实提交中“哪类测试意图经常触发有效诊断、教师是否认可、学生同题再提交是否修复”，再决定扩充测试点、调整语义，还是回修题目本身。
