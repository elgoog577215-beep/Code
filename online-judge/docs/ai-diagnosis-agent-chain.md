# AI 错题诊断 Agent 链路与后端设计说明

## 1. 定位

这份文档描述在线判题系统中 AI 错题诊断链路的工作过程。它同时面向两类读者：

- 人：理解这个系统如何分析学生错题。
- AI/工程实现者：根据本文对齐后端边界、状态、失败处理和验收标准。

本文吸收了评审意见后的核心判断是：

```text
学生实时链路要短，系统责任边界要硬。
```

学生感受到的是两步：

```text
搜索定位 -> 诊断建议
```

系统内部实际按五层责任运转：

```text
事实层 -> 召回层 -> 诊断层 -> 保障层 -> 成长层
```

三个 Agent 的位置是：

| Agent | 所在层 | 是否实时阻塞学生 |
|---|---|---|
| 搜索定位 Agent | 召回层 | 是 |
| 诊断建议 Agent | 诊断层 | 是 |
| 标准库成长 Agent | 成长层 | 否 |

最重要的原则：

```text
系统提供事实和候选，AI 负责诊断和教学表达，后端负责守边界和保可信，老师负责确认标准库成长。
```

## 2. 总体链路

完整工作过程如下：

```text
学生提交代码
-> 判题得到结果
-> 事实层整理可验证材料
-> 搜索定位 Agent 召回标准库候选
-> 诊断建议 Agent 生成诊断与建议
-> 后端保障层校验、修复、裁决、留痕
-> 学生看到结构化反馈
-> 标准库成长 Agent 事后沉淀候选
```

### 2.1 实时链路

实时链路只处理当前学生这次提交：

| 步骤 | 目标 | 是否直接面向学生 |
|---|---|---|
| 事实层 | 整理题目、代码、判题结果、错误输出、规则信号 | 否 |
| 搜索定位 Agent | 从标准库召回相关资料 | 否 |
| 诊断建议 Agent | 判断主因，生成基础层和提高层反馈 | 是 |
| 后端保障层 | 校验证据、安全和输出质量 | 否 |

### 2.2 后台成长链路

后台成长链路不阻塞学生本次反馈：

| 步骤 | 目标 |
|---|---|
| 收集 PARTIAL / MISS | 找出标准库颗粒度不足或未覆盖的地方 |
| 聚类去重 | 避免同类错因重复进入标准库 |
| 生成成长候选 | 把 AI 新发现整理为候选知识点或错因 |
| 教师审核 | 决定新增、拆分、合并、忽略 |
| 版本化发布 | 更新标准库并回测旧样本 |

## 3. 五层责任结构

### 3.1 事实层

事实层不是 Agent。它是整个诊断系统的地基。

事实层只整理客观材料，不做诊断判断。

它应整理：

- 是否编译成功。
- 是否运行时错误。
- 是否超时。
- 哪些测试点失败。
- 有无错误输出。
- 学生输出和期望输出差异。
- 题目输入格式。
- 题目约束。
- 代码语言。
- 可定位代码行。
- 规则信号。

事实层需要区分两类内容：

| 类型 | 含义 | 能否被 AI 推翻 |
|---|---|---|
| 硬事实 | 编译错误、RE、TLE、测试输入输出、题面约束 | 不能 |
| 弱信号 | 规则启发式、代码形态猜测、疑似边界问题 | 可以被 AI 基于证据覆盖 |

关键原则：

```text
事实优先，证据优先，AI 负责解释、排序和教学表达。
```

### 3.2 召回层：搜索定位 Agent

搜索定位 Agent 的职责必须收窄。

它不是轻量诊断 Agent，而是标准库召回器和上下文压缩器。

它读取：

- 事实层整理出的材料。
- 规则信号。
- 当前标准库。

它输出：

- 相关标准库条目。
- 条目 ID。
- 条目名称。
- 条目层级。
- 召回理由。
- 召回来源。
- 粗略相关度。
- 精选后的标准库包。

它不应该输出：

- 最终主因。
- 学生可读建议。
- 基础层结论。
- 提高层结论。
- 诊断排序。

搜索定位 Agent 可以使用 HIT / PARTIAL / MISS，但这里的含义只限于：

```text
标准库覆盖情况。
```

它不代表：

- AI 判断一定正确。
- 当前问题一定是主因。
- 证据一定充分。
- 学生一定应该看到这条诊断。
- 标准库一定应该成长。

### 3.3 诊断层：诊断建议 Agent

诊断建议 Agent 是唯一真正负责“诊断判断”的实时 Agent。

它拿到：

- 事实层材料。
- 学生代码。
- 判题结果。
- 搜索定位 Agent 召回的标准库候选。
- 精选标准库包。

它需要完成：

- 理解题目要求。
- 理解学生代码意图。
- 比较代码行为和题目要求的差距。
- 判断主因、次因和干扰项。
- 判断基础层问题。
- 判断提高层建议。
- 生成学生可读反馈。
- 生成教师可复盘 trace。
- 生成标准库成长线索。

诊断建议 Agent 必须判断主次，不能把召回候选平铺给学生。

学生端反馈建议保持以下结构：

```text
基础层
1. 当前最主要的问题
   发生了什么
   为什么这样判断
   学生下一步怎么检查

提高层
1. 当前写法的限制
   后续可以怎么优化
   这个能力对以后有什么价值

下一步行动
先做哪一件事
```

注意：

- 基础层几乎总是需要。
- 提高层可以根据情况变短或暂不展示。
- 下一步行动必须具体，而且只给最优先的一件事。
- 文案必须自然、完整、有教学感。
- 不能直接给完整代码、完整答案、完整递推式。

### 3.4 保障层

保障层不是 Agent。

它的角色是：

```text
格式修复器 + 安全审查器 + 证据裁判 + 降级控制器 + 留痕系统
```

它不替 AI 思考题目，也不为了可展示而把不可靠诊断修成可靠诊断。

它可以修：

- 空数组。
- 空字符串。
- 字段别名。
- JSON 小格式问题。
- 非关键字段缺失。
- 能确定映射的 evidenceRef 别名。

它不能修：

- 主因和证据不一致。
- AI 编造题面不存在的信息。
- AI 猜测隐藏测试。
- AI 直接给完整答案。
- AI 的 evidenceRef 无法对应任何真实证据。
- AI 与判题硬事实冲突。

### 3.5 成长层：标准库成长 Agent

标准库成长 Agent 不直接更新标准库，只生成候选。

更稳的成长流程是：

```text
PARTIAL / MISS 诊断记录
-> 聚类合并相似错因
-> 统计频次和代表样本
-> 生成候选错因
-> 老师审核
-> 标准库版本化发布
-> 回测旧样本
```

成长候选至少包含：

- 候选名称。
- 建议归属路径。
- 学生可读解释。
- 教师审核解释。
- 典型代码表现。
- 支持它的真实提交样本。
- 与已有条目的差异。
- 推荐动作：新增 / 拆分 / 合并 / 忽略。

关键原则：

```text
AI 可以提出成长候选，但不能直接污染标准库。
```

## 4. 证据体系

证据不是一个随便的字符串 ID。证据必须有来源、类型和可追溯内容。

### 4.1 证据类型

| 证据类型 | 示例 |
|---|---|
| 题面证据 | 题目第一行要求输入 n 和 q |
| 代码证据 | 学生代码只读取了一个整数 |
| 判题证据 | 多组查询测试点失败 |
| 运行证据 | 输入 A 时输出 B，但期望 C |
| 异常证据 | IndexError、ValueError、编译错误 |
| 复杂度证据 | n 和 q 很大，重复 sum 会超时 |
| 标准库证据 | 对应某个输入解析易错点 |

### 4.2 证据底线

```text
证据只能引用，不能编造。
```

后端可以做：

- 字段名修复。
- 空列表补默认值。
- evidenceRef 别名映射。
- 非关键字段缺失降级。

后端不应该做：

- 为 AI 判断补一个不存在的证据。
- 把不确定证据强行映射到代码行。
- 把无效 evidenceRef 修成看似有效的 evidenceRef。

更稳的处理方式：

```text
能确定映射 -> 修复并记录 warning
不能确定映射 -> 标记 evidenceRef 无效，诊断降级
关键诊断无有效证据 -> 不作为主因展示
```

## 5. HIT / PARTIAL / MISS 的正确用法

HIT / PARTIAL / MISS 只表达一件事：

```text
标准库覆盖情况。
```

它不表达：

- 诊断是否正确。
- 当前问题是否主因。
- 证据是否充分。
- 学生是否应该看到这条诊断。
- 该错因是否应该进入标准库。

因此，系统需要把状态拆成多个维度：

| 维度 | 可选值 |
|---|---|
| 标准库覆盖状态 | HIT / PARTIAL / MISS |
| 诊断角色 | 主因 / 次因 / 提高建议 / 干扰项 |
| 证据强度 | 强 / 中 / 弱 / 无有效证据 |
| 展示决策 | 展示 / 降级展示 / 不展示 |
| 成长决策 | 候选 / 暂存 / 忽略 / 等待更多样本 |

示例：

```text
MISS 也可能是准确的新错因。
HIT 也可能只是误命中。
PARTIAL 也可能只是表面相似。
```

所以，HIT / PARTIAL / MISS 不能直接决定诊断可信度。

## 6. 后端真实结构对齐

当前后端已有一条接近目标的链路，但需要进一步强化事实层、证据体系、承接层和裁决层。

### 6.1 当前核心类职责

| 后端组件 | 当前职责 |
|---|---|
| `DiagnosticAgentService` | 组织诊断入口，构造规则诊断、证据包，并调用 AI 增强 |
| `DiagnosisEvidencePackageBuilder` | 整理题目、提交、判题结果、失败样例等证据 |
| `RuleSignalAnalyzer` | 产生规则信号，作为保底和搜索定位参考 |
| `AiReportService` | 调用外部模型，协调搜索定位和建议生成 |
| `SearchLocationRetrievalService` | 本地召回标准库候选 |
| `SearchLocationOutputValidator` | 校验搜索定位输出 |
| `SearchLocationOutputNormalizer` | 修正搜索定位输出中的轻微格式问题 |
| `SearchLocationPackSelector` | 根据定位结果生成精选标准库包 |
| `ExternalModelAgentRuntime` | 构造 brief、标准库包、prompt，并承接外部模型运行计划 |
| `AdviceGenerationOutput` | 诊断建议 Agent 的结构化返回对象 |
| `AdviceGenerationOutputNormalizer` | 对建议输出做轻量标准化 |
| `AdviceGenerationOutputValidator` | 校验建议输出安全性、证据、标准库 ID |
| `AdviceGenerationFeedbackMapper` | 把建议输出映射回学生反馈结构 |
| `SubmissionAnalysisResponse` | 最终返回给前端和报告系统的诊断结果 |

### 6.2 推荐后端主流程

```text
SubmissionAnalysisService
-> DiagnosticAgentService
-> DiagnosisEvidencePackageBuilder
-> RuleSignalAnalyzer
-> AiReportService.enhanceSubmissionAnalysis
-> ExternalModelAgentRuntime.createRuntimePlan
-> SearchLocationRetrievalService.retrieve
-> call search-location-v1
-> SearchLocationOutputNormalizer
-> SearchLocationOutputValidator
-> SearchLocationPackSelector.select
-> call diagnosis-report-v2
-> AdviceGenerationOutputNormalizer
-> AdviceGenerationOutputValidator
-> AdviceGenerationFeedbackMapper
-> SubmissionAnalysisResponse
```

### 6.3 两个实时 Agent 和后端状态的对应关系

| Agent | 后端对象 | 关键状态 |
|---|---|---|
| 搜索定位 Agent | `SearchLocationResult` | `enabled/status/embeddingStatus/fallbackReason/candidateCount/selectedCount` |
| 诊断建议 Agent | `AdviceGenerationOutput` | `diagnosisDecision/studentReport/libraryGrowth/basicLayerAdvice/improvementLayerAdvice` |
| 最终调用记录 | `SubmissionAnalysisResponse.AiInvocation` | `status/fallbackUsed/searchLocationStatus/adviceGenerationStatus/diagnosisLibraryFit` |

## 7. 搜索定位阶段设计

### 7.1 输入

搜索定位 Agent 的输入应包含：

- 事实层整理后的 `ModelDiagnosisBrief`。
- 本地召回的 `SearchLocationCandidatePack`。
- 标准库候选条目。
- 规则信号，但只能作为召回参考。

### 7.2 输出

搜索定位 Agent 输出应限制为：

- 候选条目 ID。
- 条目名称。
- 条目路径。
- 条目层级。
- 召回理由。
- 粗略相关度。
- 是否建议进入精选包。

不输出：

- 学生建议。
- 主因判断。
- 次因判断。
- 提高层结论。

### 7.3 成功条件

搜索定位成功时：

- `SearchLocationResult.status = SUCCESS`。
- `candidateCount > 0`。
- `selectedCount > 0`。
- `selectedPack` 只包含精选后的标准库条目。

### 7.4 降级条件

搜索定位可以降级，但不能阻断诊断建议 Agent。

| 情况 | 处理 |
|---|---|
| embedding 不可用 | 使用文本召回，`embeddingStatus = VECTOR_DEGRADED` |
| 无候选 | 使用较粗标准库包，记录 `NO_SEARCH_LOCATION_CANDIDATES` |
| 定位模型失败 | 使用较粗标准库包，记录失败原因 |
| 定位输出格式错 | 尝试修复；失败后降级 |

搜索定位失败时，诊断建议 Agent 仍然运行，只是拿到的标准库包更大、更粗。

## 8. 诊断建议阶段设计

### 8.1 输入

诊断建议 Agent 输入包括：

- 事实层材料。
- `ModelDiagnosisBrief`。
- 搜索定位后的 `StandardLibraryPack`。
- `SearchLocationSummary`。
- 原始证据引用列表。
- 规则信号。

### 8.2 输出结构

当前 `AdviceGenerationOutput` 已包含以下核心区块：

| 字段 | 用途 |
|---|---|
| `executionGate` | 当前是否适合给出诊断、优先级和原因 |
| `diagnosisDecision` | 标准库覆盖判断、锚点、库外发现 |
| `studentReport` | 面向学生的自然语言报告 |
| `teacherTrace` | 给教师或评测系统看的诊断依据 |
| `libraryGrowth` | 标准库成长候选 |
| `caseUnderstanding` | 题目理解、代码意图、行为差距 |
| `basicLayerAdvice` | 基础层问题 |
| `improvementLayerAdvice` | 提高层建议 |
| `nextStepPlan` | 下一步行动 |
| `studentSummary` | 学生摘要 |

后续开发应逐步减少“填表式学生文案”的占比，让 `studentReport` 成为学生端主要展示来源。

### 8.3 诊断建议 Agent 必须判断主次

诊断建议 Agent 必须区分：

- 主因。
- 次因。
- 提高建议。
- 干扰项。

典型例子：

```text
规则层看到 a[l:r]，可能提示 off-by-one。
但题目第一行是 n q，代码只读 n。
这时主问题应是输入结构错误，而不是 off-by-one。
```

## 9. 后端保障层设计

### 9.1 校验分级

建议把模型输出校验分为四类：

| 等级 | 含义 | 是否可展示 |
|---|---|---|
| PASS | 完全合法 | 是 |
| REPAIRED | 有小问题，已修复 | 是，记录 warning |
| DEGRADED | 主内容可用，但部分信息丢失 | 可展示，但标记降级 |
| REJECTED | 有安全风险或结构不可用 | 不展示，使用兜底 |

### 9.2 典型失败处理

当前系统容易把小错误直接打成 fallback。后续应避免：

```text
单个 evidenceRef 错误 -> 整份诊断失败
空数组字段 -> NPE -> 整份诊断失败
标准库 ID 半命中 -> 被当作非法输出
```

### 9.3 模型原文留痕

每次 AI 调用都应尽量保留：

- 模型原始输出。
- 清理后的 JSON。
- 软修复记录。
- 校验结果。
- 最终学生反馈。
- fallback 原因。

这样后续评测和人工复盘才能知道：

```text
模型没说对
还是模型说对了但后端丢了
还是模型说对了但被规则覆盖了
```

## 10. 规则反馈与 AI 反馈协调

不要简单说“AI 优先于规则”。更准确的原则是：

```text
硬事实优先，证据优先；AI 负责解释、排序和教学表达。
```

### 10.1 冲突处理原则

| 冲突情况 | 处理 |
|---|---|
| AI 与硬事实冲突 | 降级 AI |
| AI 与弱规则冲突 | 看证据强度 |
| AI 有代码、题面、判题证据 | 可采用 AI 主因 |
| AI 只有推测，没有证据 | 不能作为主因 |
| 教师人工修正存在 | 教师优先 |

### 10.2 主因裁决

最终学生反馈必须有一个清晰主因。

错误示例：

```text
边界错、输入错、复杂度错、输出错都有可能。
```

正确示例：

```text
当前最先要处理的是输入结构。题目第一行有 n 和 q，但你的代码只读了 n。
```

## 11. 典型完整场景

### 11.1 题目

```text
输入 n 和 q
第二行 n 个整数
接下来 q 行，每行 l r
输出每个区间和
```

### 11.2 学生代码问题

- 只读了 n，没有读 q。
- 只处理了一组查询。
- 使用切片时可能漏掉右端点。
- 如果 q 很大，反复求和会超时。

### 11.3 事实层

事实层整理：

- 题面第一行有 `n q`。
- 代码第一行只读取一个整数。
- 题面有 `q` 行查询。
- 代码只读取一次 `l r`。
- 代码使用 `sum(a[l:r])`。
- 如果约束很大，重复区间求和存在复杂度风险。

### 11.4 搜索定位 Agent

搜索定位 Agent 召回：

- 输入结构理解错误。
- 多组查询漏处理。
- 区间边界错误。
- Python 切片右端不包含。
- 前缀和优化。

它不判断最终主因，只把这些作为候选资料交给诊断建议 Agent。

### 11.5 诊断建议 Agent

诊断建议 Agent 判断：

```text
主问题：输入结构读错，多组查询没有处理。
次问题：区间右端点需要检查。
提高建议：多次区间和应考虑前缀和。
```

### 11.6 学生最终反馈示例

```text
基础层：
你现在最先要处理的是输入结构。题目第一行有 n 和 q，但你的代码只读了 n，这会导致后续输入整体错位。

同时，题目有 q 次查询，你的代码只处理了一次，所以当 q 大于 1 时，后面的查询没有被处理。

提高层：
等输入结构修好后，再考虑区间和的效率。如果 q 很大，每次重新 sum 一段区间会很慢，可以思考能不能先预处理前缀和。

下一步：
先逐行对照题面输入格式，确认每一行 input 对应题目哪一行。
```

## 12. 各种情况如何处理

### 12.1 标准库准确命中

直接绑定标准库 ID，作为诊断锚点之一。

同时还要判断：

- 是否为主因。
- 证据是否充分。
- 是否适合展示。

### 12.2 标准库半命中

允许 AI 继续自由分析。

输出中应保留：

- 相近标准库条目。
- 为什么只是 PARTIAL。
- AI 实际判断出的更细问题。
- 是否生成标准库成长候选。

### 12.3 标准库未命中

不强行套标准库。

输出中应保留：

- `libraryFit = MISS`。
- `outOfLibraryFindings`。
- 建议路径。
- 证据。
- 成长候选。

### 12.4 AI 和规则反馈冲突

如果 AI 有更强证据，采用 AI。

例如：

```text
规则层：off-by-one
AI：输入结构读错
证据：题目第一行 n q，但代码只读 n
最终主因：输入结构错误
```

### 12.5 AI 输出有小格式问题

后端修复或标 warning，不直接废掉整份诊断。

典型处理：

```text
problemConstraints -> 能确定时映射为合法 evidenceRef
unknown standard library id -> OUT_OF_LIBRARY
null list -> []
```

不能确定映射时，不要编造证据，应降级或删除该证据引用。

### 12.6 AI 输出有安全风险

以下情况必须拒绝：

- 直接给完整代码。
- 直接给完整答案。
- 直接给完整递推式或完整证明。
- 猜测隐藏测试。
- 编造题面不存在的信息。

### 12.7 AI 调用失败

明确标记失败，不伪装成 AI 成功。

学生可以看到规则兜底反馈，但系统和教师侧必须知道：

```text
外部 AI 没有成功参与本次诊断。
```

## 13. 给 AI/工程实现者的硬性约束

后续任何实现都必须满足：

1. 学生实时链路只能是“搜索定位 -> 诊断建议”，不能引入一堆同步 Agent。
2. 系统内部必须显式区分事实层、召回层、诊断层、保障层、成长层。
3. 搜索定位 Agent 只做召回和上下文压缩，不做最终诊断。
4. 标准库只能服务 AI，不能变成硬规则引擎。
5. HIT / PARTIAL / MISS 只能表示标准库覆盖状态。
6. 诊断建议 Agent 必须判断主因、次因、提高建议和干扰项。
7. 学生端文案必须自然，不允许字段拼接感。
8. 小格式错误不能导致整份诊断 fallback。
9. 证据只能引用，不能编造。
10. 模型说对且证据充分时，不能被弱规则覆盖。
11. 标准库未命中时，必须允许 AI 自由提出新错因。
12. 标准库成长 Agent 只能生成候选，不能直接更新标准库。
13. 所有 fallback 必须可观测、可复盘。
14. 安全边界必须保留，不能泄露完整答案。
15. 每次重大改动必须通过真实错题评测。

## 14. 评测与验收标准

### 14.1 5 题基础回归

继续使用当前 live eval 中暴露问题的 5 题：

- `off-by-one`
- `output-format`
- `brute-force`
- `teacher-corrected-input-parsing`
- `teacher-corrected-brute-force`

验收标准：

- 0 个 NPE。
- 0 个因为单个 evidenceRef 直接整份失败。
- 至少 4/5 题 AI 结果被接受或修复后接受。
- 输入解析题主因必须是 `INPUT_PARSING`。
- brute-force 题必须指出复杂度和重复扫描。
- 模型说对时，最终学生反馈必须采纳模型主因。

### 14.2 高难题扩展回归

加入 5 个高难 fixture：

- 二维差分还原。
- 单调栈重复值。
- 区间 DP。
- 分层最短路状态维度。
- 滑动窗口状态更新顺序。

验收标准：

- HIT / PARTIAL / MISS 都能正常输出。
- PARTIAL / MISS 能进入标准库成长候选。
- 学生反馈能讲清楚主因和下一步。
- 提高层建议不能压过基础层主问题。

### 14.3 长期效果指标

除了链路是否跑通，还要持续看：

| 指标 | 含义 |
|---|---|
| 主因诊断准确率 | AI 判断的主要错因是否正确 |
| 误导率 | AI 是否把学生带向错误修改方向 |
| 修复成功率 | 学生看完反馈后是否更快 AC |
| 证据有效率 | 关键诊断是否有真实证据支持 |
| 安全违规率 | 是否泄露完整答案、代码、隐藏测试 |
| 平均响应时间 | 学生等待是否可接受 |
| fallback 率 | AI 结果被拒绝或降级的比例 |
| REPAIRED 比例 | 小修复后可用的比例 |
| MISS 转化率 | 新错因进入候选/标准库的比例 |
| 教师审核通过率 | 成长候选质量是否稳定 |

### 14.4 报告可复盘性

每条 live eval 报告至少能回答：

- 事实层整理了哪些硬事实？
- 搜索定位有没有启用？
- 候选数量是多少？
- 精选数量是多少？
- advice 阶段有没有成功？
- 是否 fallback？
- fallback 原因是什么？
- 模型原文是否可追溯？
- 关键证据是否有效？
- 最终学生反馈来自 AI 还是规则兜底？

## 15. 后续开发优先级

后续开发按以下顺序推进：

1. 修复承接层稳定性：空列表、空字段、非法 evidenceRef 不得导致 NPE 或整份失败。
2. 定义事实与证据体系：事实和推断分开，evidenceRef 有类型、有来源。
3. 收窄搜索定位 Agent 职责：只召回，不诊断。
4. 增强校验分级：区分 PASS、REPAIRED、DEGRADED、REJECTED。
5. 增强模型/规则裁决：硬事实优先，弱规则可被有证据的 AI 覆盖。
6. 改善学生文案来源：优先使用 AI 的自然语言 `studentReport`。
7. 完善 live eval 报告：保留模型原文、修复记录、最终反馈来源和证据有效性。
8. 设计标准库成长 Agent 治理：聚类、去重、样本、教师审核、版本化和回测。
9. 扩展高难真实错题评测集。

最终目标：

```text
让这个 Agent 分析学生错题时，接近一个真正懂信息学竞赛、懂教学、懂学生表达方式的高手。
```
