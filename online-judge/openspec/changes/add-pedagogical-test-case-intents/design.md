## Context

现有链路已经能保存逐测试点执行结果，但 `submission_case_results` 只记录题目内顺序号和输入输出快照；隐藏测试点的值会在模型上下文中被遮蔽，因此模型只能看到“第 N 个隐藏点失败”。同时标准库已经具备知识点、能力点、易错点、提升点和课堂/竞赛场景，缺少的是从最原始判题证据进入这些学科结构之前的中性语义层。

外部校准得到三个共同约束：计算机课程标准把测试计划、测试矩阵、边界/角落输入列为应掌握能力；竞赛题包把样例、隐藏数据、测试组和评分依赖作为正式结构；形成性反馈需要先明确学习目标，再用证据定位差距。因此本设计把“测试覆盖什么”作为题库内容，而不把“学生为什么错”预写进测试点。

## Goals / Non-Goals

**Goals:**

- 精修现有 16 题、45 个测试点，使每个测试点有唯一、可审核、可映射的学科语义。
- 让判题结果保存稳定测试点身份和语义快照。
- 让 AI 自由诊断安全消费评测意图，提高失败事实的可解释颗粒度。
- 通过 SQL 门禁和开发者 fixture 验证完整性、隐藏数据安全与主链路消费。

**Non-Goals:**

- 本轮不增加第二个诊断 Agent，不改变“判题事实 → AI 自由诊断 → 标准库挂接 → 基础/提高建议 → 数据库存储”主链路。
- 不实现 IOI 式完整子任务计分、测试组调度或部分分；`contest_role` 只为未来演进保留语义。
- 不根据开发者 fixture 宣称真实学生学习效果改善。
- 不把测试点预绑定的能力路径直接喂给自由诊断模型。

## Decisions

### 1. 语义字段归属 `test_cases`

为每个测试点增加 `semantic_code`、`intent_type`、`intent_title`、`intent_summary`、`learning_objective`、`contest_role`、`reveal_policy`、`knowledge_node_code`、`skill_unit_code`、`review_status`、`source_reference`、`library_version` 和 `reviewed_at`。

选择单个主意图而不是新建多对多标签表，是因为当前只有 45 个测试点，首要问题是建立稳定、可消费的主语义。未来需要多标签或测试组时可再增加子表，不让本轮提前长成评测编排系统。

### 2. 判题结果保存身份与安全语义快照

`submission_case_results` 增加 `test_case_id` 和一组 `test_*` 快照字段。仅保存 code、意图、标题、泛化说明、学习目标、竞赛角色和揭示策略，不复制知识点/能力点 ID。

与运行时按顺序回查当前 `test_cases` 相比，快照会增加少量存储，但能避免题库修改后历史诊断语义漂移，也能让无数据库查询的证据包构造器保持确定性。

### 3. 标准库映射用于治理，不进入自由诊断

数据库门禁验证每个正式测试点映射到存在且路径一致的知识点和能力点，用于覆盖审计；模型简报只接收自然语言测试意图，不接收这些标准库 ID。这样既能证明题库覆盖学科结构，又不会破坏“先自由诊断、再挂接标准库”的顺序。

### 4. 隐藏点只输出泛化语义

公开点使用 `PUBLIC_EXAMPLE`，隐藏点使用 `AI_GENERALIZED`。隐藏点的输入、输出和具体数值仍由既有脱敏逻辑清空；新增语义文本必须描述输入类别、状态关系或规模性质，不能出现可反推测试数据的明细。

### 5. 提示词采用新版本

新增 `free-diagnosis-v2` 和 `diagnosis-report-v4`，保留旧版本注册但正式运行切换到新版本。新 prompt 明确三条规则：测试意图不是错因、必须与代码和行为交叉验证、隐藏语义不得进入学生可见细节。

### 6. 正式门禁与开发者验证分层

SQL 质量门禁负责 45 个正式测试点的完整性、唯一性、路径一致性、内容厚度、同题互补覆盖和隐藏策略；单元测试负责快照、证据引用和脱敏；隔离 PostgreSQL 验证 V1-V8 空库、幂等、基线升级和 Hibernate `validate`。由于没有真实学生数据，本轮不设置“诊断准确率提升”阈值。

## Risks / Trade-offs

- [命题者语义可能诱导模型] → prompt 明确其不是错因证据，且不向自由诊断暴露标准库 ID；回归 fixture 覆盖冲突场景。
- [隐藏语义泄漏测试数据] → 使用单独揭示策略、禁止数值明细、单元测试扫描隐藏上下文，学生输出继续受答案泄漏门禁保护。
- [历史顺序映射错误] → V8 只在题目、顺序号能唯一对应时回填；不确定记录保持 null 并在报告中单独统计。
- [教师新建题暂时无语义] → Schema 允许 null，正式内容门禁阻止未评审测试点发布，避免破坏现有创建流程。
- [字段快照增加存储] → 每个已执行测试点只增加有限短文本，相比源代码和分析 JSON 成本很小，换取历史可追溯性。

## Migration Plan

1. 在隔离 PostgreSQL 对生产备份恢复演练，记录迁移前业务计数与 45 个测试点快照。
2. 执行 Flyway V8：加列和约束、精修 45 个测试点、确定性回填历史结果、运行迁移内断言。
3. 启动应用并由 Hibernate `validate` 验证实体映射。
4. 执行 Schema readiness、测试点语义质量门禁、全量测试和 AI 开发者 fixture。
5. 在生产发布前创建可校验备份；发布后验证 Flyway V8、45/45 覆盖、历史结果回填率、`/code/` 与 readiness。
6. 回滚应用时保留向后兼容的新增列；若必须回滚数据，使用发布前备份恢复，不执行破坏性逆向 SQL。

## Open Questions

- 真实课堂积累足够后，是否把测试点升级为可计分测试组或子任务，需要依据教师使用和学生提交分布另行决策。
- 是否向教师管理界面开放语义编辑，需要先验证当前手工精修字段是否稳定，不在本轮扩展 UI。
