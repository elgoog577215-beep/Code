## Context

系统当前已有数据库版 AI 标准库，条目分为 `BASIC_CAUSE` 和 `IMPROVEMENT_POINT`，并通过 `StandardLibraryPackBuilder` 输出给 AI 诊断链路。这个结构适合承载错因和提升建议，但还缺少中学信息学科本身的层级知识框架。

用户明确要求建设“从大章节到小知识点，再到每个知识点对应错因和延伸”的大规模知识数据库。因此本阶段的核心不是继续堆提示词，而是建立学科知识树，并把现有标准库条目挂载到知识点下面。

## Goals / Non-Goals

**Goals:**

- 建立信息学知识节点模型，支持领域、章节、子章节、知识点四级及以上层级。
- 首批种子化一套覆盖中学信息学核心范围的知识体系骨架。
- 让错因和提升点能够关联到一个或多个知识点。
- 提供教师端受保护 API，用于查看知识树、知识点详情和该知识点下挂载的错因/提升点。
- 保持现有 AI 标准库输出兼容。

**Non-Goals:**

- 本阶段不实现向量检索、复杂召回排序或 LLM prompt 全面改造。
- 本阶段不要求一次性完成全部 1000+ 知识点和错因。
- 本阶段不实现复杂版本 diff、审批流、批量导入导出。
- 本阶段不改变学生端提示分层策略。

## Decisions

### 1. 知识树与错因条目分表

采用独立 `informatics_knowledge_nodes` 表保存学科知识节点；现有 `ai_standard_library_items` 继续保存错因/提升点。标准库条目通过 `knowledge_node_codes` 关联一个或多个知识点。

这样做的原因是知识节点和错因条目的生命周期不同：知识树相对稳定，错因和提示话术会频繁由教师调整。分表可以避免把学科目录、错因、AI 证据和提示策略混成一个不可维护的大表。

### 2. 节点使用稳定编码

知识节点使用稳定 `code`，例如 `BASIC.LOOP.FOR.BOUNDARY.LEFT_CLOSED_RIGHT_OPEN`。父子关系使用 `parentCode`，并保留 `path` 便于查询和展示。

数字主键用于数据库内部，稳定编码用于种子、API、AI prompt、测试和教师维护。

### 3. 首批数据追求“骨架完整 + 样例足够细”

本阶段先覆盖大范围骨架，并为高频知识点提供较细错因。目标是让系统已经具备正确形态，后续可以持续扩充，而不是一次性生成一份难以验证的超大数据。

首批重点覆盖：

- 程序设计基础：输入输出、变量类型、表达式、条件、循环、数组、字符串、函数、递归。
- 数据结构基础：顺序表、栈、队列、集合映射、树、图。
- 算法思想：模拟、枚举、排序、二分、前缀和、双指针、搜索、贪心、动态规划、图算法。
- 数学与建模：整除、取模、组合计数、位运算、坐标与几何。
- 调试与工程习惯：编译错误、运行错误、边界测试、复杂度估计、代码可读性。

### 4. 教师端先提供简单 API

教师端视觉界面保持“越简单越好”。本阶段先实现查询接口：

- 查整棵知识树。
- 查某个知识点详情。
- 查某个知识点下的错因/提升点。

复杂编辑界面可在下一轮补上，但后端字段和服务先预留更新能力。

### 5. 与 AI 链路保持兼容

`StandardLibraryPackBuilder` 仍返回原来的 `basicCauses` 和 `improvementPoints`。知识点字段作为额外元数据进入数据库和 API，本阶段不强制模型消费，避免一次变更多处风险叠加。

## Risks / Trade-offs

- 知识点规模变大后教师端可能难以浏览 → 用树形查询、层级筛选和搜索降低复杂度。
- 生成式种子数据可能存在分类不完美 → 首批数据以“可维护骨架”为目标，允许教师后续修订。
- 关联字段使用换行字符串而非多对多表会降低查询表达力 → 本阶段为了兼容和迭代速度先采用轻量关联；若后续做召回评分，再迁移为显式关联表。
- 标准库条目突然增多可能拉长 AI 上下文 → 本阶段不把全量知识库传给模型，后续必须接检索召回。

## Migration Plan

1. 新增知识节点实体和表，启动时按稳定编码幂等种子化。
2. 给标准库条目增加 `knowledgeNodeCodes` 可选字段。
3. 更新标准库种子，使重点条目挂载到知识点；已有数据库条目不强制覆盖教师修改。
4. 新增教师查询 API。
5. 增加测试覆盖种子数量、层级完整性、知识点下挂错因查询和现有 `StandardLibraryPack` 兼容输出。

回滚时可以停用新接口；现有 AI 标准库和 `StandardLibraryPack` 不依赖知识树即可继续运行。

## Open Questions

- 后续是否要把知识体系扩到完整 NOI 提高级内容，还是先聚焦高中课堂和 CSP-J/S 常见题型。
- 教师端编辑知识树时是否需要审批或版本发布机制。
- 召回阶段采用规则召回、关键词召回、向量召回，还是先做混合轻量召回。
