## Context

当前项目的正式内容有三类来源混在一起：

- 运行时数据库：Postgres/H2 中的题目、知识树、标准库规范表和兼容快照。
- 代码 seed：`PublicProblemSeedCatalog`、`InformaticsKnowledgeSeedCatalog`、`AiStandardLibrarySeedCatalog`、`AiStandardLibraryV*Seeds`。
- 启动播种器：多个 `CommandLineRunner` 在应用启动时把代码 seed 写入数据库。

这种结构让 GitHub 变成事实内容主库，服务器正式数据库只是被反复同步的副本。用户已经明确要求正式内容直接进入数据库，并从项目中移除 seed。迁移必须同时保护现有服务器数据、避免内容丢失，并保证 AI 导航读取仍可用。

## Goals / Non-Goals

**Goals:**

- 将正式题库、知识树和标准库内容主权迁回数据库。
- 停止运行时 seed 入口，应用启动不得再修改正式内容。
- 提供一次性迁移和验证流程，把现有 seed 中仍需要保留的内容写入正式数据库。
- 删除或迁出内容型 seed 文件，使后续新增内容只能通过数据库治理流程进入正式库。
- 将测试从“seed 静态内容正确”改为“数据库正式内容可读、可导航、可治理”。

**Non-Goals:**

- 不把所有正式内容重新放进另一种长期 SQL seed 文件。
- 不在没有备份和目标数据库确认的情况下直接改生产数据库。
- 不删除教师人工治理、成长候选和兼容快照机制。
- 不把学生提交、班级运行数据或教师账号等运行时数据提交到 GitHub。

## Decisions

### 1. 数据库为内容主库

正式内容以数据库表为准：题目写入题库相关表，知识树写入 `informatics_knowledge_nodes`，标准库写入 `ai_standard_skill_units`、`ai_standard_mistake_points`、`ai_standard_improvement_points`，并同步必要的旧平铺快照。

替代方案是继续维护 seed，并在部署时由代码重播。该方案被拒绝，因为它继续让代码仓库承担内容主库职责。

### 2. 一次性迁移可以生成执行脚本，但脚本不是长期内容源

迁移过程可以从现有 seed 提取数据，生成一次性 SQL 或通过管理 API 写入正式库。迁移脚本必须带备份、幂等键、执行摘要和验证查询。迁移完成后，运行时不得依赖该脚本或 seed 重新生成内容。

替代方案是把 seed 内容转成固定 SQL 文件长期放在仓库。该方案被拒绝，因为它只是把 Java seed 换成 SQL seed。

### 3. 运行时 seed 入口先禁用再删除

实施顺序采用“两段式”：

1. 先让运行时播种器默认禁用，并通过配置或测试确认启动不会写正式内容。
2. 迁移验证后删除内容型 seed 类、资源文件和旧测试断言。

这样可以在迁移未完全执行时保留回滚空间，避免直接删除导致本地测试和服务器启动同时失效。

### 4. 内容治理入口替代 seed

后续新增或修改正式内容只能走三类入口：

- 教师端/后台治理 API。
- 标准库成长候选审核批准。
- 运维人员执行的可审计迁移脚本。

所有入口都必须写正式表，并能被数据库查询和 AI 导航读取验证。

### 5. 测试改为数据库行为测试

现有大量测试直接断言 `SeedCatalog.seeds()` 数量、名称和内容。迁移后测试应改为：

- 使用测试数据库 fixture 或 repository 构造最小正式内容。
- 验证服务从数据库读取内容。
- 验证禁用 seed 后启动不会新增正式内容。
- 验证 AI 导航和教师治理读取规范表。

## Risks / Trade-offs

- [Risk] 服务器正式库还没有完整内容，禁用 seed 后出现空题库或空标准库。
  Mitigation：迁移前必须备份并执行覆盖查询；部署前验证关键表数量、代表性路径和 AI 导航读取。

- [Risk] 旧测试大量依赖 seed，删除过快会造成测试面大面积失败。
  Mitigation：先改运行时入口和关键测试，再逐步删除静态 seed 断言。

- [Risk] 一次性迁移脚本被误用为长期内容源。
  Mitigation：脚本命名和文档明确“one-shot migration”，执行后只保留审计摘要，不作为后续新增内容方式。

- [Risk] 题库内容和学生已有提交存在外键关系。
  Mitigation：迁移不得删除已有题目，只能 upsert、禁用或迁移映射；执行前后检查引用关系。

## Migration Plan

1. 盘点 seed 入口和正式表映射，输出迁移清单。
2. 从当前服务器数据库备份开始，记录备份路径和数据库版本。
3. 将需要保留的 seed 内容写入正式数据库，使用唯一 code/title 做幂等 upsert。
4. 验证数据库：题库数量、知识树根/叶节点、标准库能力点/易错点/提升点数量、代表性 `if 条件` 路径、AI 导航诊断层。
5. 禁用运行时 seed 入口，启动应用确认没有内容写入日志。
6. 删除内容型 seed 文件和静态 seed 测试。
7. 部署后再次运行数据库验证查询。

回滚策略：恢复迁移前数据库备份，并回退禁用/删除 seed 入口的代码提交。

## Open Questions

- 是否保留开发环境最小演示数据，需要单独定义为测试/演示 fixture，不能混入正式内容 seed。
