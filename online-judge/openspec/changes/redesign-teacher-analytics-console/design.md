## Context

教师端现有实现已经具备登录、侧栏、作业列表、作业概览、题目明细、学生提交证据、诊断标签和教师校正等基础能力，但产品结构仍围绕旧的“作业详情 + 学生诊断 + 教师动作”展开。这个结构会把 AI 信息推向“建议老师怎么做”，也会让班级、作业、题目三个递进对象和平行的统计粒度混在一起。

新的教师端主线改为教学结果分析系统：老师先按“班级 -> 作业 -> 题目”递进确定观察对象，再在当前对象范围内按“章节、知识点、能力点、易错点”切换统计粒度。AI 只负责知识归因、错因聚合、路径解释和证据定位。

当前可复用资产包括：
- `TeacherShell`、教师登录和教师侧栏。
- `shared/api/client.ts` 与 `AssignmentOverview` 等 DTO。
- `shared/ui` 里的按钮、空状态、指标和状态标签。
- `AssignmentDetailPage.tsx`、`ClassOverviewPage.tsx` 中已有的提交、正确率、趋势、学生提交证据、错因标签等计算思路。
- 标准库的知识点、能力点、易错点和 `knowledgePath` 语义。

## Goals / Non-Goals

**Goals:**
- 建立新的教师端正式主线：班级分析页、作业分析页、题目分析页。
- 将分析对象和统计粒度分离：对象递进选择，粒度平行切换。
- 统一三层页面的信息架构：基础指标、数据可视化、AI 知识路径、证据样本。
- 用新的 `features/teacher-analytics/` 承载主线，避免继续在旧作业详情页上堆功能。
- 迁移旧页面中有价值的数据、校正、证据和展示能力。
- 在实现完成后让教师端正式导航只指向新主线，避免新旧入口并存。

**Non-Goals:**
- 不重做学生端做题工作台。
- 不在本变更中重构 AI 诊断主链路或标准库数据模型。
- 不让 AI 生成教师动作、讲评安排或教学决策。
- 不把所有历史教师管理功能一次性删除；班级名单、题库、AI 标准库等管理入口可继续保留，但不作为分析主线。

## Decisions

### 1. 新主线独立建模，而不是深改旧 `AssignmentDetailPage`

选择：新增 `features/teacher-analytics/`，旧 `AssignmentDetailPage` 只作为迁移参考。

理由：旧页面的内在模型是“作业详情 -> 题目/学生 -> 诊断/校正/动作”。新模型是“班级 -> 作业 -> 题目 -> 粒度统计 -> AI 归因”。二者不是简单 UI 调整，继续深改旧页面会导致新旧语义共存。

替代方案：直接在旧页面加图表和 AI 面板。放弃原因是会保留旧的“教师动作/需讲评/建议”结构，质量风险高。

### 2. 路由表达递进对象

目标路由：

```text
/app/teacher/classes/:classId
/app/teacher/classes/:classId/assignments/:assignmentId
/app/teacher/classes/:classId/assignments/:assignmentId/problems/:problemId
```

页面顶部使用面包屑表达当前位置，例如：

```text
高一 3 班 > 第 7 周课堂作业 > 两数求和
```

理由：班级、作业、题目不是平行 tab，而是教学对象的递进范围。路由应该约束产品心智，避免 UI 回到平行切换。

### 3. 数据统一成分析模型

新增前端模型层：

```text
features/teacher-analytics/model.ts
features/teacher-analytics/selectors.ts
```

核心概念：
- `AnalyticsScope`: 当前观察对象，类型为 `class`、`assignment`、`problem`。
- `AnalyticsGranularity`: 当前统计粒度，类型为 `chapter`、`knowledgePoint`、`skillUnit`、`mistakePoint`。
- `AnalyticsMetric`: 提交数、未提交数、正确率、错误次数、涉及学生数、涉及题目数、平均提交次数。
- `KnowledgePathNode`: 完整知识路径中的节点。
- `InsightBucket`: 图表中的一项统计桶，包含 label、count、rate、path、evidence。

选择：页面不直接读散乱 DTO，而是先用 selector 把后端数据整理成统一图表数据。

理由：班级页、作业页、题目页要共用图表、浮层和 AI 面板。统一模型能减少重复逻辑，也能避免旧字段名直接污染新 UI 语义。

### 4. AI 信息嵌入图表与归因面板

AI 分析版面每层都存在，但职责固定为：
- 高频知识路径。
- 高频错因分布。
- 当前图表项的完整路径。
- 涉及题目、涉及学生、典型提交证据。
- 命中、半命中、未命中的标准库归属状态。

禁止职责：
- 不输出“下一步”。
- 不输出“建议老师讲什么”。
- 不输出“应当处理哪些学生”。
- 不把 AI 不可用时的模板内容伪装成 AI 分析。

理由：AI 的价值是把散乱提交结果归因到标准知识语言，而不是替老师做教学选择。

### 5. 图表先做轻量组件，依赖后置评估

初期实现轻量 SVG 图表组件：
- `AnalyticsBarChart`
- `AnalyticsPieChart`
- `AnalyticsRankList`
- `KnowledgePathTooltip`

理由：项目当前没有图表库依赖，引入图表库会增加包体、样式一致性和交互控制成本。第一版需要的是清楚、稳定、可控的教育系统图表，不需要复杂 BI 能力。

替代方案：引入 ECharts 或 Recharts。保留为后续选择，如果后续需要复杂交互、缩放、图例、导出，再单独评估。

### 6. 旧正式入口必须退出

迁移完成后：
- 教师侧栏的班级/作业入口指向新分析主线。
- 旧 `AssignmentDetailPage` 不再作为正式主线入口。
- 旧页面中仍有必要保留的管理能力需要迁移到新页或独立管理页。

理由：用户明确要求旧的和新的只能选择一个。保留双入口会让产品质量变差，也会让后续维护继续在两套模型之间摇摆。

## Risks / Trade-offs

- [Risk] 当前后端聚合数据可能不足以按章节、知识点、能力点、易错点完整统计。 -> Mitigation: 第一阶段用 `AssignmentOverview`、`topIssues`、`abilityWeaknesses`、学生提交诊断和标准库字段做前端聚合；不足字段通过新 DTO 或新聚合接口补齐。
- [Risk] 旧页面承载了教师校正等有用能力，直接废弃会丢功能。 -> Mitigation: 迁移清单先标记可复用能力，校正入口改为题目/提交证据样本中的次级入口，不再作为主界面心智。
- [Risk] 图表过多会让教师端重新变复杂。 -> Mitigation: 每层页面默认只展示 1 个主图、1 个分布图和 1 个证据区，其他视图通过粒度切换复用同一容器。
- [Risk] AI 路径信息质量依赖标准库命中。 -> Mitigation: UI 显示归属状态，区分标准库命中、半命中和未命中，不把不确定结果包装成确定路径。
- [Risk] 新旧路由替换影响已有脚本和 smoke。 -> Mitigation: 更新视觉 smoke、浏览器 smoke 和相关 i18n 文案；旧路由可短期重定向到新路由，但不得继续作为正式页面维护。

## Migration Plan

1. 新增 `features/teacher-analytics/` 模块和统一分析模型。
2. 先实现班级分析页，并接入现有班级、作业和 overview 数据。
3. 实现作业分析页，迁移旧作业概览中的提交数、正确率、题目列表和趋势数据。
4. 实现题目分析页，迁移旧题目学生明细、提交证据和诊断标签数据。
5. 加入知识路径浮层和 AI 归因面板，确保所有文案保持客观结果表达。
6. 替换教师侧栏和相关入口，让正式路径进入新主线。
7. 将旧正式页面改为重定向、移除或归档，避免新旧页面同时存在。
8. 更新中英文 i18n、前端 typecheck/build、浏览器 smoke 和必要的后端 DTO 测试。

## Open Questions

- 后端是否需要新增一个统一的教师分析聚合接口，还是先在前端用现有 overview 聚合出第一版？
- 章节粒度的数据来源是否已经稳定存在于知识树，还是需要从知识点路径向上汇总？
- 旧教师校正入口在新主线里保留到“题目 -> 提交样本”层，还是独立放到 AI 标准库治理流程？
