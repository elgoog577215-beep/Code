## Context

当前默认实时链路已经简化为“本地召回 + 单诊断 Agent + studentReport”。真实评测显示链路稳定性已经提升，但质量风险仍藏在学生可见文本里：有些反馈过于直接，像在告诉学生怎么改；有些提高层只是重复基础层；评测报告的 `outputDetail` 还会混入内部 trace，影响人工审查。

这里不再新建 Agent，也不推翻已经跑通的链路。真正要补的是：把人工审查关注点固化为评测报告和质量门禁。

## Goals / Non-Goals

**Goals:**
- 让 live eval 报告单独导出“学生真实可见文本”。
- 自动标记学生可见文本中的高风险问题：直接改法、内部痕迹、提高层过弱、文本过长。
- 调整正式提示词，让模型把下一步写成调试动作，而不是代码修改步骤。
- 让评测结果既保留机器指标，也方便人工逐条审查。

**Non-Goals:**
- 不新增外部模型调用次数。
- 不恢复双 Agent 默认链路。
- 不新增数据库表或新的审批系统。
- 不把启发式门禁伪装成最终教学判断；它只负责筛出需要人工关注的样本。

## Decisions

1. 复用现有 `AssistantLiveEvalReport.Entry`，新增学生可见字段和质量标记。
   - 原因：现有 live eval 已经能跑真实外部模型，最短路径是在报告里补足人工审查所需信息。
   - 替代方案：新建独立评测框架。暂不采用，成本高且会分裂报告来源。

2. 学生可见文本只从 `studentReport` 和学生端实际优先展示的字段生成。
   - 原因：`combinedAnalysisText` 会混入 teacherNote、uncertainty、内部 trace，不等于学生端真实体验。
   - 替代方案：继续看 `outputDetail`。暂不采用，因为它会误导人工审查。

3. 质量门禁先用简单启发式，不调用另一个 AI 来审 AI。
   - 原因：当前缺口很明确，关键词和长度就能抓住大部分“直接给改法”和“内部痕迹”问题。
   - 替代方案：再加一个审稿 Agent。暂不采用，链路变长且会引入新不稳定性。

4. 提示词收紧“下一步行动”，校验器保留安全底线。
   - 原因：人工审查显示主要问题不是模型不懂，而是输出边界不够稳定。

## Risks / Trade-offs

- 启发式可能误杀有教学价值的提示 → 只标记为质量风险，不直接阻断生产输出。
- 过度限制可能让反馈变空泛 → 提示词仍允许小反例、手推、复杂度估算、状态复述。
- 报告字段增多可能显得重复 → 保留原字段兼容旧报告，新字段只服务人工审查和质量统计。
