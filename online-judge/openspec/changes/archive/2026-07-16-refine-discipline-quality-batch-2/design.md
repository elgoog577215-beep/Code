## Context

生产 Flyway 当前版本为 V2。第一批精修后，结构性门禁全部通过，统一知识树共有 737 个启用节点，其中仍有 562 个知识点使用“细颗粒知识点”模板描述；718 个启用能力点中有 65 个没有启用提升点。六领域能力点覆盖率中 MATH 为 9.2%、ENG 为 9.5%，BASIC 虽为 19.0%，但绝对缺口和课堂使用面最大。

第二批生产只读画像按启用知识点统计规范化能力点、易错点和提升点的主节点及相关节点引用。高引用模板节点集中在数论边界、位掩码、组合计数、坐标系统、边界调试、状态追踪、数组下标、多组输入和函数状态等教学主线。

## Goals / Non-Goals

**Goals:**

- 精修 30 个生产中已经被标准库引用的 MATH、ENG、BASIC 模板知识点。
- 为 12 个已有正式错因但缺少提升点的能力点补齐可执行提升路径。
- 将模板债务从 562 降到不高于 532，将缺少提升点的能力点从 65 降到不高于 53。
- 保持规范化主表、旧平铺兼容快照和 legacy mapping 三者一致。
- 通过 Flyway V3、隔离恢复演练和生产导航 API 验证交付。

**Non-Goals:**

- 不新增知识节点，不拆分高中库与竞赛库，不改变稳定 code。
- 不批量生成低信息密度文案来清零 562 个模板债务。
- 不补兼容占位能力点的提升内容，不扩大旧 seed 体系。
- 不改变 API 响应结构，不修改现有提交、题目、班级和反馈业务数据。

## Decisions

### 1. 选择 30 个“已被使用但仍是模板”的节点

候选必须同时满足：启用、`KNOWLEDGE_POINT`、描述仍为模板、领域属于 MATH/ENG/BASIC。排序先看规范化标准库不同条目引用数，再结合领域覆盖缺口和学科中心性人工审校。第二批采用 12 个 MATH、10 个 ENG、8 个 BASIC 的倾斜分布：优先补最低覆盖领域，同时保留程序基础的课堂高频入口。

MATH 覆盖质数边界与筛法、位掩码、排列组合与容斥、取模边界和坐标体系；ENG 覆盖最小输入、状态追踪、复杂度权衡、手算、循环不变量和典型运行错误；BASIC 覆盖边界比较、频次统计、多组状态、下标、可变对象、回溯恢复和字符串查找失败。

替代方案是直接按引用数取前 30，但会让 BASIC 继续挤占低覆盖领域；也不采用三领域各 10 个，因为当前质量债务并不均匀。

### 2. 精修只写可教学、可检查的学科语义

每个节点必须同时更新 `description`、`learning_objectives`、`typical_problems`、`aliases` 和真实 `prerequisites`，并标记 `informatics-knowledge-discipline-v2`。描述必须包含概念对象、适用边界或状态含义；目标必须可观察；典型任务必须能够用于自测；没有可靠前置时保持空值。

不沿用模板句式做同义改写，也不为了填满字段编造前置关系。

### 3. 只为正式能力点补提升闭环

选择 12 个有 1 至 3 个启用易错点、但没有启用提升点的非兼容能力点。每条新增提升点必须给出训练目标、具体练习、学生收益、教师解释和关联错因，并写入：

1. `ai_standard_improvement_points` 规范化主表；
2. `ai_standard_library_items` 的 `IMPROVEMENT_POINT` 兼容快照；
3. `ai_standard_library_legacy_mappings` 的同 code `MAPPED` 映射。

三个写入使用稳定 code 和冲突保护；重复启动由 Flyway 历史阻止，隔离测试还会验证同一 code 只有一条启用记录。

### 4. 质量门禁同时检查增量和剩余债务

V3 后门禁至少检查：第二批精修数不少于 30、累计精修数不少于 54、第二批提升点不少于 12、三表 code 集合一致、缺少提升点能力点不高于 53、模板描述不高于 532，以及既有孤儿、断链、重复和分类实现词继续为零。

模板描述和未补提升点仍作为 DEBT，而不是伪装成已经完成；只有超过本批验收上限或出现结构错误才阻断发布。

### 5. 先恢复演练，再生产发布

先保存生产迁移前计数和质量指标，生成带校验值的 PostgreSQL custom dump，在独立数据库恢复后执行 V3、门禁、内容 readiness、导航读取和重复启动验证。生产发布保留旧应用镜像，应用启动由 Flyway 执行 V3；失败时停止替换并恢复旧镜像，必要时使用迁移前备份回库。

## Risks / Trade-offs

- [风险] 人工内容虽具体但仍可能与教师术语偏好存在差异 → 保留稳定 code 和 aliases，后续按教师校正继续迭代，不重建节点。
- [风险] 同步三张表增加迁移长度和一致性风险 → V3 使用同一 VALUES 数据源语义，质量门禁比较 code 集合和关键字段。
- [风险] 固定债务上限会被后续合法数据新增打破 → 上限只服务本批迁移验收；下一批先重新固化生产基线，再制定新的下降目标。
- [风险] 30 个节点仍只覆盖部分债务 → 报告公开剩余 532 个模板节点和 53 个闭环缺口，不把本批描述为全库完成。

## Migration Plan

1. 在当前生产 V2 上保存核心表计数、第二批候选和质量指标。
2. 生成并校验生产数据库备份，在独立 PostgreSQL 数据库完成恢复。
3. 对空库、V2 结构和生产恢复库分别执行完整 Flyway 链及重复启动验证。
4. 在隔离恢复库运行 Schema、内容、学科质量与代表性导航检查。
5. 推送并部署 V3 应用镜像，确认 Flyway 历史、质量门禁、导航 API、页面和核心业务计数。
6. 归档 OpenSpec；保留备份、旧镜像和迁移前后指标用于回滚与复核。

## Open Questions

无。本批节点和能力点均由生产只读画像确定，不需要新增产品或 Schema 选择。
