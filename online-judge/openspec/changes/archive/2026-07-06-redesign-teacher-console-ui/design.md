## Context

当前教师端已有较完整的数据能力：作业概览、题目统计、学生提交、AI 诊断、教师校正、班级学情、题库管理和 AI 标准库治理都已有前后端入口。但 UI 信息架构仍偏“页面拼贴”：全局顶部导航、局部返回按钮、作业详情中的待处理区域、题目表、AI 摘要和管理工具互相抢主次。

## Goals / Non-Goals

**Goals:**
- 让教师端形成稳定的左侧栏主导航和右侧工作区。
- 让教师主路径清晰表达为：作业中心 → 作业题目 → 题目分析 → 学生提交诊断。
- 让传统教学软件信息优先展示：作业、题目、提交、正确率、学生状态。
- 让 AI 信息只在合适层级作为易错点、讲评建议、诊断校正和标准库治理出现。
- 让学生端视觉边界、按钮、返回入口与新教师端质感协调，但不重排学生端主体。

**Non-Goals:**
- 不改后端接口、数据库、AI prompt 或标准库数据结构。
- 不重写学生端做题工作台。
- 不引入新的 UI 框架或图表库。

## Decisions

- 采用 `TeacherShell` 统一渲染教师左侧栏，而不是在每个页面单独写导航。
  - 理由：当前教师页面都包在 `TeacherRoute` 内，集中改 `TeacherShell` 能降低重复和回归风险。
  - 替代方案：继续用顶部子导航。缺点是无法承载教师端多入口，也不符合已确认视觉方向。
- 教师端内容页保留现有数据加载逻辑，只重排布局与文案层级。
  - 理由：后端数据足够，重构目标是信息架构和视觉一致性。
  - 替代方案：新增聚合接口。当前不是必要范围。
- 对 AI 质量、readiness、候选治理等信息做“教师可读降级”。
  - 理由：项目记忆已记录系统阻断状态应收进侧栏或辅助区，不能横向切断主任务。
- 学生端只做视觉协调。
  - 理由：用户明确学生端主体基本可以，当前风险在教师端。

## Risks / Trade-offs

- [风险] 一次改动页面多，可能引入布局回归。→ 通过 `npm run build`、桌面和移动截图检查控制风险。
- [风险] `styles.css` 已很大，新增样式可能和旧样式冲突。→ 使用 `teacher-console-*`、`teacher-shell-*` 等明确命名，并把重构样式集中追加。
- [风险] 教师端左侧栏和现有全局顶部导航重复。→ 通过教师上下文下弱化顶部教师子导航，左侧栏作为教师主导航。
