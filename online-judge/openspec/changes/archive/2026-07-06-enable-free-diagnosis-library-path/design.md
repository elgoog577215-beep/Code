## Context

当前实时诊断已经是单诊断 Agent，但模型输入和输出仍容易被“标准库候选”和本地证据片段牵引。实际产品目标是让外部 AI 先像 Codex 一样读题、读完整代码、理解运行结果，再把诊断挂到学校标准库路径上；标准库用于命名、统计和沉淀，不替模型下结论。

## Goals / Non-Goals

**Goals:**

- 一次模型调用内同时完成自由诊断与标准库路径标注。
- 每个诊断候选必须携带标准库路径；具体错误点未覆盖时生成待审核候选。
- 输出校验能阻止“有候选但无路径”或“未命中却硬绑已有错误点”的结果。
- 增加本次误诊样本的回归测试。

**Non-Goals:**

- 不引入多 Agent 治理链路。
- 不做标准库自动合并、去重、审核 UI。
- 不迁移数据库结构；待审核候选继续走现有 `libraryGrowth` 元数据。

## Decisions

1. **单次调用，双轨输出**
   - 模型在同一份 JSON 中输出 `diagnosisCandidates` 与标准库标注。
   - 取舍：比拆成两次调用更快，也避免第二次映射丢失诊断上下文。

2. **标准库路径必填，具体错误点可缺失**
   - `HIT` 必须绑定已有标准库 ID。
   - `PARTIAL` 应绑定最接近的知识点/能力点路径，并给出待审核新错误点。
   - `MISS` 可输出建议新路径，但必须说明当前标准库为什么覆盖不了。

3. **校验只守契约，不代替诊断**
   - 后端校验路径、ID、命中状态、候选字段关系。
   - 后端不根据运行结果模板化猜错因。

4. **回归样本聚焦根因**
   - 用“分层优惠最短路”样本要求 prompt 指向 `discounted = cost + weight` 与 `weight // 2`，防止误诊为起点初始化。

## Risks / Trade-offs

- **模型仍可能硬套标准库** -> prompt 明确“标准库是参考路径，不是答案来源”，并用测试锁住关键措辞。
- **待审核候选质量参差不齐** -> V1 只写入元数据，不自动审核通过；老师后续人工治理。
- **路径字段增加输出复杂度** -> 复用现有 `diagnosisCandidates` / `libraryGrowth`，避免新增持久化表。
