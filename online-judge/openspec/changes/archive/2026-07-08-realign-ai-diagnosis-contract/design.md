## Context

当前 AI 主链路的产品目标已经清楚：后端构造上下文包，外接大模型基于题目、代码、判题结果和标准库参考包完成诊断、打标签和缺口发现；后端校验后把学生可见反馈、教师可解释诊断和标准库成长候选分别落库或展示。

问题在于实现层仍混有旧假设：摘要文本被兜底成唯一建议，`get(0)` 或 `List.of(...)` 会吞掉多条结构化 item；提升建议被要求必须有代码证据，或者反过来借用第一条修正证据；标准库成长候选让模型承担来源 ID 和证据字段，导致候选池不稳定。用户已经确认标准库成长采用候选池模式，因此本设计以“候选池是唯一入库入口”为边界。

## Goals / Non-Goals

**Goals:**

- 让 AI 输出契约以结构化数组为主：修正、提升、标签映射、库外发现和成长候选互不覆盖。
- 让证据语义可解释：修正项需要当前提交证据；提升项允许没有直接代码证据，但不得伪造或借用证据。
- 让后处理只做局部修复、过滤和补全，不再整体覆盖模型返回的有效多条 item。
- 让标准库成长候选由模型提出内容、后端补齐来源和证据、教师审核后入正式库。
- 区分学生快反馈和教师深诊断的运行时预算，减少深度诊断被低延迟压缩或截断的概率。

**Non-Goals:**

- 不重建一套新的 AI 诊断 Agent 或平行标准库模块。
- 不允许 AI 自动直接写入正式标准库。
- 不在本变更重做教师端标准库治理台视觉设计。
- 不把标准库变成规则引擎；标准库仍是教学参考规范包。
- 不强迫每条提升建议都有代码证据。

## Decisions

### 1. 输出契约分为摘要、逐条建议、标签映射和成长候选

`studentReport` 保留为自然语言摘要；`basicLayerAdvice`、`improvementLayerAdvice`、学生端 `repairItems`、`improvementItems` 才是逐条建议主数据源。mapper 可以在模型完全没给结构化数组时生成降级项，但必须标记为降级来源，并不得用摘要覆盖已有多条有效 item。

替代方案是继续让摘要承载列表语义，再由前端拆段展示。该方案会让“单条建议”和“多条建议”不可验证，因此不采用。

### 2. 证据状态显式化，而不是用空数组表达所有情况

逐条 item 的证据状态分为三类：

- `SUPPORTED`：有可跳转 evidenceRefs。
- `NO_DIRECT_CODE_EVIDENCE`：没有直接代码行证据，但建议可以基于判题、复杂度、测试策略或学习迁移成立。
- `UNSUPPORTED`：模型声明有证据但引用无效，或建议与当前提交事实脱节。

后端验证策略是：修正项缺少证据时高风险，提升项可为 `NO_DIRECT_CODE_EVIDENCE`，但任何 item 只要给了 evidenceRefs 就必须可验证。这样既满足教学真实情况，又避免“看似有证据”的伪精确。

### 3. 禁止跨 item 自动借证据

mapper 和 view assembler 不得把全局 `analysis.evidenceRefs`、第一条基础证据、第一条 anchor 证据自动合并进每条建议。全局证据可以作为诊断概览存在，逐条 item 只能展示自身 evidenceRefs 或明确的无直接证据状态。

替代方案是在展示层继续合并全局证据，保证看起来每条都有证据。该方案会误导教师和学生定位代码行，因此不采用。

### 4. 标准库成长候选由后端补齐来源上下文

模型负责提出候选内容：名称、建议路径、错误症状、典型代码模式、学生解释、相似条目和理由。后端负责补齐当前 `problemId`、`submissionId`、诊断证据线索、命中状态和聚合键。候选默认进入待审核池；教师审核、编辑、合并或拒绝后才影响正式库。

替代方案是让模型在 JSON 中填 `sourceProblemId` 和 `sourceSubmissionId`。模型不可靠地记忆后端 ID，会制造空来源候选，因此不采用。

### 5. 教师深诊断使用更完整运行时预算

学生快反馈仍可使用低延迟压缩；教师端、作业题目分析、标准库成长和质量评测应使用标准或深诊断 profile，保留更多代码、失败用例、候选结构和输出 token 预算。兼容模型请求也应尽力传递 provider 支持的输出预算字段，或在不支持时记录 trace 风险。

替代方案是全链路统一低延迟 profile。该方案成本低，但会让教师端诊断和标准库成长长期缺上下文，因此不采用。

## Risks / Trade-offs

- 提升建议允许无直接代码证据可能放过空泛建议 → validator 要求标题、建议、收益和知识路径具体；前端展示“暂无直接代码证据”而不是证据卡。
- 禁止证据借用后，部分旧模型输出会显得证据更少 → 使用结构化重试或局部降级暴露问题，不用伪证据掩盖。
- 深诊断 profile 会增加调用成本和延迟 → 只用于教师深诊断、作业分析和候选池生成；学生快反馈保持低延迟。
- 候选池会增加教师审核工作 → 聚合同路径同名称候选，展示出现次数、来源提交和相似条目，降低审核噪音。
- 收紧 mapper 后可能暴露历史数据不完整 → 对历史记录保留兼容展示，但新诊断输出按新契约校验。

## Migration Plan

1. 先调整 DTO、validator、mapper 和 view assembler，使新旧输出都能归一到明确证据状态。
2. 再调整 prompt 和 runtime profile，让后续模型输出更稳定地满足新契约。
3. 然后补齐标准库成长候选来源与证据保存，默认只写候选池。
4. 最后更新学生/教师展示和回归测试，确认多条建议、无直接证据、候选池聚合都可见可验。
