## Context

当前活跃标准库已经将自动兜底层退役为冷存档，但存档中仍有大量知识点只被兜底覆盖。用户的目标不是把兜底原文搬回主库，而是把它能提供的所有价值分层榨取：主题可用就精修迁移，内容不可用但类型有启发就按类型重写，暂不适合的保留在存档。

## Goals / Non-Goals

**Goals:**

- 建立 A/B/C 三类兜底存档价值分类，能说明每类处理方式。
- 新增 V12 手写吸收条目，覆盖一批 archive-only 高价值主题。
- V12 条目必须提供能力点、易错点和提升点，且不能复用兜底模板化表达。
- 测试能验证分类目录、V12 质量和活跃知识点覆盖增长。

**Non-Goals:**

- 不把全部 223 个 archive-only 知识点一次性迁移。
- 不让自动兜底重新进入运行时候选或后备补位。
- 不用兜底原文作为正式描述；所有进入活跃库的内容都必须人工精修。

## Decisions

1. 用显式分类目录记录榨取决策。

   A 类为“直接精修吸收”，用于主题本身明确、教学价值高、能写出具体错误行为的知识点；B 类为“类型提炼重写”，用于兜底内容不好但错因类型或覆盖方向有价值的知识点；C 类为“继续存档”，用于暂不需要独立条目的低优先级或过细知识点。显式目录能防止未来只凭数量迁移。

2. V12 批次只吸收 A/B 中高价值主题。

   首批覆盖循环控制、前缀统计、字符串构造、栈队列、哈希、调试追踪和提交检查。它们来自 archive-only 缺口，同时又是学生真实错误高发区，适合先转为活跃标准库。

3. C 类不删除、不迁移、不参与运行时。

   C 类继续留在 `archivedGeneratedFallbackSeeds()`，作为后续扩张候选和审计材料。这样既不浪费覆盖地图，又不把低质量内容混入主库。

## Risks / Trade-offs

- 分类目录不是全量人工评审 → 通过测试确保 A/B/C 三类都有明确样本，后续批次继续扩展。
- V12 增量仍无法覆盖全部 archive-only 缺口 → 本次优先覆盖高价值主题，后续可以继续 V13/V14。
- 分类和条目可能重复已有主题 → 通过知识点覆盖和 code 前缀测试控制边界，保持 V12 聚焦 archive-only 缺口。
