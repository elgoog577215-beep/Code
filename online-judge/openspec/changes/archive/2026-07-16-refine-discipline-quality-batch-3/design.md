## Context

生产数据库已经完成 Flyway V2 的结构清理和 V3 的 MATH/ENG/BASIC 内容精修。V3 后共有 586 个启用知识点，其中 532 个仍使用“细颗粒知识点”占位描述；140 个启用能力点中有 53 个没有启用提升点。ALGO 和 DS 的模板节点分别承载 457 与 194 次规范能力点、易错点、提升点引用，是第三批最有价值的精修区域。

当前只有 55 条 `submission_diagnosis_facts`，其中正式标准库挂接仅 2 条，大多数为临时路径或未归类。行为证据可以帮助识别 Dijkstra、区间 DP、区间合并等重复出现路径，但不足以替代标准库结构引用和人工学科审校。

## Goals / Non-Goals

**Goals:**

- 精修 ALGO 20 个、DS 10 个高影响模板知识点，使定义、目标、任务、别名和前置知识具备真实学科语义。
- 为 12 个已有正式易错点但没有提升点的正式能力建立可执行、可验证的练习路径。
- 将模板债务从 532 降到不高于 502，将缺少提升点能力从 53 降到不高于 41，并保持既有结构门禁为零。
- 使用 Flyway V4、生产备份隔离演练、应用导航和同口径报告形成可回滚发布闭环。

**Non-Goals:**

- 不新增知识树节点，不改变既有稳定 code，不创建新的 ALGO/DS 平行知识库。
- 不修改题目、测试用例、提交、诊断事实或课堂数据。
- 不在本批停用 `SK_COMPAT_*`；只把它们排除在提升点补齐目标之外，退役另行审计历史引用。
- 不把 55 条诊断事实解释为全体学生错误分布，也不宣称 502 个剩余模板节点已经完成质量建设。

## Decisions

### 1. 结构引用为主证据，行为路径只做有上限的加权

候选必须是启用 `KNOWLEDGE_POINT` 且仍使用模板描述。规范结构引用数按能力点、易错点、提升点的不同条目去重；诊断路径中出现同名节点时增加有限加权，公式为 `reference_count + 2 × min(path_occurrences, 4)`。人工审校再检查学科中心性、分支连贯性和是否与已精修节点重复。

选择该方案而不是只按诊断次数排序，是因为当前行为样本小、正式挂接率低；选择它而不是只按结构引用排序，是为了让真实出现的 Dijkstra、区间 DP 和区间合并得到优先处理。

### 2. 第三批以连贯算法/数据结构分支为交付单位

30 个知识点按证据和主题连贯性形成算法 20、数据结构 10 的组合，覆盖最短路、拓扑、区间与贪心、二分、DP、滑动窗口、链表、图建模、队列和映射。比例不是平均配额，而是根据 ALGO 457 次、DS 194 次结构引用和行为路径分布确定。

每个节点更新 `description`、`learning_objectives`、`typical_problems`、`aliases`、`prerequisites`，并标记 `informatics-knowledge-discipline-v3`。前置知识只引用真实存在节点；无可靠前置时留空。

### 3. 提升点只补正式能力，兼容占位不凑覆盖率

候选能力必须启用、至少关联一个启用易错点、没有启用提升点，且 code 不以 `SK_COMPAT_` 开头、内容不含“用于兼容旧 AI 标准库标签”。本批选择 10 个 ALGO 和 2 个 DS 正式能力。

每个提升点必须包含适用边界、训练目标、具体练习、自测或状态表、学生收益和教师验收方法，并关联真实启用易错点。新增内容使用稳定 code，同步规范表、平铺快照和 legacy mapping。

### 4. V4 只做幂等内容更新，不重写旧迁移

V4 使用固定 `VALUES` 清单和 conflict-safe upsert。知识点只按稳定 code 更新；提升点先写规范表，再由同一临时清单同步兼容结构。V1、V2、V3 不修改，重复启动不得产生重复条目。

### 5. 发布门禁同时保护质量增量和业务稳定性

门禁新增 V4 专属计数、累计精修、模板上限、正式能力缺口上限、三表一致性、兼容能力误补检查，并继续执行所有 V2/V3 结构门禁。隔离恢复和生产发布前后比较题目、测试用例、提交、提交分析、诊断事实和学生反馈计数。

## Risks / Trade-offs

- [诊断路径使用中文名称，可能存在同名节点] → 行为次数只做上限为 4 的加权，最终选择必须同时满足结构引用和完整路径人工审校。
- [一次精修 30 个节点仍可能出现表述不均] → 按主题逐项审校，并为代表性 ALGO/DS 导航添加自动化测试。
- [新增提升点与历史平铺接口不一致] → 三表 code、能力归属、主知识点和关联错因加入阻断门禁。
- [兼容能力继续占据 53 个缺口中的一部分] → 报告分别列出正式缺口和兼容占位，不把兼容条目补写成正式教学内容。
- [生产发布失败] → 发布前创建并验证 custom-format 备份，保留旧镜像；Flyway 事务失败自动回滚，应用异常则恢复旧镜像，必要时按维护窗口恢复备份。

## Migration Plan

1. 固化生产 V3 基线、30 个知识点和 12 个能力点选择证据。
2. 新增 V4、门禁和自动化测试；运行空库 V1→V4、重复启动、完整后端回归和 OpenSpec 严格校验。
3. 在生产服务器创建备份、SHA-256 与元数据，并完成独立 PostgreSQL 恢复。
4. 在恢复库执行 V4，再用实际应用镜像验证 Flyway、Schema、内容、导航与业务计数。
5. 提交推送，构建与服务器架构一致的镜像，保留当前生产镜像作为回滚标签。
6. 发布生产，复核质量门禁、代表性导航、页面可达性和业务计数。
7. 生成 HTML 报告、更新项目决策并归档 OpenSpec。
