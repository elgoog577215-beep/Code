## Context

当前 AI 标准库已经经过多轮手写扩展与两期兜底素材吸收，自动生成兜底层的主要价值从“运行时覆盖保障”转为“历史素材池”。上一阶段已经让候选包和检索优先使用智能条目，但兜底条目仍会被 seed 到数据库，并在没有智能条目时参与补位，这与用户希望提升标准库质量和颗粒度的目标不一致。

约束是：不做数据库结构迁移，不物理删除历史记录，不影响已人工吸收的 V10/V11 智能条目；同时保留兜底条目的可识别能力，方便审计、回滚参考和未来人工精修。

## Goals / Non-Goals

**Goals:**

- 活跃标准库 seed 不再包含自动生成兜底能力点和易错点。
- 提供冷存档入口，能完整枚举自动生成兜底素材，供质量扫描和后续吸收使用。
- Seeder 在启动时禁用历史遗留兜底记录，避免旧库继续把兜底条目放进候选包、搜索定位和外部 AI 上下文。
- 测试明确区分活跃智能库与兜底存档库，防止未来误把兜底层重新接回运行链路。

**Non-Goals:**

- 不新增数据库表、字段或迁移脚本。
- 不删除兜底识别逻辑，不删除已吸收的 V10/V11 手写条目。
- 不继续扩大自动兜底生成规模；后续扩张应通过人工精修条目进入智能标准库。

## Decisions

1. 活跃 seed 与存档 seed 分流。

   `AiStandardLibrarySeedCatalog.seeds()` 只返回智能标准库条目；新增存档方法返回自动生成兜底素材。这样 fresh 数据库天然不会再插入兜底条目，而存档仍可被测试和后续工具读取。

   备选方案是保留兜底 seed 但全部置为 disabled。这个方案会继续污染主库统计和种子同步边界，因此不采用。

2. 历史兜底记录通过 Seeder 禁用。

   旧数据库中已经存在的兜底能力点和易错点不做物理删除，而是在扁平 Seeder 与规范化 Seeder 中识别兜底 code 并设置为 disabled。这样不会破坏已有数据审计，也能保证运行时查询默认排除它们。

   备选方案是写迁移删除历史行。这个方案不可逆，且用户还希望“榨干价值”和保留存档，因此不采用。

3. 保留兜底识别和质量校验。

   `isGeneratedFallbackSeed` / `isGeneratedFallbackCode` 继续存在，但用途从运行时补位改为存档识别、禁用历史数据和测试防回归。

   备选方案是彻底删除识别逻辑。这样会让旧数据清理和存档审计失去依据，因此不采用。

## Risks / Trade-offs

- 活跃标准库数量会明显下降 → 用测试更新新的活跃数量下限，并保留存档数量断言，避免误判为数据丢失。
- 某些极冷门知识节点短期内可能没有智能条目覆盖 → 通过后续人工精修扩张标准库解决，而不是重新启用模板兜底。
- 历史 disabled 记录仍在数据库中占空间 → 当前规模有限，且保留审计价值；若未来需要物理清理，可另起迁移规格处理。
- 服务层仍能识别手工启用的兜底条目 → Seeder 默认会禁用历史兜底；若管理员刻意启用，应视为显式运维行为，不作为默认运行策略。
