## Context

当前 `AiStandardLibrarySeedCatalog` 在手写智能条目之后调用 `generatedFullCoverage(...)`，为知识树中每个知识点补齐 1 个自动能力点和 1 个自动易错点。它保证覆盖，但生成条目会进入旧扁平表和规范化表，且默认 `enabled=true`，服务层目前无法区分“智能条目”和“兜底保险条目”。

有外部 AI 模型后，标准库的核心价值变成命名、颗粒度、教学结构和审核沉淀，而不是用模板条目替代 AI 判断。因此兜底层要保留为覆盖保险和素材池，但不能再作为候选包、搜索定位和学生可见诊断的主力。

## Goals / Non-Goals

**Goals:**
- 自动兜底条目可被稳定识别。
- 候选包与搜索定位默认优先使用手写智能条目；兜底只作为没有智能覆盖时的后备。
- 自动兜底生成器不再产出明显模板化低质量表达。
- 继续把高价值兜底素材迁移为手写智能库条目，并用测试约束吸收质量。

**Non-Goals:**
- 本轮不物理删除兜底条目。
- 本轮不新增数据库表字段或迁移脚本。
- 本轮不一次性手写迁移全部兜底素材。
- 本轮不改变外部 AI prompt 的核心诊断流程。

## Decisions

1. **用 seed 代码形态和 `libraryVersion` 识别兜底层。**
   自动兜底 code 由知识点 code 和 hash 生成，且缺少 `V6/V7/V8/V9/V10` 等手写前缀。实现一个集中判断器，避免服务层散落字符串规则。同步时给自动兜底使用独立 `libraryVersion`，便于后续查询和治理。

2. **服务层过滤优先于硬删除。**
   `AiStandardLibraryService` 在返回候选包、搜索定位项时先过滤掉自动兜底；如果过滤后没有可用智能条目，再回退兜底。这样保留覆盖保障，也避免兜底抢占智能库上下文。

3. **修生成器，而不是逐条修 137 个坏文本。**
   137 条明显低质都来自 `CONCEPT` 类型默认模板。直接改 `generatedMistakeName(...)`、`generatedMistakeDefinition(...)` 和必要的分类规则，能一次性消除模板病，并保证未来新增知识点不会重新生成同类空话。

4. **新增 V11 吸收 seed 作为迁移延续。**
   V10 已完成第一批吸收。本轮新增 V11，把典型兜底素材继续转化为能力点、易错点和提升点。后续可按同一模式持续扩张。

## Risks / Trade-offs

- **风险：兜底过滤过强导致候选为空。**
  缓解：过滤后为空时允许回退兜底，并用测试覆盖。

- **风险：不改表结构导致兜底标记不够显式。**
  缓解：集中封装识别逻辑，并用 `libraryVersion` 做辅助标记；后续需要治理后台字段时再迁移。

- **风险：自动生成条目修得更具体但仍不等于高质量手写。**
  缓解：明确它仍是后备层；质量主线继续靠 V11 及后续手写吸收。

- **风险：一次性迁移全部兜底成本过高。**
  缓解：先建立统计、识别和降级机制，分批吸收高价值主题。

## Migration Plan

1. 添加兜底识别和版本标记。
2. 更新服务层候选输出过滤规则。
3. 修复自动生成器低质模板。
4. 新增 V11 手写吸收条目与测试。
5. 运行标准库 seed、规范化同步、候选包、搜索定位和 OpenSpec 校验。
