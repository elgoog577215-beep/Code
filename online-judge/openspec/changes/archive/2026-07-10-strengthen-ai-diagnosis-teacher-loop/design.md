## Context

现有主链路已经具备正确的产品方向：外部模型先独立识别多个真实问题，后端再按问题逐层提供知识目录，模型选择知识路径，最终生成学生建议；未命中问题继承最后有效父路径，形成临时标准库条目。当前缺口主要来自实现和评测仍保留旧的成本保护假设。

- 知识归类默认只处理第一条有效问题，其余问题被写成“未匹配”，真实原因却是处理上限。
- 正式链路默认最多走六层，但高级真实样本评测为了节约调用只走一层，因此无法验证到达知识点诊断层。
- 最终建议按最低条数做硬校验，多问题时强制至少两条提升建议，容易把“覆盖真实问题”变成“满足数量”。
- 临时候选只保存最后一次来源提交和一个累计次数；相同提交再次生成会直接增加次数，并可能触发自动晋升。
- 教师修正和建议后续效果已经存在于后端学习轨迹中，但教师作业主工作台没有展示效果，校正表单也只能修改粗细错因。

本变更保持现有统一诊断与标准库成长链路，不新增本地规则诊断，不建立第二套教师分析系统。

## Goals / Non-Goals

**Goals:**

- 为每条有效问题独立完成知识归类，并保留每条问题真实的归类结果。
- 让真实样本评测按正式树深验证完整归类链路。
- 让基础建议覆盖真实问题，让提升建议数量由教学价值决定。
- 让临时标准库条目的晋升次数代表独立提交证据。
- 在教师主工作台展示 AI 建议后的后续效果，并支持四类细分校正。
- 让教师细分校正进入后续诊断材料，形成可追溯闭环。

**Non-Goals:**

- 不改成关键词或本地规则自动选择知识路径。
- 不用本地模板生成学生可见 AI 内容。
- 不新建第二棵知识树或平行标准库。
- 不把“学生后续通过”宣称为 AI 导致，只记录为查看建议后的观察结果。
- 不在本轮实现每张建议卡的点击追踪或严格因果分析。

## Decisions

### 1. 所有有效问题都进入独立归类，不再使用问题数量上限

后端继续对自由诊断输出做证据、标题、置信度和重复性校验；通过校验的问题全部进入逐层知识归类。归类失败只影响当前问题，不能借“处理上限”把后续问题标成没有匹配。

选择直接处理全部有效问题，而不是提高默认上限，是因为产品契约要求多问题保真。模型通常只会返回少量主要问题；极端异常输出由结构、重复和输出预算门禁处理，而不是在归类阶段静默丢弃。

### 2. 正式链路与真实样本评测使用相同默认树深

真实样本评测默认最多走六层，与正式运行一致，足以覆盖根目录、大章节、小章节、知识点和诊断层。单轮配置只保留给专门测试“轮次耗尽”场景的单元测试，不能作为完整链路报告默认值。

完整链路测试需要断言至少一个问题能到达知识点下的正式或临时诊断节点；若测试数据库内容不足，应显式构造最小知识树夹具，而不是把轮次降为一层。

### 3. 建议校验从最低条数改为问题覆盖

基础建议必须通过 `issueId` 关联真实问题。模型漏掉 `issueId` 时，后端可以按稳定顺序补齐一次并记录局部修复；同一个问题可以有一条主要建议，但不能用一条笼统建议覆盖多个互不相同的问题。

未通过的提交至少要覆盖所有有效的阻塞和主要问题；次要问题可以保留在摘要或教师信息中。提升建议允许为空、一条或多条，只检查是否具体、是否与当前问题或算法机制相关、是否重复基础建议。

只有基础问题覆盖不足时才触发一次结构化重写；不再因为提升建议数量少而重试。

### 4. 临时条目按独立提交来源累计

成长候选新增已观察提交 ID 集合。新候选保存当前来源提交；聚合重复候选时：

- 来源提交 ID 新出现，才增加 `occurrenceCount`。
- 同一提交重试、刷新或重复生成，只更新时间、证据和相似项，不增加次数。
- 没有来源提交 ID 的候选可以创建和人工治理，但不得靠重复写入达到自动晋升门槛。

保留现有 `sourceSubmissionId` 作为最近来源兼容字段，新增集合字段用于独立计数和审计。旧候选迁移时若已有来源提交，则把它作为首个已观察来源；无来源的旧候选保留原状态，但新的自动晋升判断要求可验证的独立来源集合。

### 5. 教师主工作台复用现有效果分析

作业总览已经拥有本次作业的提交、分析和反馈查看事件，因此直接复用现有“查看建议后下一次同题提交”分析，给每个学生题目摘要增加最近建议效果：已改善、仍卡同类、问题转移、回退、阶段变化或等待后续提交。

教师题目分析的学生证据详情增加“建议后续效果”卡片。该卡片只使用“查看后”表述，不把时间先后关系写成因果结论；仍卡同类或回退时自动提高教师关注提示。

### 6. 教师校正扩展为四类问题

教师校正继续要求选择修正后的主要错因，同时新增：

- 诊断判断有误。
- 知识路径有误。
- 代码或评测证据有误。
- 建议表达或行动方向较弱。

校正记录可保存目标问题 ID、修正后的知识路径和目标证据引用。教师工作台按校正类型展示对应输入；这些字段进入校正响应、审计记录和后续诊断的教师校准摘要。这样下一次模型能知道老师是在否定错因、修改归属、纠正证据，还是只要求改善表达。

### 7. 测试按五条根因分别建立回归

先增加会在当前实现失败的测试：三条问题全部归类、五层目录到达诊断层、一条提升建议合法、同一提交重复不增加次数、教师效果进入作业工作台且细分校正进入后续诊断摘要。每条根因独立验证，避免一次大测试无法定位失败层。

## Risks / Trade-offs

- [处理全部问题会增加外部调用次数] → 只处理通过证据与重复校验的真实问题；每条问题仍受单问题最大导航轮次约束，并在 trace 中记录调用数量。
- [真实样本完整树深增加测试成本] → 完整链路样本数量保持小而真实；结构单元测试使用本地可控响应，不依赖外部网络。
- [按问题覆盖可能暴露模型长期漏写 `issueId`] → 允许一次按顺序局部补齐并记录，后续评测统计补齐次数，逐步收紧提示词。
- [旧成长候选没有来源集合] → 兼容读取旧字段；没有独立来源证据时不自动晋升，但仍可教师审核。
- [教师表单字段增加] → 默认选择“诊断判断有误”，其他字段按类型展开，避免一次显示所有高级输入。
- [建议后改善不代表因果] → 页面统一使用“查看后观察”措辞，并保留等待、无明确变化等状态。

## Migration Plan

1. 先增加兼容字段和读取逻辑，旧成长候选、旧教师校正继续可读。
2. 部署前备份正式数据库，由 JPA 兼容更新或显式数据库迁移增加来源集合及教师校正字段。
3. 上线全部问题归类、覆盖校验和完整树深评测。
4. 上线教师效果卡片与细分校正表单，观察旧数据空字段展示。
5. 验证独立来源累计后再保持自动晋升开启；若异常，可关闭自动晋升但保留候选和审计。

回滚时，新字段保留不删除；代码回退后旧字段仍可工作。全部问题归类和建议覆盖校验通过提交回退恢复，不保留平行运行开关。
