## Context

当前实时链路已经是“本地召回 + 单诊断 Agent”，但本地召回只提供基础候选，正式提示词还没有把对照实验中的自然反馈能力充分吸收，评测报告也没有专门围绕学生可见反馈和标准库误用形成闭环。标准库成长已有候选服务基础，本次应复用现有能力补齐闭环。

## Goals / Non-Goals

**Goals:**

- 保持实时链路单次外部诊断调用。
- 提升 `diagnosis-report-v2` 的学生可见文案质量和安全边界。
- 让本地召回包携带树形路径、召回来源和邻近易错点。
- 用 30-50 道 fixture 生成可审计 JSON/Markdown 质量报告。
- 让库外发现进入候选池，聚合后教师审核入库。

**Non-Goals:**

- 不恢复双外部 Agent 默认链路。
- 不引入新的向量数据库或外部检索服务。
- 不做复杂多角色审批。
- 不让 AI 实时自动修改正式标准库。
- 不把后端规则改造成第二个诊断模型。

## Decisions

### 1. 质量闭环复用现有测试侧报告能力

继续使用现有 eval/report 测试基础设施，新增 fixture 与轻量评估器，而不是新建运行时服务。理由：质量评测是离线门禁，不应污染学生提交实时路径。

### 2. 本地召回增强而不是新增搜索 Agent

召回服务继续在后端本地完成，候选包增加来源、父级路径、同层易错点和少量延伸点。理由：这能给单诊断 Agent 足够上下文，同时不增加外部调用次数。

### 3. Prompt 保持一个正式模板

只打磨 `diagnosis-report-v2`，不新增多个 prompt 版本。理由：当前真正问题是主模板质量，不是版本数量。

### 4. 标准库成长只写候选池

`OUT_OF_LIBRARY` 与 `libraryGrowth.candidates` 只进入候选池并聚合，不直接入正式库。教师批准后复用现有标准库保存能力并触发 embedding stale。

## Risks / Trade-offs

- 评测 fixture 数量增加维护成本 → 使用 JSON 资源集中维护，报告自动生成。
- 本地召回上下文过长 → 默认限制 20-40 个候选，邻近条目限量。
- Prompt 长度约束过严影响表达 → 先做软约束和测试门禁，不在运行时截断语义。
- 教师审核界面可能变复杂 → 第一版只做候选列表、详情、批准、拒绝。

## Migration Plan

1. 创建 OpenSpec change 并通过严格校验。
2. 修改 prompt、召回包、trace、评测和成长候选服务。
3. 增加后端测试和 fixture。
4. 如触碰教师端，补前端 typecheck。
5. 跑 `openspec validate <change> --strict` 与 `bash ./mvnw -Dskip.frontend=true test`。
