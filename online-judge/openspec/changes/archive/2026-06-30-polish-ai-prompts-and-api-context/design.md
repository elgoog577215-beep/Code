# 设计

## 正式链路
继续保持简单链路：

```text
题目、代码、判题结果、证据
-> 本地召回树形标准库候选包
-> 单诊断 Agent
-> studentReport + 结构化元数据
-> 后端校验
-> 学生端优先展示 studentReport
```

## 提示词优化原则
1. 模型先整体理解题目、代码和判题结果，再参考标准库候选。
2. 标准库候选只提供知识路径、能力点和易错点语言，不替模型做最终判断。
3. 如果候选不匹配，模型必须可以输出 `OUT_OF_LIBRARY` 并说明原因。
4. 学生可见文本要自然，不能把结构化字段感直接暴露给学生。
5. 基础层优先解释当前阻塞 AC 的问题；提高层只给修复后的学习方向。
6. 输出仍保留结构化元数据，供后端校验、教师追踪和评测闭环使用。

## API 上下文优化原则
- 输入分区清楚：任务、题目、代码、判题、证据、标准库候选、输出要求。
- 标准库上下文必须说明树形路径含义，减少模型把孤立条目当答案。
- 约束语要少而硬：不泄露完整答案、不猜隐藏测试、不强行套库。
- 学生文案长度要受控，但不强迫每句话填固定字段。

## 最小实现范围
- 修改正式 `diagnosis-report-v2` 模板。
- 修改必要的 API 输入说明文字。
- 如辅助模板已经进入学生或教师可见 AI 功能，只做统一边界提示，不做重构。
- 补测试验证关键提示词约束存在。
