## Context

当前系统已有统一信息学知识树、规范标准库表和旧扁平兼容表。知识树节点包含 `DOMAIN / CHAPTER / TOPIC / KNOWLEDGE_POINT`，规范标准库包含能力点、易错点和提升点。问题不在于缺少物理表，而在于语义和 AI 包装还没有完全收敛：候选包的知识组名仍偏 code，能力点主锚点没有被显式约束为知识点，prompt 仍需要更明确地表达“标准库诊断层属于知识点以下”。

## Goals / Non-Goals

**Goals:**

- 把数据库语义统一为一棵树：知识树到知识点为止，知识点下面挂能力点，能力点下面挂易错点。
- 让 AI 候选包按知识点中文路径组织能力点、易错点和提升点。
- 让标准库同步器和测试约束能力点主知识节点必须是 `KNOWLEDGE_POINT`。
- 保持旧接口和兼容字段可用，避免破坏现有教师端和历史数据。

**Non-Goals:**

- 不新增高中库或竞赛库。
- 不做破坏性数据库迁移，不删除旧扁平表。
- 不在本次实现 embedding 召回或外部搜索 Agent 默认开启。
- 不一次性清理所有历史 KB_* 展示问题。

## Decisions

1. **复用现有物理表，强化语义约束。**
   - 理由：现有表已经表达三层结构，直接迁移表会增加风险。
   - 取舍：旧兼容表仍存在，但 AI 主链路和新增测试以规范表为准。

2. **能力点主锚点只允许指向知识点。**
   - 理由：用户要求知识树末端是知识点，能力点是知识点下面的诊断层。
   - 取舍：相关知识节点仍可保存 topic 或其他辅助节点用于检索，但不能作为主路径。

3. **AI 候选包用知识节点仓库补全中文名称和路径。**
   - 理由：模型和教师需要看到“基础语法 / 字符串构造 / 游程编码”，而不是只看到 code。
   - 取舍：如果历史数据找不到节点，仍回退到 code，避免运行失败。

4. **召回仍负责圈范围，最终判断交给单诊断 Agent。**
   - 理由：标准库是教学坐标系和候选地图，不是规则引擎。
   - 取舍：本地召回可能仍有弱相关候选，但 prompt 和 metadata 会继续允许 HIT、PARTIAL、MISS 和 OUT_OF_LIBRARY。

## Risks / Trade-offs

- **历史 seed 可能主锚点不是知识点** -> 通过测试扫描现有活跃 seed，先定位并修正高风险条目，必要时保留相关节点为辅助上下文。
- **候选包中文路径缺失** -> 查询知识树失败时回退 code，但测试覆盖 V14 代表路径。
- **旧教师管理视图仍混入 KB_* 条目** -> 本次不扩大到 UI 清理，后续单独收敛教师视图。
- **结构更严格后可能暴露坏数据** -> 同步器先记录/测试约束 seed，避免启动时对历史库做破坏性删除。
