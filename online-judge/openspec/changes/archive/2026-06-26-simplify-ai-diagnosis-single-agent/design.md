## Context

当前 AI 诊断实时链路包含两个大模型阶段：搜索定位阶段和诊断报告阶段。搜索定位阶段已经需要阅读题目、代码、判题结果和标准库候选，实际上会做一次诊断判断；诊断报告阶段随后再次阅读同类上下文并重新判断，造成重复推理、延迟增加和表达割裂。

本项目已有本地标准库、标准库召回服务、树形知识结构、诊断报告 v2 输出、学生端自然段落展示和后端校验能力。本次改造应复用这些基础设施，不新增复杂服务。

## Goals / Non-Goals

**Goals:**

- 实时链路默认只调用一次外部大模型。
- 搜索定位从“大模型 Agent”降级为本地召回和上下文组装步骤。
- 单诊断 Agent 同时负责定位、取舍、学生反馈、标准库命中和库外发现。
- 保留树形标准库上下文、证据引用、答案泄露校验和降级机制。
- 保持现有学生端响应字段兼容，优先展示自然语言报告。

**Non-Goals:**

- 不重写标准库内容。
- 不引入新的向量数据库或外部检索服务。
- 不做教师审批流改造。
- 不删除历史测试和历史归档，只停止默认实时链路使用外部搜索定位调用。
- 不把后端校验升级成第二个诊断器。

## Decisions

### 1. 搜索定位不再默认调用外部模型

默认实时链路跳过 `search-location-v1` 外部调用。后端仍可执行 `SearchLocationRetrievalService.retrieve` 获取候选包，并用本地选择逻辑组装给诊断报告阶段。

理由：召回阶段的目标是“不漏主要方向”，不是最终诊断；这类工作由本地结构、关键词、规则和向量召回完成更稳定、更便宜。

替代方案：继续保留双模型调用。放弃原因是重复推理、延迟高，且对照实验显示单次完整诊断更自然。

### 2. 单诊断 Agent 使用大提示词完成完整判断

`diagnosis-report-v2` 继续作为正式提示词版本，但文案要强调：模型需要完整阅读题目、代码、判题结果和候选标准库，自己判断候选命中、干扰或库外问题，并生成学生自然反馈。

理由：这样保留提示词模板的教学表达能力，同时用 metadata 保持后端可校验性。

### 3. 输出保持“学生报告 + 结构化元数据”

`studentReport` 是学生端主输出；`diagnosisDecision`、`teacherTrace`、`libraryGrowth` 是后端和教师端使用的结构化信息。

理由：学生需要自然、通俗、连贯的解释；系统需要可统计、可追踪、可沉淀的数据。两者不应互相绑架。

### 4. 后端校验只守底线

校验只检查非空、基础层、证据引用、标准库 ID、安全泄露、JSON 可解析等底线问题，不重新判断诊断是否正确。

理由：正确性应靠评测集、人工复盘和模型质量迭代，不靠硬规则模拟老师判断。

## Risks / Trade-offs

- 搜索定位 trace 语义变化 → 在状态中明确标记为本地召回/跳过外部搜索，避免误判为模型失败。
- 本地召回漏掉主因 → 用评测 fixture 验证 Top 候选覆盖率，后续再调召回权重或引入向量补充。
- 单次模型调用输出过长 → 在提示词中限制学生报告结构和篇幅，并在后端保留长度截断/降级保护。
- 模型给出过多答案细节 → 保留现有答案泄露校验和安全失败降级。

## Migration Plan

1. 调整默认配置，使实时搜索定位外部模型调用关闭。
2. 调整编排逻辑：本地召回候选后直接进入 `diagnosis-report-v2`。
3. 更新 `diagnosis-report-v2` 提示词，强调单诊断 Agent 完整判断。
4. 更新测试，断言默认链路只调用一次外部模型。
5. 保留旧搜索定位模型调用开关，便于临时对照和回滚。
