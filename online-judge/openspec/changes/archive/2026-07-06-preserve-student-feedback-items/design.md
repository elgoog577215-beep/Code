## Context

学生快反馈链路已经把 `studentReport` 定位为摘要，把 `repairItems` 与 `improvementItems` 定位为证据驱动的逐条列表。Prompt 和 normalizer 均允许 0 到多条结构化建议，现有测试也证明模型返回多条时基础归一化能保留多条。

实际页面仍出现“永远像一条”的体验，根因集中在模型输出之后：运行时越界兜底会用 `List.of(...)` 整体覆盖修正项和提升项，提高层漂移校正会用单条边界样例覆盖所有提升项，前端摘要区域又把第一条 item 的证据和逐条列表重复渲染。

## Goals / Non-Goals

**Goals:**

- 后处理只修正不可信条目，不整体吞掉模型返回的其他有效结构化建议。
- 越界兜底在模型未命中主因时补充或置顶可信越界项，同时保留其他非重复建议。
- 提高层漂移校正只替换跑偏的“精简代码/删除 helper”类提升项，保留其他提升项。
- 前端让摘要和逐条建议分工清楚，避免第一条建议的证据重复出现。
- 用回归测试覆盖后端数组保真和前端展示去重。

**Non-Goals:**

- 不更换外部大模型、不新增模型调用阶段。
- 不修改数据库结构或学生反馈 API 字段。
- 不让系统为了数量凑建议；仍然按证据决定 0 到多条。
- 不重做整页视觉设计，本次只修复结构展示错误。

## Decisions

1. **保留数组，局部替换。**  
   越界兜底不再直接构造唯一的 `repairItems` / `improvementItems`。当模型未提到 `IndexError` 主因时，后端会生成可信越界主项并与原有有效项合并；重复或漂移项会被过滤。这样比完全信任模型更安全，也比整体覆盖更符合多条契约。

2. **提高层漂移只替换坏项。**  
   `alignIndexRuntimeImprovement` 只删除或替换明显跑偏的代码整理建议，保留已有自测、变量追踪、边界样例等有效提升项。如果替换后没有有效提升项，则补一条边界样例意识。

3. **摘要不承载 item 证据。**  
   前端 `studentReport` 只显示总结文本，逐条列表展示每条 item 的标题、正文和证据。若保留主证据展示，逐条列表必须从第二条开始；本次采用更清晰的“摘要 + 全量逐条列表”结构，避免重复。

4. **测试先覆盖已知断点。**  
   后端测试用 stub 模型构造多条输出，并触发 runtime grounding 与 drift alignment；前端测试用静态渲染验证同一个标题和代码证据不会被摘要区重复输出。

## Risks / Trade-offs

- [Risk] 保留模型原有其他项可能保留轻微低质量建议。→ 后处理继续按安全和重复规则过滤，漂移检测只清除明确不可信项。
- [Risk] 前端从“主证据 + 逐条”改为“摘要 + 逐条”会改变视觉密度。→ 保持现有区块和 CSS 类，先减少重复，后续再做完整视觉重构。
- [Risk] 运行时兜底生成的主项可能与模型已有越界项重复。→ 合并时按标题、正文和知识路径去重，并优先保留后端可信主因项。
