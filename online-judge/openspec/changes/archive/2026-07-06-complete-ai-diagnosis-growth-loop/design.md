## Context

上一阶段已经完成主 AI 调用契约：模型在同一次诊断里自由判断真实问题，并同步输出标准库路径、命中状态和待审核候选。当前缺口不在“方向是否成立”，而在完整产品闭环：真实模型是否稳定按契约输出、上下文包是否足够接近 Codex 的局部全貌、教师是否能在页面处理候选、标准库治理是否可度量。

现有代码已经提供了重要基础：

- `diagnosis-report-v2` prompt、DTO 和 validator 已支持 `libraryPath` 与 `libraryGrowth.candidates`。
- `AiStandardLibraryGrowthAgentService` 已支持候选创建、聚合、忽略、拒绝、批准、合并和自动合并门禁。
- `/api/teacher/ai-standard-library/growth-candidates` 已有列表、详情和审核动作接口。
- 教师端 AI 标准库页面已经是树状管理空间，但尚未展示成长候选和治理摘要。

## Goals / Non-Goals

**Goals:**

- 用真实样本和自动化门禁验证“主因 + 标准库路径 + 待审核候选”是否稳定。
- 明确上下文包质量要求：完整题目、完整代码、判题参考信号和结构化标准库邻域必须同时存在。
- 把已有成长候选后端能力接到教师端，让老师能看到、编辑、通过、合并、拒绝或忽略候选。
- 增加治理摘要，展示待审核积压、重复聚合、路径薄弱和高频未命中。
- 保持标准库正式入库必须经教师审核或显式自动合并门禁。

**Non-Goals:**

- 不新增独立治理 Agent。
- 不把全部标准库无差别塞进模型 prompt。
- 不重做标准库主结构或迁移既有表。
- 不把标准库候选直接展示给学生。

## Decisions

1. **真实模型回归先接现有 eval 体系**

   复用 `diagnosis-report-v2` 真实样本和 live eval 报告，而不是新建一套评测框架。新增质量维度检查候选路径、`PARTIAL` 合理性、成长候选状态和学生可见文案隔离。

2. **上下文包采用“完整代码 + 相关结构包”**

   题目和学生代码应尽量完整进入正式诊断；标准库仍采用相关结构包，包含知识路径、能力点、易错点、提升点和相邻候选。这样既释放模型诊断能力，也避免 token 被全库吞掉。

3. **教师审核复用现有成长候选状态机**

   后端已有 `PROPOSED / NEEDS_REVIEW / BLOCKED / MERGED_SIMILAR / TEACHER_APPROVED / REJECTED / MERGED / IGNORED`。本轮只补摘要、筛选和前端入口，不新增平行状态。

4. **治理质量先做可解释摘要**

   治理摘要由候选池实时聚合：按状态、路径、出现次数和相似条目统计。先帮助老师发现积压和薄弱路径，后续再考虑自动治理 Agent。

5. **前端入口融入 AI 标准库页面**

   AI 标准库页面保留大树状管理空间，新增“正式库 / 待审核 / 治理”视图切换。候选详情以同页面板或抽屉呈现，避免回到拥挤的左右栏。

## Risks / Trade-offs

- 真实模型回归可能受网络、配额或模型波动影响 → 保留离线 fixture 门禁，并把 live eval 作为可选增强。
- 教师端候选审核可能增加页面复杂度 → 使用视图切换和摘要卡片，不把候选列表塞进树节点内部。
- 候选自动合并可能污染标准库 → 默认仍走教师审核；自动合并只在已有配置开启且置信度达标时生效。
- 上下文包变大可能影响速度 → 只扩大完整题目/完整代码保障，标准库继续使用相关结构包与邻域。
