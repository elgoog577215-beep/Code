## Context

正式知识树和标准库已改为数据库主库，生产 PostgreSQL 当前包含 737 个知识节点、140 个启用能力点、362 个启用易错点和 75 个启用提升点。结构层级本身完整，但内容质量存在五类可复现问题：全部 586 个知识点仍是同一占位描述；主题和知识点普遍把父目录当作前置知识；启用关系和兼容映射大量指向已停用的自动兜底条目；分类字段泄漏历史扩库版本；一条教师批准易错点写入了不存在的占位知识路径。

学科参照采用两层来源：教育部《普通高中课程方案和语文等学科课程标准（2017 年版 2020 年修订）》用于高中核心语境，CCF《NOI 大纲（2025 年修订版）》用于入门级、提高级和 NOI 级竞赛范围。二者存在交集，不能继续用单个 `stage` 字符串表达互斥分支。

## Goals / Non-Goals

**Goals:**

- 建立同一知识节点可映射多个学科框架和范围的数据库结构。
- 修复可以由结构证据直接判定的关系与元数据错误，不改变稳定 code。
- 首批人工精修一组高引用核心知识点，并留下可量化的后续债务。
- 让生产发布能够自动阻断孤儿关系、无效正式条目、重复正式错因和实现词泄漏。
- 通过 Flyway、隔离 PostgreSQL、备份与恢复演练完成可回滚发布。

**Non-Goals:**

- 不在本轮一次性重写全部 586 个知识点，也不自动生成大批描述。
- 不拆分高中库、CSP 库和 NOI 库，不重建现有知识树 code。
- 不改变 AI 诊断协议、前端治理页面或现有 API 返回结构。
- 不自动批准待审核成长候选，不覆盖教师审核结论。

## Decisions

### 1. 用多对多范围映射代替重载 `stage`

新增 `informatics_discipline_scope_mappings`，以 `knowledge_node_code + framework_code + scope_code` 唯一标识学科范围。映射保存来源 URL、覆盖角色、证据说明和审核状态。首批只映射有高置信依据的领域或章节锚点，下级节点通过最近祖先继承范围；不为每个叶子伪造精确出处。

备选方案是直接改写 `stage`。该方案无法表达一个知识点同时属于高中课程和 NOI 入门级，且会破坏既有读取语义，因此不采用。

### 2. 区分阻断性错误和渐进式质量债务

质量脚本输出两组指标：孤儿正式条目、失效启用关系、无效启用映射、重复正式错因和占位知识路径属于阻断项；模板描述、别名不足、提升点覆盖率和人工精修覆盖率属于债务项。发布门禁只允许债务按已记录基线逐批下降，不会因尚未一次清零而失去可执行性。

### 3. 确定性清理采用一次性 Flyway 数据迁移

迁移会清空等于父节点 code 的伪前置字段、清空等于自身主名的无效别名、停用端点不存在或已停用的关系、将失效兼容映射标记为归档，并把教师批准的周期等待易错点重挂到现有最短路松弛能力与知识点。所有操作使用稳定条件和幂等 SQL，不删除历史业务记录。

### 4. 分类只清除实现来源，不改稳定 ID

规范化表和兼容快照的 `category` 移除 `V13/V14/V15/V16`、`兜底吸收`、`兜底榨取-A类/B类` 等生产实现词，保留其后的主题语义。`code` 不改名，避免历史诊断、教师审核和学生反馈失去引用。

### 5. 首批人工精修按“学科中心性 × 当前引用密度”选择

第一批优先覆盖程序基础、数据结构、算法、数学、调试和竞赛实践中的高引用节点。每条精修必须写清概念边界、可观察学习目标、典型任务、有效别名和真实前置知识；没有可靠前置关系时留空，不再用父目录冒充。

### 6. 精确错因节点通过能力引用进入诊断层

能力点可以覆盖多个相关知识点，易错点和提升点的主知识节点应保留最精确的学科归属。诊断层展开某个知识点时，除直接挂在该节点的能力点外，还要解析该节点易错点或提升点引用的启用能力点，并把这些直接内容放回对应能力点下；不要求为了页面可见性把精确错因粗暴改挂到能力点的主节点。

## Risks / Trade-offs

- [风险] 首批精修覆盖率仍较低 → 质量报告明确剩余模板数量，后续按主题包继续下降，禁止用批量模板清零。
- [风险] 停用关系可能影响历史兼容读取 → 只停用端点本身已停用或不存在的关系，规范化启用条目的关系保留，并在导航 API 上复验。
- [风险] 清理分类后旧分类搜索结果变化 → 稳定 code、名称和知识路径不变；只删除无教学含义的版本或兜底来源词。
- [风险] 学科来源映射被误解为逐条官方认证 → 映射区分直接锚点与继承范围，并保存证据说明；首批不对叶子做过度精确声明。
- [风险] 数据迁移不可用普通 SQL 回滚 → 发布前生成可校验 custom-format 备份并完成隔离恢复演练；生产异常时恢复备份和上一应用镜像。

## Migration Plan

1. 在隔离 PostgreSQL 上执行完整 Flyway 链并通过 Hibernate `validate`。
2. 在生产执行只读质量画像，保存迁移前指标和业务计数。
3. 生成 custom-format 备份、校验 SHA-256，并在隔离容器完成恢复演练。
4. 推送代码，服务器快进到目标提交，应用启动时由 Flyway 执行 V2。
5. 运行学科质量硬门禁、Schema readiness、内容 readiness 和代表性导航 API smoke。
6. 若迁移或导航验证失败，停止替换；若上线后出现数据回归，恢复备份并回到上一镜像。

## Open Questions

- 叶子知识点与教育部课程模块的逐条直接映射需要后续学科教师审校，本轮只建立机制和高置信锚点。
- 余下模板知识点的优先级将结合真实诊断命中、教师候选积压和学科中心性继续排序。
