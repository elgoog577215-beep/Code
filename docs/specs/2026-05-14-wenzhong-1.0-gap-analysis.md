# 温中 AI 编程学习平台 1.0 目标对比与精进路线

## 1. 目标状态

1.0 的理想状态不是一个换皮 OJ，而是一个可用于温岭中学信息技术课堂试点的过程评价与学习辅助平台。

核心标准：

- 学生通过邀请码进入老师布置的任务，完成 Python/C++ 练习。
- AI 反馈提供问题类型、定位方向和修改思路，但不直接给答案。
- 每次提交都沉淀为学习证据，能看出学生从语法、边界、逻辑、复杂度到代码质量的变化。
- 教师不需要逐个翻代码，也能看到班级卡点、高频错因和需要关注的学生。
- 原 OJ 只作为评测内核存在，外层产品语言、业务模型和数据结构都服务教学。

## 2. 当前已达成

### 2.1 学生闭环

- 已有学生入口 `/student.html`。
- 支持邀请码进入任务，默认演示邀请码为 `WZAI01`。
- 支持学生填写班级、姓名、学号/座号，生成学生画像。
- 学习任务卡片会携带 `assignmentId` 和 `studentProfileId` 跳转到做题页。
- Python 提交流程已验证可跑通。

### 2.2 教师闭环

- 已有教师工作台 `/teacher.html`。
- 后端已有班级、作业、邀请码、作业任务、学生画像模型。
- 教师端能看到作业、邀请码、参与人数、尝试次数、通过次数、需关注人数。
- 教师端已显示高频诊断标签和学生过程摘要。

### 2.3 AI 诊断闭环

- 提交后会生成结构化诊断。
- 诊断结果包含：
  - `issueTags`
  - `abilityPoints`
  - `studentHint`
  - `teacherNote`
  - `progressSignal`
  - `confidence`
  - `answerLeakRisk`
- 已增加二次安全检查，疑似完整代码、直接答案或隐藏测试信息会降级为方向性提示。
- 教师端已能统计标准化错因标签。

### 2.4 工程与演示稳定性

- 已将 HTML 缓存改为不可长期缓存，避免学生端拿到旧页面。
- CSS/JS 仍通过版本号和长缓存保证性能。
- Maven 编译编码已锁定 UTF-8。
- 本地 8081 服务可启动，核心页面可访问。

## 3. 与目标的主要差距

### 差距 A：学生学习体验还不够“闭环”

当前学生能提交并看到 AI 反馈，但缺少专门的“我的学习轨迹”视图。学生现在需要在题目页里看历史，体验仍偏工程工具。

补齐方案：

- 新增学生任务详情页或学生侧轨迹区。
- 按任务展示最近提交序列、问题标签变化、是否连续卡在同一能力点。
- 用学生能理解的语言展示：这次比上次改善了什么、下一步只做哪一件事。

优先级：P0。

### 差距 B：教师端还缺真正的作业创建与数据导入

当前教师端主要展示试点作业，后端已有创建接口，但前端还没有完整表单和导入流程。

补齐方案：

- 教师端新增创建作业抽屉：
  - 选择班级
  - 选择题目
  - 设置开始/截止时间
  - 设置提示层级上限
  - 生成/重置邀请码
- 新增学生名单导入：
  - CSV
  - XLSX
  - 纯文本名单
  - 支持列名识别和预览确认
- 新增题目导入：
  - Markdown + 测试点
  - ZIP
  - CSV
  - 手动粘贴

优先级：P0。

### 差距 C：诊断标签库还只是代码常量，不是可治理标准库

当前标签已经进入结构化结果，但还没有独立标准库、标签解释、能力映射和版本管理。

补齐方案：

- 建立 `diagnosis_taxonomy` 标准库：
  - 标签 ID
  - 中文名称
  - 学生解释
  - 教师解释
  - 所属能力点
  - 推荐提示层级
  - 示例
- AI 只能从标准库中选择标签，不能自由发明。
- 教师端展示中文标签和解释。

优先级：P0。

### 差距 D：AI 防泄题还需要更严格的流水线

当前已有关键词级二次检查，但成熟产品需要更强的结构化校验。

补齐方案：

- 将 AI 输出拆成四级提示：
  - L1 问题类型
  - L2 定位方向
  - L3 局部解释
  - L4 参考改法
- 默认只开放 L1-L2。
- AI 输出后做结构化校验：
  - 是否包含完整代码
  - 是否包含隐藏测试点
  - 是否直接给最终公式或完整步骤
  - 是否越过当前作业提示上限
- 高风险输出不入库，只保存安全降级版本。

优先级：P0。

### 差距 E：C++ 能力入口有了，但本机运行环境未完成

前端保留 Python/C++，后端也支持 C++ 语言 ID，但当前 Windows 环境没有 `gcc/g++`，所以本机 C++ 无法实际运行。

补齐方案：

- 本地试点安装 MinGW-w64 或 LLVM。
- 长期改为容器化执行沙箱。
- 后端启动时检测运行环境，教师端展示“Python 可用 / C++ 未配置”等状态。

优先级：P0，尤其演示前必须处理。

### 差距 F：旧 OJ 页面仍有残留

首页、学生入口、教师端、做题页已明显转向学习平台，但任务编辑页和排行榜页仍有旧 OJ 痕迹。

补齐方案：

- `problem-create.html` 改为“学习任务编辑器”。
- 隐藏或重构 `leaderboard.html`，改成“班级学习概览”或暂时不入口。
- 清理旧文案、旧缓存键、旧 CSS 变量中明显的 OJ 语义。

优先级：P1。

### 差距 G：过程评价模型还比较朴素

当前已能统计反复同类错误，但还没形成完整学习画像。

补齐方案：

- 学生画像增加：
  - 最近 10 次错因分布
  - 同题连续卡点
  - 编译错误趋势
  - WA 到 TLE 的阶段性进步
  - 首次通过前的尝试次数
- 教师看板增加：
  - 全班高频错因
  - 题目卡点热力
  - 需要干预学生
  - 只差一步学生
  - 疑似复制答案但缺少过程的学生

优先级：P1。

## 4. 近期精进顺序

### 第一优先级：演示可信

- 完成 C++ 本机运行环境。
- 教师端补创建作业和邀请码管理。
- 学生端补“我的学习轨迹”。
- 彻底清理入口中的旧 OJ 语言。

### 第二优先级：教学含金量

- 标签库标准化。
- 能力点映射。
- 多次提交趋势分析。
- 教师可行动建议模板。

### 第三优先级：学校试点可运营

- 学生名单导入。
- 题目和测试点批量导入。
- 班级/作业数据导出。
- 试点数据备份与恢复。

### 第四优先级：长期产品壁垒

- 多 Agent 教学架构。
- 竞赛题智能诊断。
- 个性化练习推荐。
- 容器沙箱与资源隔离。

## 5. 1.0 完美版验收标准

- 学生能在 60 秒内从邀请码进入任务并完成一次提交。
- AI 反馈不出现完整答案代码。
- 教师能在一个页面看到班级高频错因和需要关注学生。
- 至少支持 Python 完整运行，C++ 环境明确可用或明确提示未配置。
- HTML 不被旧缓存卡住。
- 核心页面无前端控制台错误。
- 试点演示数据中文显示正常。
- 所有教学诊断数据都来自标准化字段，不依赖自由文本猜测。
