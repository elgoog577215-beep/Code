# Spec: Education Agent Objective Loop

## 先校正几个认识

1. 这不是在训练大模型参数。

   真正训练大模型通常会有损失函数、梯度、反向传播和权重更新。我们现在做的是 agent engineering：优化提示词、规则信号、标准库、评测集、输出结构、置信度和人工校正闭环。它也像训练，但更新对象不是模型权重，而是系统设计。

2. 目标函数不是单一分数。

   教育 agent 不能只追求“猜中错因”。如果它乱贴标签，即使主错因对了，也会误导学生和污染学习画像。所以目标函数必须同时看召回、误报、安全、教学可操作性和长期画像质量。

3. 评测集不是越贴合当前规则越好。

   如果只用我们自己生成的正例，很容易过拟合。必须同时保留正例、负例、真实提交 replay、教师校正样本，形成互相制衡。

4. 目标本身可以迭代，但不能随便漂移。

   目标可以随着阶段升级，但每次升级都要写清楚为什么、改了什么权重、旧能力是否退化。否则就会变成“看起来一直在优化，其实每轮都换题目”。

## 更好的执行提示词

```text
你是一个教育编程 agent 优化器。你的任务不是让测试数字好看，而是让学生收到更准确、更克制、更可执行的代码诊断提示。

每一轮按以下闭环执行：

1. 定义当前目标函数：
   - 正例召回：能否识别真实错因、细粒度错因、教学动作、证据锚点。
   - 负例精度：相似但无该错误的代码不能被乱贴标签。
   - 安全性：不能泄露完整答案、参考代码或隐藏测试点。
   - 教学性：必须给学生下一步可执行动作，而不是只给结论。
   - 画像质量：只把强证据写入长期错因标签，弱证据保留为观察。

2. 跑当前评测，找最大偏差：
   - 如果正例漏判，补强证据识别。
   - 如果负例误报，降级弱信号或增加上下文约束。
   - 如果提示太直接，收紧安全门。
   - 如果提示不可执行，改教学动作映射和追问方式。

3. 做最小必要改动：
   - 优先修改标准库、规则链、提示词模板或评测集。
   - 不为了单个样例写死逻辑。
   - 每次都保留正例和负例双向回归。

4. 复跑测试并记录结果：
   - 记录目标分数、召回、误报、安全、结构完整性。
   - 写明本轮变强在哪里、仍然可能错在哪里。

5. 决定下一轮目标：
   - 只有当当前目标稳定后，才升级到更真实的数据，比如真实学生提交 replay。
```

## 当前目标函数

第一版目标函数不追求复杂数学形式，先用可解释的加权评分：

```text
objective =
  0.45 * positiveRecall
+ 0.35 * negativePrecision
+ 0.10 * safety
+ 0.10 * structureCompleteness
```

其中：

- `positiveRecall`：正例错因、细粒度错因、教学动作、证据锚点的平均命中率。
- `negativePrecision`：负例 forbidden issue/fine tags 不出现的比例。
- `safety`：学生可见文本不出现泄题短语的比例。
- `structureCompleteness`：每次诊断都有完整 `studentHintPlan` 的比例。

## 当前验收线

- `objective >= 0.95`
- `positiveRecall >= 0.90`
- `negativePrecision >= 0.98`
- `safety >= 1.00`
- `structureCompleteness >= 1.00`

这不是最终目标，只是当前阶段的自动化门槛。下一阶段应加入真实学生提交 replay 和教师校正样本。
