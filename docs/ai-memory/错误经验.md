# 错误经验

## 2026-07-12 生产启动不得隐式触发镜像构建

- 背景：生产服务器长期使用 `docker compose up --build -d` 同时完成启动和构建，调用方便，但把 Node、Maven、GCC 的资源压力直接带到只有约 3.5GB 内存和 30GB 系统盘的 ECS。
- 触发：一次 Java 源码变更使 Maven 构建层失效，服务器重新下载依赖并编译测试；叠加约 10GB Build Cache 和 84% 磁盘使用率后，系统进入 memory pressure，SSH、HTTP 和正常关机都失去响应。
- 过程：强制恢复服务器后先备份并验证 PostgreSQL、`.env`、Compose 和服务器本地安全配置；只执行 `docker builder prune --all --force`，回收 10.03GB 构建缓存，未删除镜像、容器或 Volume。随后通过 OpenSpec `separate-production-build-release` 把启动改为 `--no-build`，构建拆成必须显式确认的独立脚本，并增加静态回归测试。
- 复发证据：2026-07-16 完成外部镜像部署并同步主分支后，最终探针正好捕获到一次并发 Compose 全量重建：已验收的应用容器被终止，`latest` 被服务器构建覆盖，接口在构建窗口内短暂不可达。git hook、systemd、cron 和存活进程均未发现固定触发器，说明仅加固项目脚本仍不能防住直接执行 Compose 构建或并发运维命令；最终通过 Docker events、镜像 ID、Flyway 日志和 readiness 重新验收才确认恢复。
- 结论：生产发布必须“外部构建、服务器只启动”。首次安装和受控构建可以显式执行镜像构建，但生产启动、配置重载和应用容器替换不得隐式编译，也不得用 `docker system prune`、`volume prune` 或 `down -v` 清理现场。
- 复用：部署前检查磁盘、内存、运行容器、数据卷和备份；新镜像保留旧镜像回滚标签；服务器只运行安全启动脚本。清理空间时优先审计 `docker system df`，只删除确认未被运行时使用的 Build Cache，数据库 Volume 永远单独保护。
- 位置：`online-judge/scripts/start-school.sh`、`start-school.ps1`、`build-school-images.sh`、`build-school-images.ps1`、`online-judge/src/test/java/com/onlinejudge/deployment/SchoolDeploymentScriptSafetyTest.java`、`openspec/specs/production-release-safety/spec.md`。

## 2026-07-10 学生 AI 反馈有代码证据但无知识路径先查旧 READY 契约

- 背景：服务器测评页里“修正建议/提升建议”已经能显示代码证据，但知识路径消失，用户质疑是不是前端功能跟着快链路一起被删掉。
- 触发：截图中每条建议都有 `代码证据 第 14 行 ...`，但没有知识路径标签；说明前端展示证据链正常，缺口更可能在后端 `knowledgePath` 字段。
- 过程：排查发现旧的 `READY` 学生反馈会被直接复用，不会因为缺 `knowledgePath`、`latencyMs` 等新版契约字段而重新生成；另外当模型建议没有返回标准库 `skillUnitId / mistakePointId / improvementPointId`，或生产库无法解析这些 ID 时，后端只能保留空路径。
- 结论：这种现象不是单纯换模型能解决。学生反馈缓存必须识别“字段契约不完整”的旧 `READY` 并触发重生成；新生成时如果标准库 ID 缺失，要用知识树名称/别名从标题、正文和摘要中反查，最后至少给出“未归类”兜底，不能返回空 `knowledgePath`。
- 复用：以后看到“证据有、知识路径没”的反馈，先查 `student_ai_feedbacks.feedback_json` 中对应 item 的 `knowledgePath` 是否为空，再查该记录是否旧 `READY` 被缓存、模型 advice 是否缺标准库 ID、知识树节点别名是否覆盖该概念。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/application/StudentAiFeedbackService.java`、`online-judge/src/main/java/com/onlinejudge/submission/api/SubmissionController.java`、`online-judge/src/test/java/com/onlinejudge/submission/application/StudentAiFeedbackEventTest.java`。

## 2026-07-09 学生 AI 反馈失败先查输出截断、限流和事务边界

- 背景：删除快链路、默认走完整 AI 诊断链路后，服务器上学生提交显示 AI 无法生成建议，本地复现同样出现 `FAILED / AI_UNAVAILABLE`。
- 触发：本地提交“两数求和”错误代码后，`/api/submissions/{id}/student-ai-feedback` 长时间 `GENERATING`，随后失败；日志先出现 advice JSON `OUTPUT_TRUNCATED`，再出现 `Transaction silently rolled back because it has been marked as rollback-only`。
- 过程：先确认本地 8081 旧进程不是最新代码，再用当前 `.env` 启动后端；`/api/system/ai-smoke` 通过，说明 key 和 provider 基本可用。继续提交真实失败代码后发现 advice 阶段输出预算不足会截断，连续完整链路调用还可能触发 ModelScope `429 limit_burst_rate`，而这些失败发生在数据库事务内或被事务型成长候选写入污染，最终把根因掩盖成事务回滚。
- 结论：学生 AI 反馈“无法生成”不能只看前端状态；要按顺序查 `ai-smoke`、真实提交反馈、后端日志里的 `OUTPUT_TRUNCATED / 429 / UnexpectedRollbackException`。外部模型网络调用不应包在长数据库事务里；标准库成长候选是附属沉淀，必须独立事务，失败不能拖垮诊断主链路。
- 复用：遇到学生反馈无建议时，先用一条匿名失败提交复现，再查 `/analysis` 是否仍 `PENDING`。若出现事务回滚，优先检查 `SubmissionAnalysisService.generateAndStoreAnalysis*` 是否仍开启事务、`AiStandardLibraryGrowthAgentService.proposeFromDiagnosisOutput` 是否独立事务，以及 advice 阶段 token 预算/结构化重试是否打开。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/application/SubmissionAnalysisService.java`、`online-judge/src/main/java/com/onlinejudge/submission/application/AiReportService.java`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibraryGrowthAgentService.java`、`online-judge/src/test/java/com/onlinejudge/submission/application/AiReportServiceAdviceGenerationRuntimeTest.java`。

## 2026-07-08 长代码高级题先验证导航链路是否完成

- 背景：把真实样本 AI 测试前 5 题换成 200 行以上的高级长代码后，重跑网站 AI 链路，想验证基础建议和提高建议是否能返回多条。
- 触发：5 道题都没有生成建议，网站建议数为 0；第一轮出现 `source:line 148` 非法证据引用、`CONTINUE navigation requires selectedBranches` 和 timeout，修复 `source:line` 证据别名后，第二轮仍主要卡在标准库导航空分支、导航超时和 free diagnosis 输出截断。
- 过程：先确认样本不是 `helper_N` 人为填充，前 5 题分别为 226、215、239、243、235 行；再跑真实模型链路，报告显示失败发生在 `FREE_DIAGNOSIS` 或 `STANDARD_LIBRARY_NAVIGATION / AI_NAVIGATION`，没有进入 advice generation。
- 结论：长代码高级题下，先判断 free diagnosis 和标准库导航是否完成；如果前置阶段失败，不能用“建议数 0”推断多条 advice 不支持。多条建议能力要用已进入 advice generation 的样本或本地结构化回归测试验证。
- 复用：跑 `DiagnosisReportV2RealSamplesSimulationTest` 时先看 `failureStage/failureReason`；若是 `CONTINUE navigation requires selectedBranches`、`TIMEOUT`、`OUTPUT_TRUNCATED`，优先处理导航/输出预算/模型能力，而不是修改 `basicLayerAdvice` 或 `improvementLayerAdvice` 映射。
- 位置：`online-judge/src/test/resources/diagnosis-eval-fixtures/diagnosis-report-v2-real-samples.json`、`online-judge/src/test/resources/diagnosis-eval-fixtures/long-code-samples/`、`online-judge/target/ai-simulation-reports/real-samples-website-vs-codex-20260708-191739.md`。

## 2026-07-08 多条 AI 建议不要只看报告首项

- 背景：排查 `online-judge` 真实样本 AI 测试时，用户质疑基础建议和提高建议为什么看起来每层都只有一条，明明样本里可能有多个独立错误。
- 触发：检查 AI 链路发现 prompt、DTO、mapper 和学生端 view 都支持 `basicLayerAdvice` / `improvementLayerAdvice` 多条；但真实样本对照报告在逐题明细中用 `findFirst()` 只展示第一条基础建议和第一条提高建议。
- 过程：先确认 prompt 已要求“每个独立问题返回一个建议项”，再跑 `AdviceGenerationFeedbackMapperTest` 和 `StudentFeedbackViewAssemblerTest`，验证 2 条基础建议 + 2 条提高建议能在结构化反馈和学生端 view 中保留；最后把报告展示改成按序拼出全部建议。
- 结论：判断“模型是否只能返回一条建议”时，不要只看生成的对照报告明细；要同时看 `basicAdviceCount / improvementAdviceCount`、原始结构化字段和学生端 view。报告展示层的单条摘要可能制造假象。
- 复用：以后跑 `DiagnosisReportV2RealSamplesSimulationTest` 或其他 AI eval 报告时，若建议数量和明细不一致，先查报告生成代码是否有 `findFirst()`、`get(0)` 或 `limit(1)`，再判断模型或 prompt 是否真的收窄。
- 位置：`online-judge/src/test/java/com/onlinejudge/submission/application/DiagnosisReportV2RealSamplesSimulationTest.java`、`online-judge/src/main/java/com/onlinejudge/submission/application/AdviceGenerationFeedbackMapper.java`、`online-judge/src/main/java/com/onlinejudge/submission/application/StudentFeedbackViewAssembler.java`。

## 2026-07-08 真实样本 AI 验证先固定 key 与模型来源

- 背景：接力 `online-judge` 的 AI 标准库导航修复时，需要跑 `DiagnosisReportV2RealSamplesSimulationTest` 的 5 题真实模型链路。
- 触发：直接运行测试时，当前 shell 的 `AI_EVAL_API_KEY` 和 `AI_MODEL` 覆盖了 `.env` 中的 `OJ_MODELSCOPE_API_KEY / OJ_AI_MODEL`，导致测试实际请求 `deepseek-ai/DeepSeek-V4-Pro` 并返回 401，而不是验证目标 Qwen 模型。
- 过程：显式从 `.env` 读取 `OJ_MODELSCOPE_API_KEY` 并设置 `AI_EVAL_MODEL=Qwen/Qwen3-235B-A22B-Instruct-2507` 后，认证错误消失，返回变成 ModelScope 对该模型的今日配额 429。
- 结论：真实样本验证前必须明确本轮使用的 key 与模型来源，只报告 key 是否存在/长度，不输出密钥；不要让 shell 里的 `AI_EVAL_*` 或 `AI_MODEL` 静默覆盖 `.env`。
- 复用：跑 `DiagnosisReportV2RealSamplesSimulationTest` 时优先用 `AI_EVAL_API_KEY="$OJ_MODELSCOPE_API_KEY"` 和显式 `AI_EVAL_MODEL=...`；若返回 401，先查环境变量优先级；若返回 429，按 provider 配额阻断记录，不继续误判代码链路。
- 位置：`online-judge/src/test/java/com/onlinejudge/submission/application/DiagnosisReportV2RealSamplesSimulationTest.java`、`online-judge/target/ai-simulation-reports/real-samples-website-vs-codex-20260708-182040.md`。

## 2026-07-08 标准库旧 BASIC_CAUSE 兜底条目漏清理

- 背景：服务器和本地库里仍残留大量 `KB_...`、`知识点错因/...`、`standard-library-db-v2-full-coverage` 的旧条目，文案类似“某知识点掌握偏差”，在 `if 条件` 等知识点下会显示成低质量、模板化答案。
- 触发：用户指出数据库里还有很多枚举式、低质量答案，之前处理过但没有处理干净。
- 过程：排查发现旧清理逻辑只识别 `SK_` 能力点和 `MP_` 易错点生成兜底，没有把旧全覆盖生成器留下的 `BASIC_CAUSE / KB_*` 纳入 `isGeneratedFallbackCode(...)`，导致启动 seeder 不会禁用这些旧行。
- 结论：清理标准库历史低质数据时，要同时覆盖三类旧生成码：`SK_*`、`MP_*` 和 `BASIC_CAUSE / KB_*`。只检查正式 seed 是否高质量不够，还要验证现有数据库旧行是否会被幂等禁用。
- 复用：遇到“教师端/AI 标准库仍出现模板化掌握偏差答案”时，先查 `ai_standard_library_items` 中 enabled 的 `BASIC_CAUSE`、`code like KB_%`、`category like 知识点错因/%` 或 `library_version = standard-library-db-v2-full-coverage`，再跑启动 seeder 或 `scripts/disable-legacy-full-coverage-basic-causes.sh`。
- 位置：`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibrarySeedCatalog.java`、`online-judge/src/test/java/com/onlinejudge/learning/standardlibrary/AiStandardLibrarySeederTest.java`、`online-judge/scripts/disable-legacy-full-coverage-basic-causes.sh`。

## AI runtime 测试夹具必须保持语义一致

- 背景：在“学习动作证据进入下一次诊断”变更中，扩展 targeted tests 时发现 `AiReportServiceExternalRuntimeTest` 有两个用例失败。
- 触发：测试期望外部模型输出 `LOOP_BOUNDARY / OFF_BY_ONE`，但实际结果被校验链路保留为 `TIME_COMPLEXITY / OVER_SIMULATION`。
- 过程：排查后发现测试本身传入的是 `scaleFallback()` 和 `scaleRuleSignals()`，也就是复杂度上下文，却断言循环边界结果。外部模型输出、ruleSignals、fallback 和 teacher expectation 不在同一个语义场，导致失败看起来像生产逻辑回归。
- 结论：AI eval 或 runtime 单测中，模型输出、规则信号、fallback baseline、老师期望必须指向同一类错因；否则质量门禁会把正确的上下文约束误判成模型能力问题。
- 复用：以后遇到“模型输出与断言不一致”时，先检查 fixture 四件套是否一致：`model output`、`ruleSignals`、`fallback`、`teacherExpectation/assertion`。
- 位置：`online-judge/src/test/java/com/onlinejudge/submission/application/AiReportServiceExternalRuntimeTest.java`
