# 错误经验

## AI runtime 测试夹具必须保持语义一致

- 背景：在“学习动作证据进入下一次诊断”变更中，扩展 targeted tests 时发现 `AiReportServiceExternalRuntimeTest` 有两个用例失败。
- 触发：测试期望外部模型输出 `LOOP_BOUNDARY / OFF_BY_ONE`，但实际结果被校验链路保留为 `TIME_COMPLEXITY / OVER_SIMULATION`。
- 过程：排查后发现测试本身传入的是 `scaleFallback()` 和 `scaleRuleSignals()`，也就是复杂度上下文，却断言循环边界结果。外部模型输出、ruleSignals、fallback 和 teacher expectation 不在同一个语义场，导致失败看起来像生产逻辑回归。
- 结论：AI eval 或 runtime 单测中，模型输出、规则信号、fallback baseline、老师期望必须指向同一类错因；否则质量门禁会把正确的上下文约束误判成模型能力问题。
- 复用：以后遇到“模型输出与断言不一致”时，先检查 fixture 四件套是否一致：`model output`、`ruleSignals`、`fallback`、`teacherExpectation/assertion`。
- 位置：`online-judge/src/test/java/com/onlinejudge/submission/application/AiReportServiceExternalRuntimeTest.java`

## Windows PowerShell 读取 live eval 报告时必须显式使用 UTF-8

- 背景：在外部模型 live eval 后，用 PowerShell `Get-Content ... | ConvertFrom-Json` 解析 `target/ai-eval-reports/assistant-live-eval-*.json` 失败，看起来像报告 JSON 字符串未闭合。
- 触发：Node `JSON.parse` 能解析同一份报告，但 PowerShell 默认读取后中文字段变成乱码，`ConvertFrom-Json` 在破损字符串处报错。
- 过程：对比 `Get-Content -Raw`、`Get-Content -Raw -Encoding UTF8` 和 Node `fs.readFileSync(..., 'utf8')`，确认报告文件本身是合法 UTF-8 JSON；失败来自 Windows PowerShell 默认编码误读，而不是 Jackson 写坏报告。
- 结论：live eval 报告应继续使用标准 UTF-8 JSON。Windows PowerShell 解析报告时必须使用 `Get-Content -Raw -Encoding UTF8 | ConvertFrom-Json`，否则可能把编码问题误判成 AI 评测或报告生成问题。
- 复用：以后分析 `assistant-live-eval` 报告时，先用 UTF-8 方式读取；如果解析失败，再判断是否是报告生成逻辑问题。相关测试应覆盖 UTF-8 JSON 写入和回读。
- 位置：`online-judge/src/test/java/com/onlinejudge/eval/AssistantLiveEvalTest.liveEvalReportWritesUtf8JsonThatCanBeReadBack`。
