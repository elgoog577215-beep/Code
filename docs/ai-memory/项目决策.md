# 项目决策

## 2026-07-15 PostgreSQL Schema 由 Flyway 版本化管理

- 背景：正式内容完成数据库主库迁移后，学校 PostgreSQL 仍由 Hibernate `ddl-auto: update` 隐式改表，缺少结构版本历史、迁移校验值、漂移阻断和可恢复发布证据。
- 决策：学校与生产 PostgreSQL 以 `src/main/resources/db/migration/` 为唯一 Schema 发布源；Flyway 负责执行版本化 SQL，`school` profile 的 Hibernate 固定使用 `ddl-auto=validate`。全新空库执行 V1；现有非空库默认拒绝隐式接纳，只能在结构预检、custom-format 备份校验、计数快照和显式确认后登记一次性 V1 基线。
- 边界：H2 暂时保留为本地快速环境，但不代表生产 Schema；已发布迁移不得修改。正式内容仍以数据库为主库，不因引入 Flyway 恢复 seed。DDL 失败时不自动伪造回滚，优先保留旧镜像、数据库 Volume 和已验证备份，采用向前修复或受控恢复。
- 生产落地：2026-07-16 在正式服务器生成并校验 `backups/onlinejudge-20260716-135612.dump`，完成不挂载正式 Volume 的隔离恢复演练；随后将既有非空库登记为 Flyway V1 基线。基线前、基线后和新应用部署后的 10 张关键表计数完全一致，Schema readiness、内容 readiness、应用 readiness 数据库项和题库接口均通过；受控切换阶段只替换 `app`，PostgreSQL 容器和 `online-judge_postgres-data` Volume 未变，旧应用镜像保留为 `wenzhong-oj-app:rollback-pre-flyway-20260716`。
- 现场补充：主分支最终同步后，Docker 事件捕获到一次 Compose 全量重建，短暂覆盖了 `latest` 并重建 `app` 和 C++ runner。GitHub Actions 运行记录随后确认，根因是仓库仍在每次 `main` push 后 SSH 执行服务器 `deploy-online-judge`，而该全局脚本绕过项目门禁直接运行 `docker compose up -d --build`；连续 push 还会取消前一构建并启动下一构建。最终应用仍从同一提交构建，Flyway V1 校验、readiness 和题库接口复验通过，数据库容器和 Volume 始终未变；外部验证镜像继续保留为 `wenzhong-oj-app:flyway-v1-991c7ca8`。以后服务器同步代码后必须同时检查 GitHub 部署任务和 Docker 事件，不能把 `git pull` 成功视为发布收束。
- 复用：以后新增字段、表、索引或约束时，必须新增 Flyway 版本并运行真实 PostgreSQL 空库、幂等和 Schema 漂移测试；不得恢复 `ddl-auto=update`，不得只用 H2 测试推断生产兼容性。迁移前执行备份与隔离恢复演练，迁移后同时检查 Flyway history、Schema readiness、内容 readiness 和应用 readiness。
- 位置：`online-judge/src/main/resources/db/migration/`、`online-judge/scripts/baseline-postgres-flyway.sh`、`online-judge/scripts/check-database-schema-readiness.sh`、`online-judge/scripts/backup-postgres.sh`、`online-judge/scripts/rehearse-postgres-restore.sh`、`online-judge/docs/database-migration-guide.md`、`online-judge/openspec/changes/archive/2026-07-16-engineer-database-migrations/`。

## 2026-07-15 历史 seed 只允许通过一次性迁移入口运行

- 背景：正式内容数据库优先的第一阶段已经关闭默认启动播种，但 `app.content-seed.enabled=true` 仍能在普通应用启动时重新启用全部历史 seed，且缺少备份、幂等复跑和审计摘要，尚未真正形成数据库主权迁移闭环。
- 决策：删除普通 `app.content-seed` 开关，历史题库、知识树和标准库播种器只受 `app.content-migration.enabled=true` 控制；正式迁移必须使用 `scripts/migrate-seeded-content-once.sh`，显式确认目标 PostgreSQL，先备份，连续执行两次并比较关键表计数，最后运行内容 readiness 查询。正常 dev/school 启动均无法激活历史内容写入。
- 边界：2026-07-15 已在正式服务器备份 PostgreSQL，并连续执行两次迁移验证关键表数量不变；历史 seed 源码、资源 seed、catalog 与 V6-V16 内容类已从 `src/main` 移出，只在 `src/test` 作为隔离夹具保留。生产包不再包含任何可重新播种正式内容的入口。
- 复用：以后数据库缺内容时不得恢复普通启动 seed。先检查正式库迁移审计、readiness 和治理写入链路；只有一次性迁移窗口可以显式激活历史导入，且必须保留备份和两次计数一致的证据。
- 位置：`online-judge/src/test/java/com/onlinejudge/problem/application/`、`online-judge/src/test/java/com/onlinejudge/learning/knowledge/application/`、`online-judge/src/test/java/com/onlinejudge/learning/standardlibrary/application/`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/ArchivedFallbackCodePolicy.java`、`online-judge/openspec/changes/archive/2026-07-14-migrate-seeded-content-to-database/`。

## 2026-07-11 学生问题成长采用原始事实与可重算生命周期双层模型

- 背景：逐提交诊断事实已经能保存多条修正建议和提升点，但 `factKey` 包含分析 ID 与序号，只能保证单次投影幂等，不能表达同一问题在多次提交中的持续、消失、恢复和复发；教师统计也只有错误次数与涉及人数，无法区分同代码重复点击和真实修改后的持续困难。
- 触发：用户明确要求每次提交的问题完整展示并真实落库，同一学生多次解决不了的问题应成为个人难点，班级统计同时提供“只要错过就算一人”的正常口径与“同一学生多次出现也累计”的加权口径。
- 过程：通过 OpenSpec `build-student-issue-lifecycle-and-class-difficulty-analytics` 为诊断事实增加带版本的标准问题键，新增独立 `SubmissionIssueTransition` 投影 NEW、PERSISTED、NOT_OBSERVED、RECOVERED、RECURRED 和 UNCOMPARABLE；相同代码且评测与问题集合不变时只增加原始次数，不增加有效次数；学生反馈查询时动态补成长状态，教师聚合并列返回去重人数、原始次数、有效次数、未解决、复发和恢复率。
- 决策：`SubmissionDiagnosisFact` 是不可覆盖的逐提交原始事实，问题身份优先使用正式易错点/提升点 ID，其次能力点与文本指纹；生命周期、个人持续难点、复发易错、跨题薄弱和班级重难点都是可幂等重算的派生投影，不得反写篡改历史事实。新问题只标记新出现或单次观察，不能直接判为长期缺陷。
- 复用：以后调整阈值、分类或前端表达时，先保留原始事实与证据提交链，只重算生命周期投影；班级难度不能只看总错误数或涉及人数，必须同时看有效重复、当前未解决和恢复情况；完全相同代码的重复提交保留审计记录但不放大加权难度。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/domain/SubmissionDiagnosisFact.java`、`SubmissionIssueTransition.java`、`online-judge/src/main/java/com/onlinejudge/submission/application/SubmissionIssueLifecycleService.java`、`StudentIssueLifecycleReadService.java`、`online-judge/src/main/java/com/onlinejudge/classroom/application/SubmissionEvidenceAnalyticsService.java`、`openspec/changes/build-student-issue-lifecycle-and-class-difficulty-analytics/`。

## 2026-07-10 ModelScope 调用采用模型池故障转移

- 背景：服务器 AI API 一直显示限流，用户怀疑是不是魔搭某些模型额度用完。直接用当前 `.env` key 做 ModelScope OpenAI-compatible `OK` smoke 后，Qwen Coder、Qwen3-235B Instruct、Qwen3-235B Thinking、Qwen3-Next、Qwen3.5 均可用；`MiniMax/MiniMax-M2.7` 返回模型不支持；`DeepSeek-V4-Pro` 返回 `INSUFFICIENT_QUOTA / token-limit`。
- 决策：后端 AI 调用默认引入模型池，按 `OJ_AI_MODEL` / `AI_MODEL` 作为主模型，再按 `OJ_AI_MODEL_POOL` / `AI_MODEL_POOL` 中的候选顺序尝试。遇到额度不足、限流、模型不支持、超时或 provider API 错误时，当前请求自动切换到下一个模型。2026-07-10 进一步实测 `Qwen/Qwen3.5-122B-A10B`、`Qwen/Qwen3.5-35B-A3B`、`Qwen/Qwen3.5-27B` 可用；完整 AI 链路中 `122B-A10B` 多次出现 timeout 或 `choices:null`，因此默认模型池收缩为 `Qwen/Qwen3.5-35B-A3B,Qwen/Qwen3.5-27B,Qwen/Qwen3.5-122B-A10B`，以 35B 为主、27B 次级、122B 后置兜底。
- 边界：认证失败通常是同一个 key 的全局问题，不靠换模型解决；默认模型池只放本轮流式 smoke 能返回内容的 Qwen 系列，暂不把 `DeepSeek-V4-Pro` 放入默认池。模型支持和额度会随时间变化，遇到新 429 仍需用最小 `OK` 请求重新核验。
- 复用：以后排查 ModelScope 429 时，先区分 `RATE_LIMITED`、`INSUFFICIENT_QUOTA`、`MODEL_UNSUPPORTED` 和 `AUTHENTICATION_FAILED`；如果只是模型级额度或短时限流，应让后端模型池承接，而不是恢复本地预算保护或误判为账号总额度用完。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/application/ExternalModelPool.java`、`online-judge/src/main/java/com/onlinejudge/submission/application/AiReportService.java`、`online-judge/src/main/java/com/onlinejudge/classroom/application/CoachAgentService.java`、`online-judge/src/main/resources/application.yml`。

## 2026-07-09 AI 预算保护删除

- 背景：学生端出现“AI 调用保护已开启 / 系统检测到连续失败，暂时停止继续消耗 API 调用”，实际代码中一次额度不足、限流、认证失败或模型不可用就会打开短期预算保护，后续请求在冷却期内被本地短路，容易让用户误以为 AI 功能本身坏了。
- 决策：删除 `ExternalModelBudgetGuard`、`BUDGET_GUARD_OPEN`、预算保护配置和调用前短路检查。外部模型失败后可以展示真实原因，但系统不能因为上一次失败而制造新的本地阻断状态。
- 边界：保留真实失败归因、AI smoke、超时、结构校验、输出截断、安全拒绝和 provider 错误展示；这些是解释真实问题，不是本地预算保护。高成本批量评测如果需要节流，应在评测任务层显式控制样本规模，而不是在学生/Coach 体验链路里短路 AI。
- 复用：以后遇到学生端 AI 反复显示“未完成/暂不可用”时，先查真实失败原因 `INSUFFICIENT_QUOTA / RATE_LIMITED / AUTHENTICATION_FAILED / MODEL_UNSUPPORTED / OUTPUT_TRUNCATED / INVALID_JSON / TIMEOUT / SAFETY_REJECTED`，不要恢复本地预算保护。

## 2026-07-08 运行时 seed 默认退出启动主链路

- 背景：用户进一步要求把细颗粒题库、知识树和标准库内容切回正式数据库，并把 seed 从项目主链路移除。直接删除全部 seed 类会在正式库迁移完成前造成题库/标准库断供，因此采用“先切断运行时写入，再迁移和删除内容型代码”的顺序。
- 决策：`DataInitializer`、`PublicProblemSeeder`、`InformaticsKnowledgeSeeder`、`AiStandardLibrarySeeder`、`AiStandardLibraryNormalizedSeeder` 默认不再随应用启动写正式内容，统一受 `app.content-seed.enabled=true` 的 legacy 开关控制。默认配置保持关闭；旧测试或一次性迁移过渡需要时才显式打开。
- 边界：关闭运行时 seed 不等于正式库已经完成迁移。部署前必须先备份正式 Postgres，执行内容迁移/验证，并用 readiness 的 `database-content` 检查确认题库、知识树、能力点、易错点和提升点均存在。
- 复用：以后遇到“数据库内容缺失”时，不要重新打开启动播种器当修复；应检查正式库迁移状态、管理 API/治理台写入链路和验证查询。删除内容型 seed 类前，先确认旁路依赖已改成数据库读取或纯规则判断。

## 2026-07-08 正式题库和标准库内容不再用 seed 交付

- 背景：用户明确否定“通过种子文件补正式数据库内容”的方式，认为题库和标准库内容应直接进入正式数据库，而不是继续在代码里堆 Java seed、版本化 seed 批次或启动播种逻辑。
- 决策：后续正式题库、知识树、标准库能力点、易错点和提升点的新增与修订，默认以正式数据库为内容主库。GitHub 只保存数据库结构、API、管理界面、校验工具、审计脚本、备份/恢复脚本和必要迁移脚本；不得再把新增正式内容写成新的 `SeedCatalog`、`VxExpansionSeeds` 或资源 seed 文件作为主要交付。
- 边界：既有 seed 代码暂作为历史兼容、测试夹具或迁移来源保留；新增内容不得继续扩张 seed。确需批量导入时，应使用可审计 SQL / 管理 API / 后台治理流程写入正式规范化表，并在执行前备份，执行后用数据库查询和 AI 导航读取验证。
- 复用：以后用户要求扩题库、扩标准库、替换低质量答案或修正式内容时，先确认目标数据库、备份方式、写入表和验证查询，再改正式数据库；不要以“新增 seed + 重启播种”作为完成标准。

## 2026-07-06 学生 AI 反馈后处理必须保留有效逐条建议

- 背景：学生端 AI 诊断结果出现“修正建议证据重复”和“建议看起来永远只有一条”的问题。深扫发现 prompt 与 normalizer 已允许 `repairItems` / `improvementItems` 返回 0 到多条，但运行时越界兜底和提高层漂移校正仍用 `List.of(...)` 覆盖数组，前端摘要区域也会重复渲染第一条 item 的代码证据。
- 触发：用户要求从 AI 链路输入、输出和内部控制层深度扫描并解决“永远只有一条建议”的根因。
- 过程：通过 OpenSpec `preserve-student-feedback-items` 增加学生可见反馈数量保真要求；后端将越界兜底改为“可信主项合并 + 局部过滤漂移项”，提高层只替换跑偏的代码整理建议；前端将摘要与逐条建议分离展示，并区分代码证据、非代码证据和暂无代码证据。
- 结论：以后修改学生 AI 反馈链路时，后处理不得整体覆盖模型返回的多条有效结构化建议；只能删除、替换或补充不可信、重复、泄题或跑偏的单项。前端不得把同一条建议同时作为摘要证据和逐条 item 重复展示。
- 复用：涉及 `studentReport`、`repairItems`、`improvementItems`、runtime grounding、drift alignment、证据展示或前端反馈卡片时，必须检查是否存在新的 `List.of(...)` 覆盖、`get(0)` 主项误用、摘要与 item 双重渲染、或无代码证据空态不可达。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/application/AiReportService.java`、`online-judge/frontend/src/features/problem/ProblemPage.tsx`、`online-judge/src/test/java/com/onlinejudge/submission/application/StudentAiFeedbackModelTest.java`、`online-judge/openspec/specs/student-visible-ai-feedback-quality/spec.md`。

- 背景：教师端页面曾把作业、题目、学生诊断、班级学情和管理入口分散在顶部导航、局部卡片和管理页状态块里，老师难以按真实教学路径判断下一步。

## 2026-07-06 自动兜底标准库终态为静态冷备份

- 背景：自动兜底层最初用于没有外部模型时保证知识树覆盖，后来经过退役、降级、A/B/C 榨取和 V10/V11/V12 手写吸收后，继续保留运行时兜底库已经会增加误接入风险。
- 触发：用户明确要求删除兜底型库，但删除前先本地备份；备份可以随 Git/云端同步，但只能作为备份目录里的东西，永远不参与前后端。
- 过程：通过 OpenSpec `remove-generated-fallback-archive-runtime` 导出删除前快照到 `online-judge/backups/standard-library/generated-fallback-archive-2026-07-06/`，删除 `archivedGeneratedFallbackSeeds()`、自动生成器和 `AiStandardLibraryFallbackArchiveValueCatalog`，只保留 `isGeneratedFallbackCode(...)` 用于禁用历史数据库里的旧兜底记录。
- 结论：兜底库不再是运行时存档素材池，也不再由后端代码枚举。正式库只保留已经人工精修吸收的 V10/V11/V12/V13 条目；未吸收兜底只存在于静态备份 JSON，不能被 Seeder、候选包、质量报告、AI 上下文、前端或后端读取。
- 复用：以后如果想重新利用备份素材，必须先人工筛选、重写、测试，再进入正式手写标准库文件；不要直接读取 `backups/`，不要恢复自动 full coverage 生成器。
- 位置：`online-judge/backups/standard-library/generated-fallback-archive-2026-07-06/`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibrarySeedCatalog.java`、`online-judge/openspec/specs/standard-library-normalized-schema/spec.md`。

## 2026-07-06 标准库扩张必须以主题包保证质量

## 2026-07-11 AI 诊断采用单核心、完整路径与可恢复并行编排

- 背景：学生提交后的完整诊断、历史版本和教师统计已经形成证据闭环，但外部模型链路仍以进程内串行调用为主，单个问题的逐层挂接、学生输出、教师输出和派生投影会累加等待时间，进程中断后也缺少可审计的阶段恢复依据。
- 触发：用户明确拒绝 AI 调用次数上限和普通/快速降级路径，要求所有提交都走同一条完整高质量路径，同时接受“不同问题并行挂接、学生与教师输出并行、保存后投影并行”，但核心诊断必须唯一。
- 过程：通过 OpenSpec `parallelize-complete-ai-diagnosis-workflow` 新增持久化诊断 Run 与阶段记录；核心诊断每个 Run 只执行一次，不同 issue 在专用受控线程池中并行、单个 issue 内仍按标准库层级顺序导航；学生反馈与教师洞察共享同一组规范 issues 和证据后并行生成；正式结果与 Run/修订版本关联，派生投影独立记录状态；学生端只展示学习化进度和恢复状态，不暴露 provider、token、配额等技术细节。
- 结论：完整路径不等于全链路串行。以后优化 AI 效率时，只能并行互不依赖且共享同一诊断事实的分支，不能复制核心诊断 Agent、不能用低质量规则结果伪装成功、不能通过限制调用次数牺牲问题完整性。每个可重试边界必须有稳定阶段键、输入指纹、尝试次数、状态、输出、耗时和失败原因，成功阶段恢复时直接复用。
- 复用：新增 AI 分支、教师消费或数据投影时，先判断它属于核心事实、问题级挂接、受众输出还是派生投影；核心事实保持唯一，问题级挂接按 issue 并行，受众输出在事实齐备后并行，投影在正式结果保存后并行。真实验收同时检查 Run/Stage 数据、并行时间证据、学生历史连续性、教师统计可消费性和失败恢复，不只看最终页面是否有文字。
