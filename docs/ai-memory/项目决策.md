# 项目决策

## 2026-07-08 正式题库和标准库内容不再用 seed 交付

- 背景：用户明确否定“通过种子文件补正式数据库内容”的方式，认为题库和标准库内容应直接进入正式数据库，而不是继续在代码里堆 Java seed、版本化 seed 批次或启动播种逻辑。
- 决策：后续正式题库、知识树、标准库能力点、易错点和提升点的新增与修订，默认以正式数据库为内容主库。GitHub 只保存数据库结构、API、管理界面、校验工具、审计脚本、备份/恢复脚本和必要迁移脚本；不得再把新增正式内容写成新的 `SeedCatalog`、`VxExpansionSeeds` 或资源 seed 文件作为主要交付。
- 边界：既有 seed 代码暂作为历史兼容、测试夹具或迁移来源保留；新增内容不得继续扩张 seed。确需批量导入时，应使用可审计 SQL / 管理 API / 后台治理流程写入正式规范化表，并在执行前备份，执行后用数据库查询和 AI 导航读取验证。
- 复用：以后用户要求扩题库、扩标准库、替换低质量答案或修正式内容时，先确认目标数据库、备份方式、写入表和验证查询，再改正式数据库；不要以“新增 seed + 重启播种”作为完成标准。

## 2026-07-06 学生 AI 反馈后处理必须保留有效逐条建议

- 背景：学生端 AI 诊断结果出现“修正建议证据重复”和“建议看起来永远只有一条”的问题。深扫发现 prompt 与 normalizer 已允许 `repairItems` / `improvementItems` 返回 0 到多条，但运行时越界兜底和提高层漂移校正仍用 `List.of(...)` 覆盖数组，前端摘要区域也会重复渲染第一条 item 的代码证据。
- 触发：用户要求从 AI 链路输入、输出和内部控制层深度扫描并解决“永远只有一条建议”的根因。
- 过程：通过 OpenSpec `preserve-student-feedback-items` 增加学生可见反馈数量保真要求；后端将越界兜底改为“可信主项合并 + 局部过滤漂移项”，提高层只替换跑偏的代码整理建议；前端将摘要与逐条建议分离展示，并区分代码证据、非代码证据和暂无代码证据。
- 结论：以后修改学生 AI 反馈链路时，后处理不得整体覆盖模型返回的多条有效结构化建议；只能删除、替换或补充不可信、重复、泄题或跑偏的单项。前端不得把同一条建议同时作为摘要证据和逐条 item 重复展示。
- 复用：涉及 `studentReport`、`repairItems`、`improvementItems`、runtime grounding、drift alignment、证据展示或前端反馈卡片时，必须检查是否存在新的 `List.of(...)` 覆盖、`get(0)` 主项误用、摘要与 item 双重渲染、或无代码证据空态不可达。
- 位置：`online-judge/src/main/java/com/onlinejudge/submission/application/AiReportService.java`、`online-judge/frontend/src/features/problem/ProblemPage.tsx`、`online-judge/src/test/java/com/onlinejudge/submission/application/StudentAiFeedbackModelTest.java`、`online-judge/openspec/specs/student-visible-ai-feedback-quality/spec.md`。

## 2026-07-06 教师端 UI 以左侧栏和教学下钻为主结构

- 背景：教师端页面曾把作业、题目、学生诊断、班级学情和管理入口分散在顶部导航、局部卡片和管理页状态块里，老师难以按真实教学路径判断下一步。
- 触发：用户明确指出旧作业详情界面凌乱，并补充教师真实流程应是“先看有哪些作业 → 点进作业看题目 → 点进题目看正确率、易错点和学生证据”；视觉草稿确认采用左侧栏 + 右侧内容区。
- 过程：通过 OpenSpec `redesign-teacher-console-ui` 重构教师端骨架：`TeacherShell` 统一渲染侧栏，作业中心改为作业列表 + 待处理队列，作业详情让题目推进表成为主栏，班级学情、新建作业、题库管理和 AI 标准库统一进入同一工作台语言；学生端只协调按钮、返回入口和面板边界。
- 结论：后续教师端 UI 默认使用左侧栏主导航，不再新增顶部教师子导航；教师主路径按“作业中心 → 作业题目 → 题目分析 → 学生诊断”下钻。传统教学信息优先，AI 信息只作为易错点、讲评建议、诊断校正和标准库治理辅助。
- 复用：以后新增教师端页面时，先判断它属于作业、班级学情、班级名单、题库管理还是 AI 标准库；不要再做独立入口拼贴。学生端轻改必须不重排做题、提交和 AI 反馈主体。
- 位置：`online-judge/frontend/src/features/teacher/TeacherShell.tsx`、`online-judge/frontend/src/features/teacher/TeacherPage.tsx`、`online-judge/frontend/src/features/teacher/AssignmentDetailPage.tsx`、`online-judge/frontend/src/features/teacher/TeacherHomeRefresh.css`、`online-judge/openspec/specs/teacher-console-ui/spec.md`。

## 2026-07-06 自动兜底标准库终态为静态冷备份

- 背景：自动兜底层最初用于没有外部模型时保证知识树覆盖，后来经过退役、降级、A/B/C 榨取和 V10/V11/V12 手写吸收后，继续保留运行时兜底库已经会增加误接入风险。
- 触发：用户明确要求删除兜底型库，但删除前先本地备份；备份可以随 Git/云端同步，但只能作为备份目录里的东西，永远不参与前后端。
- 过程：通过 OpenSpec `remove-generated-fallback-archive-runtime` 导出删除前快照到 `online-judge/backups/standard-library/generated-fallback-archive-2026-07-06/`，删除 `archivedGeneratedFallbackSeeds()`、自动生成器和 `AiStandardLibraryFallbackArchiveValueCatalog`，只保留 `isGeneratedFallbackCode(...)` 用于禁用历史数据库里的旧兜底记录。
- 结论：兜底库不再是运行时存档素材池，也不再由后端代码枚举。正式库只保留已经人工精修吸收的 V10/V11/V12/V13 条目；未吸收兜底只存在于静态备份 JSON，不能被 Seeder、候选包、质量报告、AI 上下文、前端或后端读取。
- 复用：以后如果想重新利用备份素材，必须先人工筛选、重写、测试，再进入正式手写标准库文件；不要直接读取 `backups/`，不要恢复自动 full coverage 生成器。
- 位置：`online-judge/backups/standard-library/generated-fallback-archive-2026-07-06/`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibrarySeedCatalog.java`、`online-judge/openspec/specs/standard-library-normalized-schema/spec.md`。

## 2026-07-06 标准库扩张必须以主题包保证质量

- 背景：V12 之后用户继续要求扩张标准库数量，但明确强调“提升的前提是保证质量，绝对不能为了凑数而凑数”。
- 触发：需要在数量增长和内容质量之间建立硬门槛，避免后续扩库退回模板化批量生成。
- 过程：通过 OpenSpec `expand-standard-library-v13-quality` 新增 V13 高质量手写扩库批次，按 12 个主题包扩张，每个主题固定包含 1 个能力点、3 个具体易错点和 1 个提升点；测试同时校验数量、合法知识节点、V13 内部归属、提升点关联和反模板表达。
- 结论：后续扩库可以追求明显数量增长，但增长单位应是“能力点 + 多个易错点 + 提升点”的主题包，而不是孤立标签。每个易错点必须写出具体触发条件、代码症状、状态偏差或验证动作。
- 复用：V14/V15 扩库时，先选高频且未充分覆盖的主题，再成组补条目；数量阈值只作为底线，专项测试必须继续检查反凑数质量。
- 位置：`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibraryV13QualityExpansionSeeds.java`、`online-judge/src/test/java/com/onlinejudge/learning/standardlibrary/AiStandardLibraryV13QualityExpansionSeedsTest.java`、`online-judge/openspec/specs/standard-library-normalized-schema/spec.md`。

## 2026-07-06 兜底存档按 A/B/C 三类榨取价值

- 背景：自动兜底层已经退役为冷存档，但存档中仍有大量知识点只被兜底覆盖，活跃标准库覆盖明显少于兜底存档。
- 触发：用户明确希望“榨干兜底存档价值”，不是把兜底原文搬回主库，而是能用的精修迁移，内容不好但类型有启发的提炼后重写，暂时没价值的继续存档。
- 过程：通过 OpenSpec `extract-fallback-archive-value-v12` 建立兜底存档价值分类目录，分为 A 类直接精修吸收、B 类提炼类型后重写、C 类继续存档观察；新增 V12 手写吸收批次，将循环控制、前缀统计、字符串构造、栈队列、哈希、分支覆盖、调试追踪和提交检查转为活跃标准库条目。
- 结论：兜底存档以后定位为“素材池和覆盖地图”，不得恢复为运行时自动补位；进入活跃标准库的内容必须人工精修，不复用“适用条件混用”“理解或应用偏差”等模板表达。
- 复用：后续 V13/V14 扩库时，先从 archive-only 知识点中选 A/B 高价值主题；每个主题按 1 个能力点、若干具体易错点和 1 个提升点重写，并保留 C 类作为冷存档观察对象。
- 位置：`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibraryFallbackArchiveValueCatalog.java`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibraryV12ArchiveValueSeeds.java`、`online-judge/openspec/specs/standard-library-normalized-schema/spec.md`。

## 2026-07-06 知识树与标准库优化先治理表达质量

- 背景：用户指出现有知识库、标准库部分内容可能不够准确、偏口语化或质量不够高，希望进入优化提升阶段。
- 触发：上一轮已经完成知识树扩建和 V8 手写易错点/提升点补充，数据库规模达到可用基础；下一步重点应从“继续堆数量”转向“提高现有内容可教学性”。
- 过程：通过 OpenSpec `polish-knowledge-standard-library-quality` 增加知识树和标准库内容质量要求；将知识树自动描述从“细颗粒知识点”“基本用法”等元描述，改为对象、边界、状态、调试证据导向；将标准库兼容标签从“能识别某某相关知识点和错误表现”改为具体诊断动作；并为低信息模板、口语化表达和兼容 fallback 增加回归测试或关键词校验。
- 结论：优化标准库时，不应只追求新增条目数；应先让自动兜底条目达到最低教学质量，再逐步人工精修高频薄弱点。正式 seed 文案应避免“会炸”“长得像不像”“细颗粒知识点”这类口语或元描述，改用可观察症状、边界样例、状态变量、复杂度估算和调试轨迹。
- 复用：以后新增或批量生成知识树/标准库内容时，先检查是否包含低信息模板；若是兼容旧标签，也必须写出具体代码对象、触发条件、失败症状和验证动作。
- 位置：`online-judge/openspec/changes/polish-knowledge-standard-library-quality/`、`online-judge/src/main/java/com/onlinejudge/learning/knowledge/application/InformaticsKnowledgeSeedCatalog.java`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibrarySeedCatalog.java`、`online-judge/src/test/java/com/onlinejudge/learning/standardlibrary/AiStandardLibrarySeederTest.java`。

## 2026-07-07 AI 诊断输出契约以逐条数组、证据状态和候选池为准

- 背景：诊断结果页面出现代码证据重复、提升建议没有证据时表达混乱、建议数量长期像是只有一条等问题；用户进一步明确 AI 链路应是“上下文包 → 外部大模型评测和打标签 → 新标签进入候选池 → 教师治理入库”。
- 触发：用户确认标准库成长采用“候选池”模式，而不是模型直接填充正式库；并要求从后端 AI 链路重新扫描并修复输入输出契约。
- 过程：通过 OpenSpec `realign-ai-diagnosis-contract` 将 `studentReport` 固定为摘要，把 `basicLayerAdvice`、`improvementLayerAdvice` 作为真实逐条数组；提升建议允许没有直接代码证据，但给出 `evidenceRefs` 时必须有效；禁止 `sourceCode/code/verdict/problemConstraints` 等泛化别名自动绑定到第一条证据；标准库成长候选由后端补当前题目和提交来源，并保存 `SUPPORTED / NO_DIRECT_CODE_EVIDENCE / UNSUPPORTED` 证据状态。
- 结论：以后 AI 诊断链路不能把摘要当列表来源，不能用全局证据灌到每条建议，不能把模型自报的来源 ID 当可信来源。教师深诊断必须使用标准上下文，学生快反馈才可以低延迟压缩。
- 复用：修改 prompt、validator、mapper、候选池服务或学生/教师展示时，同时检查三条边界：逐条建议数量由模型数组和证据决定；证据引用只能来自当前上下文包；成长内容先入候选池并保留来源、频次、证据状态和相似条目，正式入库由教师治理。
- 位置：`online-judge/openspec/changes/realign-ai-diagnosis-contract/`、`online-judge/src/main/java/com/onlinejudge/submission/application/AdviceGenerationOutputValidator.java`、`AdviceGenerationFeedbackMapper.java`、`AiReportService.java`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibraryGrowthAgentService.java`、`online-judge/frontend/src/features/problem/ProblemPage.tsx`、`online-judge/frontend/src/features/teacher/TeacherManagementPage.tsx`。

## 2026-07-06 AI 标准库成长系统收缩为主诊断链路和人工治理闭环

- 背景：完成 AI 诊断成长闭环后，曾把后续方向拆成质量观察、治理 drill-down、自动评测样本沉淀和自动治理 Agent。用户担心链路过多会导致实现变杂、效果不集中。
- 触发：用户明确判断“主诊断链路已经对标准库进行成长，后面人工治理收尾就够了”，并要求先把标准库页面做成简洁高效、视觉清爽的人工治理页面。
- 过程：通过 OpenSpec `polish-standard-library-governance-workbench` 将“待审核”和“治理摘要”合并为同一个人工治理台；保留正式库视图和人工治理视图两个入口；治理摘要只作为审核辅助，点击高频路径或待补强路径直接筛选候选列表。
- 结论：当前 AI 标准库成长系统只保留两条执行链路：主诊断链路负责诊断、路径标注和候选生成；人工治理闭环负责教师审核、修正、合并、拒绝和正式入库。独立质量运营链路、自动评测样本沉淀和自动治理 Agent 暂不进入当前执行范围。
- 复用：以后继续改 AI 标准库页面时，优先提升人工治理台效率，不新增平行治理页面；AI 可以辅助提出候选，但正式库变更仍需教师确认。
- 位置：`online-judge/frontend/src/features/teacher/TeacherManagementPage.tsx`、`online-judge/openspec/specs/standard-library-review-workflow/spec.md`、`online-judge/openspec/specs/standard-library-governance-quality/spec.md`。

## 2026-07-06 数据库扩张先细化知识树再补手写标准库 V8

- 背景：用户要求进一步扩张数据库，并明确第一步细化知识树、第二步人工添加高质量细颗粒易错点与提升点。
- 触发：前一轮数据库体检显示知识树已有 707 节点、标准库已有较大规模，但自动铺底条目占比高，规范提升点仍为空，真实教师修正数据不足。
- 过程：通过 OpenSpec `expand-knowledge-tree-standard-library-v8` 扩展既有知识树 topic 下的小知识点，并新增 V8 标准库模块，采用“能力点 + 具体易错点 + 提升点”的手写结构；同时让 V8 `IMPROVEMENT_POINT` 进入规范化 `ai_standard_improvement_points`。
- 结论：后续扩库不要只堆自动 full coverage 数量，应优先补手写高质量条目；新增内容应挂到既有知识树路径，提升点必须和能力点、相关易错点形成结构关系。
- 复用：下一轮扩库先选薄弱 topic，再补 1 个能力点、3 个左右易错点和 1 个提升点，并同步更新 Vx 专项测试、规范化同步测试和 OpenSpec 任务。
- 位置：`online-judge/openspec/changes/expand-knowledge-tree-standard-library-v8/`、`online-judge/src/main/java/com/onlinejudge/learning/knowledge/application/InformaticsKnowledgeSeedCatalog.java`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibraryV8ExpansionSeeds.java`、`online-judge/src/test/java/com/onlinejudge/learning/standardlibrary/AiStandardLibraryV8ExpansionSeedsTest.java`。

## 2026-07-06 AI 诊断成长闭环按五层收束

- 背景：第一阶段已经把正式诊断契约改成“AI 自由诊断时同步标注标准库路径、命中状态和成长候选”，但用户进一步追问这是否已经达到原始意图，以及是否只是完成了第一阶段。
- 触发：用户要求继续规划并执行所有阶段，重点不是再做一个 prompt，而是让 AI 链路真正达到“诊断真实问题，同时挂接标准库路径，未命中时进入可审核成长池”的产品闭环。
- 过程：将链路重新明确为五层：上下文打包层、自由诊断层、标准库映射层、标准库成长层、输出与治理层。本轮通过 OpenSpec `complete-ai-diagnosis-growth-loop` 补齐真实样本回归、上下文包门禁、成长候选治理摘要 API、教师端“正式库 / 待审核 / 治理”视图和归档前验证。
- 结论：第一阶段只完成主调用协议，不等于完整目标达成。完整闭环必须同时具备：模型能看到完整题目和代码、输出能绑定最近标准库路径、`PARTIAL` 能生成待审核候选、教师能审核入库或拒绝合并、治理视图能暴露高频未命中和薄弱路径。标准库仍是教学参考规范包，不是硬套答案库。
- 复用：以后继续做 AI 诊断、标准库扩容或教师端治理时，先确认五层是否都被覆盖；不要只改 prompt 就宣称链路完成。新增错误点应优先挂最近路径进入待审核池，`MISS` 只在上级路径也找不到时使用。
- 位置：`online-judge/openspec/changes/complete-ai-diagnosis-growth-loop/`、`online-judge/src/main/java/com/onlinejudge/submission/application/PromptTemplateRegistry.java`、`online-judge/src/main/java/com/onlinejudge/learning/standardlibrary/application/AiStandardLibraryGrowthAgentService.java`、`online-judge/frontend/src/features/teacher/TeacherManagementPage.tsx`。

## 2026-07-06 AI 诊断链路默认模型与安全校验边界

- 背景：8 题真实链路仿真确认 ModelScope key 可用，但 `deepseek-ai/DeepSeek-V4-Flash/Pro` 在当前路径返回空结果；`Qwen/Qwen3-235B-A22B-Instruct-2507` 能完成真实诊断。同时有 1 题被 `SAFETY_RISK` 拦截，暴露安全关键词误杀算法诊断的风险。
- 触发：用户要求在简化 AI 链路的同时执行修改，重点提升默认可用性、减少无谓拦截，并保持标准库只是教学参考规范包。
- 过程：通过 OpenSpec `tune-ai-diagnosis-runtime-quality` 将默认模型改为 Qwen 主模型；安全策略从“大词表误杀”收缩到“直接答案/逐行改法/可复制替换表达式”拦截；`diagnosis-report-v2` prompt 明确 `studentReport` 是学生自然反馈，`diagnosisDecision`、`diagnosisCandidates`、`teacherTrace`、`libraryGrowth` 是后端 metadata。
- 结论：默认诊断模型应优先使用最近真实链路验证可用的模型；安全校验不能仅因“状态维度、窗口状态、枚举顺序、边界手推”等正常诊断词触发 `SAFETY_RISK`；prompt 必须分离学生报告和后端 metadata。
- 复用：以后调整模型、prompt、validator 或 live eval 时，先做模型 smoke test；安全策略只把可复制答案式表达作为硬风险；学生可见文案不得混入后端字段名或标准库审计信息。
- 位置：`online-judge/src/main/resources/application.yml`、`online-judge/src/main/java/com/onlinejudge/submission/application/ModelOutputSafetyPolicy.java`、`online-judge/src/main/java/com/onlinejudge/submission/application/PromptTemplateRegistry.java`、`online-judge/openspec/specs/ai-diagnosis-orchestrator-v2/spec.md`。

## 2026-07-05 标准库在 AI 链路中定位为教学参考规范包

- 背景：标准库已经重构为“知识树 → 能力点 → 易错点 / 提升点”，但 prompt 和 validator 中仍残留把标准库 ID 当硬命中条件的倾向，可能限制外接大模型对题目、代码和判题结果的真实判断。
- 触发：用户明确指出标准库的意义是“像老师拿着教育部教案讲课”，应引导大模型按规范表达，而不是把标准库写成死纲或答案约束。
- 过程：通过 OpenSpec `position-standard-library-as-reference` 明确标准库是 teaching reference pack；搜索定位层允许 `MISS`、证据不足或需扩库时不选择候选；建议层将未知标准库 ID 从硬失败降级为清空 ID 并记录 `softFixes`；prompt 要求先自由诊断真实问题，再映射 `HIT/PARTIAL/MISS/OUT_OF_LIBRARY`。
- 结论：标准库提供术语、颗粒度、教学分层和扩库线索；诊断权仍属于正式诊断 Agent，当前提交 evidenceRefs、安全边界和结构合法性才是硬门槛。标准库 ID 不匹配时应使用 null、`MISS` 或 `OUT_OF_LIBRARY`，不要硬套弱 ID。
- 复用：以后调整 AI prompt、normalizer、validator、搜索召回或标准库成长时，只能把标准库作为参考规范包使用；不要新增并行链路，也不要把未知标准库 ID 重新变成整包失败条件。

## 2026-07-05 AI 链路必须以标准库结构包为主协议

- 背景：标准库已重收纳为“知识树 → 能力点 → 易错点 / 提升点”，但 AI 链路仍主要消费旧扁平列表，容易出现 prompt 知道结构、选择器却只塞单条或无关条目的割裂。
- 触发：用户要求“大改 AI 链路”，使 AI 链路与标准库相互适配，同时不要另起混乱后端模块。
- 过程：在现有搜索定位和诊断链路内扩展 `SearchLocationCandidate` 与 `StandardLibraryPack`，加入 `knowledgeGroups`、父能力、子易错、兄弟易错、同能力提升和结构路径；`SearchLocationPackSelector` 根据模型选中的候选补齐同能力局部邻域；`ExternalModelAgentRuntime` compact 保留结构视图；prompt 明确标准库主结构与证据优先规则。
- 结论：AI 链路的主协议是结构化候选包，旧 `basicCauses`、`improvementPoints`、`skillUnits`、`mistakePoints` 只是兼容合法 ID 列表。邻域扩展必须围绕同一个能力点，不能按同知识节点泛化带入无关能力点。
- 复用：以后调整召回、选择器、validator、prompt 或输出 mapper 时，先检查 `knowledgeGroups` 是否完整保留，且多个真实易错/提升按证据返回；不要新增一套平行 V2/V3 后端模块。

## 2026-07-05 AI 标准库主结构收纳为知识树下的能力、易错和提升

- 背景：标准库曾通过 `ai_standard_library_items.layer` 同时承载 `SKILL_UNIT`、`MISTAKE_POINT`、`BASIC_CAUSE` 和 `IMPROVEMENT_POINT`，结构越来越像扁平标签堆，难以解释和继续扩张。

## 2026-07-05 AI 分层建议数量由证据决定

- 背景：AI 诊断链路曾把基础层、提高层等学生反馈压缩成单条文本或少量固定条目，导致多个独立错误或多个真实提升方向被合并、丢失或只展示第一条。
- 触发：用户明确要求“有几个错误就返回几条，有几个提升建议就返回几条”，不能由后端 prompt 模板或代码硬限制成固定一条。

- 结论：以后修改 AI 反馈链路时，不要把基础层/提高层/修复项/提升项硬编码为单条或固定 1-2 条；可以做安全、证据、去重和展示过滤，但数量应来自真实独立问题和真实提升方向。
