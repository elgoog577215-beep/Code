# 项目决策

## 2026-05-25 长期学习记忆必须接入诊断主证据链

- 背景：讨论是否借鉴 Hermes 等成熟 Agent 的长期记忆能力时，明确不能新建一套旁路学生记忆，否则会导致诊断、教师统计、推荐和评测各算各的。
- 触发：用户强调“要保持改动连贯性，不要拆东墙补西墙”，并要求排查原本是否已有旁路功能需要统一。
- 过程：排查发现项目已经有 `StudentAbilityProfileService`、`StudentTrajectoryService`、`LearningActionEvidenceAnalyzer`、`DiagnosisReportReader`、教师修正和 AI 质量统计等相关能力。第一期选择不新增数据库表、不重写教师端，而是把学习记忆作为 `DiagnosisEvidencePackage` 的证据进入 `ModelDiagnosisBrief` 和外部模型 prompt。
- 结论：长期记忆应定义为 `StudentLearningMemorySnapshot` 这类可追溯聚合快照，来源必须是现有提交、诊断报告、学习动作反馈和教师修正；当前提交直接证据优先，记忆只能辅助诊断、提示粒度和教师关注。
- 复用：以后继续做学生画像、教师端统计、推荐、Coach 反馈时，优先复用诊断报告 JSON、证据包和统一 reader，不要再新建孤立事实源。
- 位置：`openspec/changes/integrate-student-learning-memory/` 与本文件。

## 2026-05-26 学生记忆必须经过诊断校准后才能影响主错因

- 背景：继续升级教育 AI agent 的记忆能力与诊断能力时，上一轮已经让 `StudentLearningMemorySnapshot` 进入诊断主链路，但系统仍主要依赖 prompt 文字约束模型“不要被记忆带偏”，缺少程序化校验。
- 触发：用户明确要求本轮升级必须与原有代码衔接，不能另起山头形成屎山代码；同时目标是提升外部模型接入后的真实诊断质量，而不是只让离线规则看起来更好。
- 过程：本轮没有新建记忆表、向量库或独立 memory agent，而是在现有 `DiagnosisEvidencePackage -> ModelDiagnosisBrief -> StandardLibraryPack -> PromptTemplateRegistry -> ModelOutputValidator -> DiagnosticAgentService` 链路里新增 `MemoryEvidencePolicy` 和 `memoryCalibration`。策略只判断记忆与当前证据是 `ALIGNED`、`CONFLICTING` 还是 `TEACHING_ONLY`，不重新推断错因。
- 结论：学生长期记忆只能作为辅助先验。当前提交的编译、运行、评测、源码候选信号必须优先；如果模型主诊断只引用 `memory:*`，或选择了 teaching-only / conflicting 的记忆标签而没有当前证据支撑，必须被 validator 拒绝或回退。
- 复用：以后继续做学生画像、教师端统计、coach agent 或外部模型 prompt 升级时，先判断新增信息是否会影响主诊断。如果会影响，必须接入证据校准和测试；不能只把信息塞进 prompt。
- 位置：`openspec/changes/upgrade-memory-aware-diagnosis/`、`MemoryEvidencePolicy`、`ModelDiagnosisBrief.memoryCalibration`、`ModelOutputValidator`。

## 2026-05-26 外部模型评测必须区分模型输出和本地兜底

- 背景：用户要求真实测试外部模型能力，并明确测试代码不能是短代码或无 bug 示例，至少应使用 20 行以上、贴近学生提交的诊断样本。
- 触发：第一次 ModelScope DeepSeek-V4-Pro live eval 在 10 个长代码样本上因为 `insufficient_quota` 全部走本地兜底，但报告仍把兜底结果的 `expectedSignalHit=true` 计入统计，容易误读为外部模型命中。
- 过程：本轮筛选 `assistant-live-cases.json` 中 10 个 20 行以上的 `SUBMISSION_DIAGNOSIS` 样本真实调用外部模型。随后修正 `AssistantLiveEvalTest` 的汇总与单条 entry 口径：只有 `completedOutput=true` 的模型输出才能计入 `expectedSignalHit`、`evidenceValid`、`teachingActionValid`；fallback 只记录失败阶段、失败原因和兜底文本。
- 结论：教育 agent 的外部模型质量评测必须分层统计：外部模型完成率、阶段部分完成率、运行失败率、质量命中率、安全通过率、本地兜底可用性。不能把本地规则兜底的正确性算作外部模型能力。
- 复用：以后测试外部模型时，默认使用 20 行以上真实或半真实样本；报告中必须明确 `MODEL_COMPLETED`、`MODEL_PARTIAL_COMPLETED`、`MODEL_RUNTIME_FALLBACK`，并单独展示老师期望、模型输出、AI 优势、老师优势和下一轮优化建议。
- 位置：`AssistantLiveEvalTest`、`AssistantLiveEvalQualityGateTest`、`src/test/resources/assistant-eval-fixtures/assistant-live-cases.json`、`target/ai-eval-reports/assistant-live-eval-20260526-111108.json`。

## 2026-05-27 教育 agent 目标必须从准确率升级为学生与教师结果

- 背景：用户要求重新反思目标是否具体、可执行、够远大。原先目标已经包含外部模型完成率、错因命中率、细粒度命中率和安全率，但仍偏单次模型输出，缺少学生改善与教师价值。
- 触发：用户明确要求“全部执行”，因此本轮把目标体系写入 OpenSpec，并把第一阶段目标落实到 live eval 报告字段中，而不是只停留在讨论。
- 过程：新增 `establish-education-agent-goal-system` change，定义五层目标：北极星目标、第一阶段外部模型诊断可用、第二阶段学生学习效果可见、第三阶段教师教学价值可见、长期成熟化。同步扩展 `AssistantLiveEvalReport.goalSnapshot`，让每次报告展示完成率、运行失败率、质量率、安全率、教学动作率和目标缺口。
- 结论：后续优化不能只问“AI 这次说得对不对”，而要持续追问“学生下一次是否更会改、教师下一节是否更会教、系统是否更懂学生”。第一阶段仍先解决外部模型完成率、长代码评测集、主错因和教学动作，因为没有这些基础就无法可靠衡量后续教育结果。
- 复用：以后做 AI 能力升级时，先看 live eval 的 `goalSnapshot.goalGaps` 决定优先级；如果完成率不达标，先处理模型调用与配额；如果完成率达标但质量差，再改 prompt、标准库、validator；如果单次质量稳定，再推进学生改善率和教师端统计。
- 位置：`openspec/changes/establish-education-agent-goal-system/`、`AssistantLiveEvalReport.goalSnapshot`、`AssistantLiveEvalTest.buildGoalSnapshot`。

## 2026-05-27 外部模型评测必须先区分容量失败和质量失败

- 背景：继续朝“外部大模型真实接入后高质量反馈”的目标推进时，3 条长代码 single-call smoke 曾出现一次全部通过，也出现一次 `429 insufficient_quota` 后全部失败。
- 触发：用户要求不到目标不要停止，因此本轮继续分析目标缺口，发现旧报告在容量失败时容易把质量率也拉低，进而误导下一步去改 prompt、标准库或 validator。
- 过程：把 live eval 默认外部诊断 runtime 改为 `single-call`，降低单条样本请求成本；新增 `runtimeMode`、`sampleProfile`、`routeProfile`、`coverageGaps`、`failureReasonCounts`、`nextOptimizationFocus` 和 `nextAction`。同时修正质量率分母：完成率和运行失败率统计全部请求样本，信号命中、证据有效和教学动作质量只统计 `completedOutput=true` 的模型输出。live eval 中的 `AiReportService` 与 `CoachAgentService` 共享 `ExternalModelBudgetGuard`，第一次 `INSUFFICIENT_QUOTA` 或 `RATE_LIMITED` 后，后续样本快速记录为 `BUDGET_GUARD_OPEN`。如果容量失败且 `routeProfile` 只有一个路由，下一步焦点指向 `MODEL_ROUTE_CONFIGURATION`。
- 结论：外部模型评测要先判断 provider 容量是否可用。没有完成模型输出时，不能评价模型诊断质量；只能评价完成率、运行失败率和容量失败类型。小样本 smoke 成功只能证明链路可用，不能证明整体质量达标。
- 复用：以后跑 10 条、100 条或更多真实样本时，先看 `failureReasonCounts`、`routeProfile`、`goalSnapshot.goalGaps` 和 `goalSnapshot.nextOptimizationFocus`。如果主因是 `INSUFFICIENT_QUOTA`、`RATE_LIMITED` 或 `BUDGET_GUARD_OPEN`，优先换额度、换模型路由、降低输出 token、增加调用间隔或切换供应商；不要先调 prompt。
- 位置：`AssistantLiveEvalReport.runtimeMode/sampleProfile/failureReasonCounts`、`AssistantLiveEvalTest.liveEvalRuntimeMode`、`AssistantLiveEvalQualityGate`、`openspec/changes/establish-education-agent-goal-system/`。

## 2026-05-27 外部模型容量瓶颈要进入生产路由能力，而不是只停留在评测解释

- 背景：`goalSnapshot.nextOptimizationFocus=MODEL_ROUTE_CONFIGURATION` 已经说明当前瓶颈是外部模型路由与容量，而不是单条诊断提示词质量。只在 live eval 报告里提示“配置备用路由”还不够，生产诊断、Coach 追问和成长报告必须真的能使用备用 OpenAI-compatible 路由。
- 触发：10 条长代码真实评测中，已完成输出的信号、证据、安全和教学动作达标，但主路由 quota/guard 导致整体完成率不达标。继续改 prompt 会优化错方向。
- 过程：新增 `ExternalModelRoute`，把 provider/baseUrl/apiKey/model 抽成统一路由对象；在 `AiReportService` 与 `CoachAgentService` 原链路内增加主路由和备用路由配置，不另起调用链；调用时按路由顺序尝试，预算门禁仍按 provider/model 隔离；live eval 构造服务时同步注入 `AI_EVAL_FALLBACK_*`，让评测配置和生产服务配置一致。
- 结论：外部模型 agent 的稳定性不能只靠“失败后本地兜底”。本地兜底保证可用，备用路由保证外部模型能力有机会继续发挥。容量失败时应先做路由、额度、模型供应商和调用预算治理，再谈 prompt、标准库和 validator。
- 复用：以后看到 `INSUFFICIENT_QUOTA`、`RATE_LIMITED`、`BUDGET_GUARD_OPEN` 且质量指标只在 completed output 上达标时，优先检查 `routeProfile` 与生产路由配置；新增模型供应商时必须接入同一 `ExternalModelRoute` 与 `ExternalModelBudgetGuard`，不要新建孤立 HTTP 客户端。
- 位置：`openspec/changes/add-external-model-route-fallback/`、`ExternalModelRoute`、`AiReportService`、`CoachAgentService`、`AssistantLiveEvalTest`。

## 2026-05-27 生产默认 runtime 要和真实在线目标一致

- 背景：live eval 已默认使用 `single-call` 降低外部模型请求次数，但生产 `AiReportService` 默认仍是 `staged`。这会造成评测路径比真实线上路径更省请求，评测结论无法准确代表学校使用时的体验。
- 触发：使用 ModelScope key 跑 3 条长代码真实 live eval 时，第一条完成并质量达标，后两条因 `INSUFFICIENT_QUOTA` 和 `BUDGET_GUARD_OPEN` 失败。报告继续定位到 `MODEL_ROUTE_CONFIGURATION`，同时说明在线完成率仍是最大瓶颈。
- 过程：将 `ai.external-runtime-mode` 默认改为 `single-call`，并在 `application.yml` 暴露 `AI_EXTERNAL_RUNTIME_MODE`。staged 测试全部显式设置 `externalRuntimeMode=staged`，保留 staged 的诊断、教学拆分、partial 和 fallback 回归能力。
- 结论：当外部模型质量在 completed output 上已达标，而完成率不达标时，减少每条诊断的请求次数是比继续调 prompt 更高杠杆的生产优化。默认 single-call 是线上低预算、高完成率路径；staged 适合作为显式调试或高预算路径。
- 复用：以后新增评测或运行模式时，必须检查“评测默认”和“生产默认”是否一致；如果不一致，要在 spec 和报告中明确说明，否则很容易把评测表现误认为线上表现。
- 位置：`openspec/changes/promote-single-call-runtime-default/`、`AiReportService.externalRuntimeMode`、`application.yml`、`AiReportServiceExternalRuntimeTest`。

## 2026-05-28 live eval 必须记录每条样本的真实路由归因

- 背景：项目已经支持主路由和备用 OpenAI-compatible 路由，但 live eval entry 只有 `model`、`status` 和 `fallbackUsed`，无法判断某条样本到底由主供应商完成、备用供应商接管，还是本地规则兜底。
- 触发：继续朝“外部模型真实在线效果”目标迭代时，`goalSnapshot.nextOptimizationFocus=MODEL_ROUTE_CONFIGURATION`。如果没有 per-entry route attribution，后续即使配置备用路由，也无法评估路由治理是否真的提升完成率。
- 过程：新增 `AssistantLiveEvalReport.Entry.actualProvider/actualModel/routeRole`。诊断类 entry 从 `AiInvocation.provider/model/fallbackUsed` 读取真实归因；Coach 和成长报告暂时没有统一 invocation DTO，因此只做保守归因，不伪造精确调用链。route role 判断同时看 provider 和 model，避免“同模型不同供应商”被误判为主路由。
- 结论：外部模型 agent 的评测报告不能只回答“有没有输出”，还必须回答“是谁输出的”。路由归因是后续做供应商切换、备用模型评估、容量治理和质量对比的基础。
- 复用：以后扩展 live eval 或教师端 AI 质量统计时，按 `completedOutput` 评质量，按 `actualProvider/actualModel/routeRole` 分析容量和路由；本地兜底必须标为 `LOCAL_FALLBACK`，不能混入外部模型完成率。
- 位置：`openspec/changes/add-live-eval-route-attribution/`、`AssistantLiveEvalReport.Entry`、`AssistantLiveEvalTest.resolveDiagnosisRouteRole`。

补充：外部路由失败后即使最终走本地兜底，`AiInvocation` 也应保留最后尝试的外部 provider/model。否则 route outcome 会把 `INSUFFICIENT_QUOTA`、`RATE_LIMITED`、`BUDGET_GUARD_OPEN` 全部归到 `LOCAL_RULES`，后续无法判断真实容量瓶颈属于哪个外部供应商或模型。

## 2026-05-28 教学提示安全判断必须以学生可见内容为准

- 背景：一次外部模型 live eval 中，模型已经命中 `IO_FORMAT / INPUT_PARSING`，但最终学生提示被安全链路降级，教学动作从 `COMPARE_INPUT_SPEC` 退回到 `COLLECT_EVIDENCE`。
- 触发：用户要求继续朝“外部模型真实在线效果提升”推进，不允许只停留在本地规则效果。分析报告发现，已完成模型输出的错因判断并不差，落差主要出现在输出后处理的安全误判。
- 过程：排查 `ModelOutputValidator`、`AiReportService.resolveRuntimeAnswerLeakRisk` 和 `HintSafetyService` 后，将安全边界拆成学生可见字段和非学生可见字段。`studentHint`、`studentHintPlan`、`learningInterventionPlan` 必须继续严格检查；顶层 `teachingHint.answerLeakRisk` 和 `teacherNote` 不能单独导致学生端降级。
- 结论：教育 agent 的安全不是越保守越好，而是要把保护对象说清楚。学生可见内容出现完整代码、直接替换写法、最终答案或隐藏数据时必须拦截；教师备注或模型自评字段只应作为内部参考，不能把安全提示误伤成低质量提示。
- 复用：以后新增安全字段、教师端备注、模型自评风险或报告调试信息时，先判断它是否学生可见。如果不可见，不应直接参与学生端 `answerLeakRisk` 或提示降级；如果未来产品把它展示给学生，必须重新纳入 `HintSafetyService` 扫描。
- 位置：`openspec/changes/harden-student-visible-safety-risk/`、`ModelOutputValidator.firstSafetyTrigger`、`AiReportService.resolveRuntimeAnswerLeakRisk`、`HintSafetyServiceTest.doesNotDowngradeSafeStudentHintBecauseTeacherNoteMentionsForbiddenExactFix`。

## 2026-05-28 低预算上下文必须只压缩模型输入，不能弱化后端验证

- 背景：外部模型 live eval 出现“成功返回时质量达标，但配额和完成率不稳定”的状态。继续增加提示词规则会推高 token 成本，反而可能降低在线完成率。
- 触发：3 条长代码 live eval 中，2 条 `MODEL_COMPLETED` 且信号、证据、安全、教学动作均通过，1 条因 `INSUFFICIENT_QUOTA` 失败。下一步瓶颈从诊断质量转向单次调用成本和路由容量。
- 过程：本轮没有删内部 `ModelDiagnosisBrief` 和 `StandardLibraryPack` 字段，而是在 `AiReportService` 发起外部调用前构造 compact prompt context：限制题面和代码长度，保留候选信号、证据、允许标签、教学动作、记忆校准和隐藏数据边界；标准库只发送 ID、父标签、教学动作和学生任务模板等必要字段。validator、normalizer、报告和安全层仍使用完整 `RuntimePlan`。
- 结论：教育 agent 的低预算优化应区分“模型输入压缩”和“系统能力削弱”。可以压缩发给模型的上下文，但不能删除本地证据链、标准库、validator 或安全门禁。
- 复用：以后遇到配额、token、延迟瓶颈时，优先做 prompt context compaction、输出 token 限制、路由 fallback 和请求次数治理；不要通过删除验证器、减少评测字段或降低安全标准来换完成率。
- 位置：`openspec/changes/compact-external-model-prompt-context/`、`AiReportService.compactBriefForPrompt`、`AiReportService.compactStandardLibraryForPrompt`、`AiReportServiceExternalRuntimeTest.singleCallRuntimeSendsCompactPromptContextWithoutDroppingCoreSignals`。

## 2026-05-28 外部完成率瓶颈应优先扩路由容量，而不是继续改提示词

- 背景：10 条 20 行以上长代码 `SUBMISSION_DIAGNOSIS` 真实 live eval 覆盖率达标，但 ModelScope DeepSeek-V4-Pro 只完成前 2 条，后续因 `insufficient_quota` 和预算门禁阻断，整体完成率只有 20%。
- 触发：2 条已完成输出的主错因、细粒度错因、证据、安全和教学动作均达标，说明此轮最大缺口不是模型不会分析，而是单一路由容量不可持续。
- 过程：沿已有 `primary -> fallback` 设计继续扩展，而不是另起新调用链。新增 `ai.routes` 路由池，按 `provider|baseUrl|apiKey|model;...` 解析额外 OpenAI-compatible 路由；`AiReportService`、`CoachAgentService` 共用同一 `ExternalModelRoute.parseRoutes`；预算门禁仍按 provider/model 隔离，主路由 quota 不阻断后续路由。
- 结论：当 live eval 的质量指标在已完成样本上达标，而完成率被 quota、限流或预算门禁拖垮时，下一步应提升外部路由容量和可观测性。继续微调 prompt 可能优化错方向。
- 复用：后续评测如果出现 `MODEL_ROUTE_CONFIGURATION` 或 route outcome 集中在单一 provider/model 的 quota/limit，应优先检查多路由配置、备用 key、供应商容量和门禁策略，再讨论提示词质量。
- 位置：`ExternalModelRoute`、`AiReportService`、`CoachAgentService`、`openspec/changes/add-external-model-route-pool`、`target/ai-eval-reports/assistant-live-eval-20260528-125308.json`。

## 2026-05-28 路由健康预检只判断配置容量，不替代真实质量评测

- 背景：多路由池已经接入生产诊断、Coach 和 live eval，但本机仍只有一个 ModelScope key。部署者需要在上线前知道当前是不是单路由风险，而不是等学生提交后才看到 `BUDGET_GUARD_OPEN`。
- 触发：10 条 live eval 报告已经能说明真实质量和容量缺口，但 live eval 会消耗额度；日常部署检查需要一个不调用外部模型、不消耗额度、不泄露密钥的只读预检。
- 过程：新增 `/api/system/ai-route-health`，读取 `ai.provider/base-url/api-key/model`、`ai.fallback.*` 和 `ai.routes`，输出 `DISABLED`、`NO_ROUTE`、`SINGLE_ROUTE_RISK`、`MULTI_ROUTE_READY` 等健康等级，并展示脱敏后的 provider、baseUrl、model、缺失字段和建议。
- 结论：路由健康预检回答“当前配置有没有外部模型容量冗余”；live eval 回答“外部模型真实完成率和诊断质量是否达标”。健康预检不能替代真实长代码评测，但能提前发现单路由、缺 key 和配置缺口。
- 复用：后续排查学校环境时，先看 `/api/system/ai-route-health` 判断配置风险，再用 10 条以上长代码 live eval 验证真实完成率、route outcome 和质量指标。
- 位置：`openspec/changes/add-ai-route-health-check/`、`AiRouteHealthService`、`AiRouteHealthResponse`、`SystemController.getAiRouteHealth`。

## 2026-05-28 学生端也必须区分外部模型完成和本地兜底

- 背景：后端已经在 `AiInvocation.status/fallbackUsed/provider/model` 中记录外部模型调用状态，但学生题目页只显示“分析 READY/PROCESSING”。如果外部模型失败后走本地规则，学生和部署者仍可能误以为这是外部大模型给出的反馈。
- 触发：继续朝“外部大模型真实在线效果”目标推进时，发现教师端已经能看到路由健康，但学生端还缺少反馈来源可见性。上线后这会掩盖外部 API 不稳定问题，也会让产品验收口径不清。
- 过程：前端补充 `AiInvocation` 类型，并在题目页提交结果指标中用 `aiInvocation.status` 与 `fallbackUsed` 显示“外部模型完成”“外部模型部分完成”或“本地兜底”。验证时用 mock 提交结果模拟 `MODEL_RUNTIME_FALLBACK`，确认页面显示“本地兜底”，且不会误标为“外部模型完成”。
- 结论：本地兜底是可用性保障，不是外部模型能力。学生端和教师端都应该保留这个区分，否则后续 live eval、学校验收和真实体验会混在一起。
- 复用：以后新增学生可见 AI 能力时，如果后端有 invocation/status/fallback 字段，前端必须同步展示或至少不误导；本地规则结果不能被包装成外部模型结果。
- 位置：`ProblemPage.analysisSourceLabel`、`SubmissionAnalysis.aiInvocation` 前端类型、`AiInvocation.status/fallbackUsed`。

## 2026-05-28 预算门禁是按路由隔离，不应被误判为提示词质量问题

- 背景：10 条长代码 live eval 中，前 2 条完成且质量达标，后续 8 条主要是 `INSUFFICIENT_QUOTA` 和 `BUDGET_GUARD_OPEN`。继续调 prompt 不会解决这种失败。
- 触发：新增路由池后，需要确认预算门禁是否会误伤备用路由或路由池。
- 过程：排查 `ExternalModelBudgetGuard` 与 `AiReportService/CoachAgentService` 调用方式，确认门禁 key 是 `provider|model`；已有测试覆盖主路由 guard 打开后仍可使用 fallback 或 `AI_ROUTES` 路由池。因此单一路由连续出现 `BUDGET_GUARD_OPEN` 代表该路由短期熔断，不代表所有外部模型都不可用。
- 结论：看到 `BUDGET_GUARD_OPEN` 时，先看 route outcome 的 provider/model。如果只集中在一个 provider/model，应增加可用路由、换 key、延长评测间隔或调整供应商；不要先改提示词。
- 复用：后续 live eval 报告中若 `routePoolCount=0` 且 `BUDGET_GUARD_OPEN` 高发，结论应是路由容量不足；若多条路由都高发，再检查全局调用频率、测试节奏或供应商共同限制。
- 位置：`ExternalModelBudgetGuard.key`、`AiReportService.firstAllowedRouteDecision`、`CoachAgentService.firstAllowedRouteDecision`、`AiReportServiceExternalRuntimeTest.singleCallRuntimeUsesRoutePoolAfterPrimaryAndFallbackFail`。
